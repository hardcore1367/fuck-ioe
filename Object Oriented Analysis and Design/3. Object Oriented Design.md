# Object Oriented Design

## 3.2 Use Case and Use case Diagrams

### Terminologies

**1. Association**
A structural relationship between two classes where objects of one class are connected to objects of another.

- Notation: `A ——— B` (solid line)
- Can have multiplicity: `1`, `*`, `1..*`, `0..1`
- Example: `Student` and `Course` — a student enrolls in many courses

```
Student ————————— Course
        enrolls in
        1         *
```

**2. Dependency**
A relationship where one class uses or depends on another. Change in one affects the other.

- Notation: `A - - - - > B` (dashed arrow)
- Weaker than association — temporary/short-term use
- Example: `Order` depends on `PaymentService` to process payment

```
Order - - - - - > PaymentService
       <<uses>>
```

---

**3. Generalization**
A relationship where a child class inherits properties and behavior of a parent class (IS-A relationship).

- Notation: `Child ——▷ Parent` (solid line with hollow arrowhead pointing to parent)
- Promotes reusability
- Example: `Dog` and `Cat` are generalized as `Animal`

```
        Animal
        ▲    ▲
        |    |
       Dog  Cat
```

**4. Include Relationship**
A use case that is always invoked/required by another use case. The included use case is mandatory.

- Notation: `A ----<<include>>----> B` (A always calls B)
- Example: `Login` is always required when `View Account`

```
View Account ----<<include>>----> Login
```

**5. Extend Relationship**
A use case that is optionally invoked by another use case under certain conditions.

- Notation: `A ----<<extend>>----> B` (A optionally extends B)
- Example: `Apply Discount` optionally extends `Checkout`

```
Apply Discount ----<<extend>>----> Checkout
```
