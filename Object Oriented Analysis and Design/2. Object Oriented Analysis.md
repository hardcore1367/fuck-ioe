# Object Oriented Analysis

## 2.1 Building Domain/Conceptual Models

### Domain Model

A domain modeling is the visual representation of conceptual classes or real real-world objects in a domain of interest.

> Class diagram but only with: (components of Domain Model)
> Class name
> Attributes
> Associations (with multiplicity)

**Q. How does domain model lower the representation gap between conceptual and software model? [3 marks]**

**Representation Gap** — The difference between how we think about the real world and how it is represented in software.

Domain model lowers this gap by:

- **Using real-world concepts directly** — classes in domain model are named after real-world things (`Student`, `Course`, `Booking`) not technical terms, making it easy to map to software classes later.

- **Preserving relationships** — associations and multiplicities in domain model directly translate to relationships in class diagram, reducing misinterpretation.

- **Shared understanding** — both developers and stakeholders can read domain model, ensuring software reflects real-world accurately.

> In short — domain model makes software **look like** the real world, so transition from concept to code is smooth with minimal information loss.

### Conceptual Class

A real-world concept, thing, or object in the problem domain that is relevant to the system being developed; represented with a name, attributes, and associations.

#### Strategies to identify Conceptual Class

**1. Noun Phrase Identification**
Extract nouns and noun phrases from the requirements — each noun is a potential conceptual class. Eliminate irrelevant nouns; remaining become candidate classes.

**Example:**
> "A *student* enrolls in a *course* taught by a *teacher*."

→ `Student`, `Course`, `Teacher`

Note: Not all nouns become classes — some become attributes (e.g. `grade` → attribute of `Course`).

**2. Category List**
Use a predefined checklist of common real-world categories to identify conceptual classes systematically.

| Category | Example |
|---|---|
| Physical objects | `Book`, `Car` |
| Roles | `Admin`, `Customer` |
| Events/Transactions | `Payment`, `Order` |
| Places | `Library`, `Branch` |
| Descriptions | `ProductDescription` |

**Example:** For a library system → `Book` (physical), `Member` (role), `Loan` (transaction), `Branch` (place)

## 2.2 Association and Attributes

- Association: An Association is a relationship between classes (more precisely, instances of those classes) that indicates some meaningful connections between them.
- Multiplicity in Association: Defines how many instances of one class are associated with instances of another class in an association relationship.

| Notation | Meaning |
|---|---|
| `1` | Exactly one |
| `*` | Zero or more |
| `0..1` | Zero or one (optional) |
| `1..*` | One or more |
| `0..*` | Zero or more |

## 2.3 Representation of System Behaviours

### Contract

A formal description of what a system operation does, defined in terms of preconditions and postconditions.

**Contains:**

- **Operation** — name and parameters of the operation
- **Cross-reference** — related use cases that trigger this operation
- **Precondition** — what must be true before operation executes
- **Postcondition** — what must be true after operation executes

**Example:**

```
Operation:       enrollStudent(studentID, courseID)
Cross-reference: Enroll in Course (use case)
Precondition:    Student exists, Course exists, seat available
Postcondition:   Student enrolled, seat count decreased by 1
```
