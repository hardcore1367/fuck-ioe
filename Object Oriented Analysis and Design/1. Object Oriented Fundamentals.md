# Fundamentals

## 1.1. Introduction

### Technical Terms

- Class: Template or blueprint from which objects are created.
- Object: Instance of a class with their own state behavior and identity.
- Method:  Way to manipulate data of the respective object; defines the behavior of an object.
- Attribute: Property or data of an object that defines its state.

### Principles of OOP Approach

- Abstraction: Hiding complex implementation details and showing only essential features to the user. Focus on what an object does, not how.
- Encapsulation: Bundling data (attributes) and methods together inside a class, and restricting direct access using access modifiers (private, public, protected).
- Inheritance: Mechanism where a child class acquires properties and behaviors of a parent class, promoting code reusability.
- Polymorphism: Ability of an object to take many forms — same method behaves differently based on the object calling it (method overloading/overriding).
- Modularity: System is divided into independent, self-contained objects/classes that can be developed and maintained separately.

| OOP | Functional Programming |
|---|---|
| Programs are divided into objects | Programs are divided into functions |
| Importance is given to the data | Importance is given to the functions |
| Data and methods bundled together | Data and functions are separate |
| Supports inheritance, polymorphism, encapsulation | No inheritance, polymorphism or encapsulation, uses pure functions |
| Follows 'Bottom-Up Approach' | Follows 'Top-Down' Approach|
| e.g. Java, C++, Python | e.g. Haskell, Lisp, Erlang |

## 1.2. OOAD

OOA: Object Oriented Analysis is defined as an investigation of the problem and requirements to understand them better, rather than a solution.
OOD: Object Oriented Design is defined as a conceptual solution that fulfills the requirements, rather than it's implementation.
OOP: Object Oriented Programming is an implementation of OOD using a programming language, applying OOPs concept like class, object, inheritance, and polymorphism.

### How they are interrelated?

OOA, OOD, and OOP are three successive phases of object-oriented software development.

**OOA** investigates the problem domain and produces a conceptual model — identifying real-world objects and their relationships. This model becomes the input to **OOD**, which transforms it into a software blueprint by defining classes, methods, and interactions. Finally, **OOP** takes that design blueprint and implements it as actual working code using an OO language.

The same objects flow through all three phases — just with increasing technical detail. Analysis finds them, design structures them, programming builds them.

## 1.3 Object Oriented Model (BaSICS)

**1. Conceptual Model** — Describes real-world concepts and their relationships in the problem domain, independent of any technology or implementation.

**2. Structural Model** — Describes static structure of the system; shows classes, attributes, methods, and relationships like association, inheritance, and aggregation.

**3. Behavioral Model** — Describes dynamic behavior of the system; shows how objects interact and change state in response to events over time.

**4. Specification Model** — Describes interfaces and behavior of objects precisely; focuses on what objects do without specifying how they do it.

**5. Implementation Model** — Describes how design is mapped to actual code; shows physical organization and realization of software components.

## 1.4 Requirement Analysis

| Functional Requirements | Non-Functional Requirements |
|---|---|
| Defines **what** system should do | Defines **how well** system does it |
| Describes system behavior | Describes system quality/constraints |
| Directly related to user needs | Related to performance, security, reliability |
| Testable by input/output | Tested by benchmarks/standards |
| Changes affect functionality | Changes affect system quality |
| e.g. "User can login with email" | e.g. "Login must load within 2 sec" |

### Requirement Process  (MoVES)

**Requirement Elicitation** — Process of discovering and gathering requirements from stakeholders, customers, and users through techniques like interviews, questionnaires, observation, and prototyping.

**Requirement Specification** — Process of documenting the elicited requirements in a precise, clear, and formal way, usually as a Software Requirement Specification (SRS) document.

**Requirement Verification and Validation**
Verification — Process of checking that the specified requirements are complete, consistent, correct, and unambiguous — ensures requirements are well-defined before development. (are we building it right?)
Validation — Process of confirming that the documented requirements actually reflect the real needs of the user — ensures the right system is being built. (are we building the right thing?)

**Requirement Management** — Process of tracking, organizing, and controlling changes to requirements throughout the software development lifecycle to maintain consistency.

## 1.7 OOD Cycle

### 1. Rational Unified Process

RUP is a framework for software engineering processes. RUP is an Iterative and incremental approach to improving problem knowledge through consecutive revisions. It is an architecture-centric and use-case-driven approach that manages risk and is flexible to change.

#### Phases of RUP

```txt
Inception → Elaboration → Construction → Transition
```

#### 1. Inception

- Define scope, feasibility, cost
- Identify key use cases
- Go/no-go decision

#### 2. Elaboration

- Analyze problem domain
- Establish base architecture
- Resolve major risks

#### 3. Construction

- Actual coding and testing
- Multiple iterations, each giving working build

#### 4. Transition

- Deliver to users
- Beta testing, bug fixing, deployment

### 2. Agile Development

**Agile Development** is an iterative and incremental software development approach that delivers working software in short cycles called sprints or iterations. Unlike traditional waterfall model, agile embraces changing requirements even late in development, focusing on continuous customer collaboration and feedback throughout the process.

**Key Features:**

- Development in short iterations (1-4 weeks)
- Continuous customer feedback and collaboration
- Adaptive to changing requirements
- Working software delivered after each iteration
- Popular methods: Scrum, XP, Kanban
