Data Mining is the process of discovering meaningful patterns and insights from large datasets using statistical, machine learning and computational techniques. 

It helps organizations analyze historical data and make data-driven decisions.

## 1. KDD Process (Data Mining Process)

**KDD (Knowledge Discovery in Databases)** is an iterative, cyclic process of extracting useful knowledge from raw data. It has four stages that feed back into each other.

<img width="728" height="373" alt="image" src="https://github.com/user-attachments/assets/737538ee-8c3b-4643-a9e6-974ba9efbc97" />


### 1. Problem Definition
- The project objective is understood from a **business perspective** and translated into a data mining problem with a clear success criterion.
- Example: *"How can I sell more?"* → *"Which customers are most likely to purchase?"*

### 2. Data Gathering and Preparation
- **Data collection** — relevant data is identified and gathered from available sources.
- **Data quality assessment** — missing values, noise, and inconsistencies are identified.
- **Data preparation** — data is cleaned, transformed, and features are selected to build the final modeling dataset.
- These steps are iterative and performed in no fixed order.

### 3. Model Building and Evaluation
- Modeling techniques (classification, clustering, association, etc.) are applied to prepared data.
- Model **parameters are calibrated** through experimentation.
- The model is **evaluated** against the original business goal; if unsatisfactory, the process loops back to earlier stages.

### 4. Knowledge Deployment
- The validated model is **deployed** into a production system or business process.
- Outputs are presented to decision-makers as reports, dashboards, or automated recommendations.


## 2. Applications of Data Mining

- **Business** — market basket analysis, customer churn prediction.
- **Finance** — credit card fraud detection.
- **Healthcare** — disease outbreak prediction, drug interaction analysis.
- **Web** — clickstream analysis, recommendation systems.
- **Science** — astronomical data classification, genomics.

## 3. Data Warehouse

A **data warehouse** is a large, centralized repository of integrated data collected from multiple heterogeneous sources, organized to support management decision-making and analytical reporting. Unlike operational databases that handle day-to-day transactions, a data warehouse is designed for **read-heavy analytical queries** over historical data.


## 4. Key Features of a Data Warehouse
 
- **Subject-oriented** — organized around major subjects of the enterprise (sales, customers, products) rather than operational processes.
- **Integrated** — data from multiple heterogeneous sources (databases, flat files, external feeds) is cleaned, transformed, and merged into a consistent format.
- **Time-variant** — stores historical data over a long time horizon (5–10 years); every record has a time dimension for trend analysis.
- **Non-volatile** — data is loaded in bulk and not updated or deleted in real time; it is stable and read-only for analysts.


## 5. Data Warehouse Schemas

### Star Schema
The simplest and most common schema. A central **fact table** (containing measurable data like sales amount, quantity) is surrounded by multiple **dimension tables** (like Time, Product, Customer, Location). Dimension tables are **denormalized**.

<img width="1280" height="720" alt="image" src="https://github.com/user-attachments/assets/6fb6b381-e697-49b7-b83b-915610fe872f" />


- Easy to query, fast performance.
- Redundancy in dimension tables.

### Snowflake Schema
An extension of the star schema where dimension tables are **normalized** into multiple related tables, resembling a snowflake shape.
<img width="768" height="384" alt="image" src="https://github.com/user-attachments/assets/6cabf17b-3aca-4b18-b96b-c96df11c22d7" />

Example: In a star schema, the Product dimension has `product_id, product_name, category, supplier`. In a snowflake schema, `category` and `supplier` are split into separate tables linked by foreign keys.

- Reduces redundancy and saves storage.
- More complex queries; slower performance than star schema.


### Fact Constellation (Galaxy Schema)
Multiple fact tables share dimension tables. Used for complex enterprise data warehouses with multiple subject areas.

## Data Mart

A **data mart** is a smaller, subject-specific subset of a data warehouse focused on a particular department or business function. Example: a Sales Data Mart contains only sales-related data for the sales team, while the full warehouse contains data from all departments.

## Data Warehouse vs Database (RDBMS)

| Data Warehouse | Database (RDBMS) |
|---|---|
| Designed for analytical queries (OLAP) | Designed for transactional operations (OLTP) |
| Stores historical, integrated data | Stores current, operational data |
| Read-heavy; bulk loads | Read/write; frequent small transactions |
| Data from multiple sources | Data from a single application |
| Denormalized (star/snowflake schema) | Normalized (3NF) |
| Supports decision making | Supports day-to-day operations |

**Similarities:** both store structured data, use SQL for querying, support indexing, enforce data integrity, and are managed by a DBMS.

## Data Warehouse vs Data Mining

| Data Warehouse | Data Mining |
|---|---|
| Stores and organizes integrated data | Extracts patterns and knowledge from data |
| Output is queryable, structured data | Output is rules, models, patterns |
| Uses ETL pipelines | Uses ML/statistical algorithms |
| Goal: centralized data storage | Goal: knowledge discovery |
| Passive — answers known queries | Active — discovers unknown patterns |

A data warehouse provides the clean, integrated data on which data mining is performed.
