# 4. Association Analysis

## 1. Introduction and Basic Concepts

### 1.1 What is Association Analysis
Association analysis is a data mining technique used to discover interesting relationships hidden in large datasets. The relationships are expressed as **association rules** of the form $\{X_1, X_2, \dots, X_n\} \Rightarrow Y$, meaning that if a transaction contains all of $X_1, X_2, \dots, X_n$, then it is also likely to contain $Y$. For example, a rule might state that a customer who buys a computer also tends to buy a printer.

### 1.2 Market Basket Analysis
Market basket analysis (also called affinity analysis) is the most common application of association analysis. It studies the items that are purchased together in a single transaction, in order to understand purchasing behavior. Its practical uses include:
- Developing combo offers based on products frequently sold together.
- Placing associated products near each other in a store, or arranging an e-commerce catalog layout.
- Controlling inventory based on which products sell together.
- Designing better promotions and cross-selling/up-selling strategies.

Beyond retail, the same idea is applied in financial services (e.g., analyzing credit/debit card purchases) and telecom (e.g., analyzing calling patterns or value-added service usage).

### 1.3 Key Terminology
- **Itemset:** A collection of one or more items, e.g., {Milk, Bread, Diaper}. An itemset containing k items is called a **k-itemset**.
- **Frequent itemset:** An itemset whose support is greater than or equal to a minimum support threshold.
- **Association rule:** An implication of the form $X \Rightarrow Y$, where X and Y are itemsets, e.g., {Milk, Diaper} $\Rightarrow$ {Beer}.
- **Maximal frequent itemset:** A frequent itemset is maximal if none of its immediate supersets is also frequent.
- **Closed itemset:** An itemset is closed if none of its immediate supersets has the same support as the itemset itself.

### 1.4 Support and Confidence
Support and confidence are the two fundamental measures used to decide whether a discovered rule is worth considering.

**Support** indicates how frequently/popularly an itemset occurs across all transactions:
$$Support(A) = \frac{\text{Number of transactions containing } A}{\text{Total number of transactions}}$$
$$Support(A \Rightarrow B) = \frac{\text{Number of transactions containing both } A \text{ and } B}{\text{Total number of transactions}}$$

A high support means the itemset (or rule) is popular; this can be used, for example, to decide pricing/placement strategy for a popular product bundle.

**Confidence** measures the certainty or trustworthiness of a rule — of the transactions that contain A, how many also contain B:
$$Confidence(A \Rightarrow B) = \frac{\text{Number of transactions containing both } A \text{ and } B}{\text{Number of transactions containing } A}$$

Confidence is useful for product placement — e.g., if customers who buy dolls also buy high-margin candies with high confidence, candies can be placed near the dolls.

**Importance of minimum support/confidence thresholds:**
- If the minimum support (minsup) is set too high, rare but interesting itemsets (e.g., involving expensive products) may be missed.
- If minsup is set too low, the number of itemsets generated becomes huge and computation becomes very expensive.
- Rules are only considered interesting if they meet both the minsup and minconf thresholds.

### 1.5 Interestingness Measures — Lift
Support and confidence alone do not always indicate a truly useful rule, since a high confidence rule can occur simply because B is common overall. **Lift** corrects for this by comparing the rule's confidence against the baseline probability of the consequent:
$$Lift = \frac{P(Y \mid X)}{P(Y)}$$
- If lift = 1, occurrence of X and Y are independent — the rule provides no extra predictive value.
- If lift > 1, X and Y are positively associated, meaning transactions containing X are more likely to also contain Y — making the rule genuinely useful for prediction.
- Lift can also be computed as confidence divided by the unconditional probability of the consequent, or as support divided by the product of the individual supports of the antecedent and consequent.


## 2. Apriori Algorithm

### 2.1 Apriori Principle
The Apriori principle states: **supersets of a non-frequent itemset are also non-frequent**; equivalently, if an itemset is frequent, then all of its subsets must also be frequent. This anti-monotone property is the basis for pruning the search space — once an itemset is found to be infrequent, none of its supersets need to be checked, which drastically reduces the number of candidates to examine.

### 2.2 Apriori Algorithm — Steps
Apriori is a two-step approach: frequent itemset generation followed by rule generation. It uses a **level-wise search**, where frequent k-itemsets are used to generate candidate (k+1)-itemsets.
1. Scan the transaction database and count the support of each individual item; compare with minsup to obtain the frequent 1-itemsets.
2. Use a join step on the frequent itemsets of the current level to generate candidate itemsets for the next level.
3. Scan the database again to compute support for these candidates, and retain only those meeting minsup to get the frequent itemsets at that level.
4. Repeat steps 2–3, increasing the itemset size by one each time, until no new frequent itemsets can be generated.
5. From the frequent itemsets (from 2-itemsets onward), generate association rules that satisfy the minimum confidence threshold.

### 2.3 Reducing Computational Cost
Since the brute-force approach (listing all possible rules and computing support/confidence for each) is computationally very expensive, Apriori improves efficiency using these general strategies:
- **Reduce the number of candidates (M):** use pruning techniques based on the Apriori principle instead of considering all $2^d$ possible itemsets.
- **Reduce the number of transactions (N):** as itemset size increases, transactions that cannot possibly contain the larger itemset can be ignored.
- **Reduce the number of comparisons (M×N):** use efficient data structures such as hash trees, so that each transaction is matched only against relevant candidates rather than all of them.

### 2.4 Factors Affecting Complexity
- **Minimum support threshold:** lowering it increases the number of frequent itemsets and candidates, increasing computation.
- **Dimensionality (number of items):** more items require more space to store support counts, and increase computation/I-O cost.
- **Size of the database:** since Apriori scans the database multiple times (once per level), runtime grows with the number of transactions.
- **Average transaction width:** denser transactions (wider itemsets) increase the maximum length of frequent itemsets and the number of subsets to check.

### 2.5 Advantages and Disadvantages
**Advantages:**
- Simple and easy to understand, based on a clear principle (anti-monotonicity).
- Effectively prunes the search space, so it is much faster than the brute-force approach.
- Widely applicable to market basket–style transaction data.

**Disadvantages:**
- Requires multiple scans of the entire database, once for every itemset size — costly for large databases.
- Generates a very large number of candidate itemsets, especially when minsup is low or the data is dense.
- Computationally expensive compared to newer methods such as FP-Growth.


## 3. FP-Growth and FP-Tree

### 3.1 Why FP-Growth
FP-Growth (Frequent Pattern Growth) was developed to overcome Apriori's biggest weakness — the need for repeated database scans and massive candidate generation. FP-Growth mines frequent itemsets **without generating candidates at all**, using a divide-and-conquer strategy.

### 3.2 FP-Tree Structure
The core idea of FP-Growth is to compress the entire transaction database into a compact tree structure called the **FP-tree (Frequent Pattern Tree)**, which retains all the itemset association information needed for mining:
- The tree has a root node labeled "null".
- Each transaction is processed by first sorting its items in descending order of frequency, and then a branch is created (or an existing branch extended) representing that transaction.
- Common prefixes among transactions share the same branch, which is what allows the tree to compress the database.

### 3.3 FP-Growth Mining Process
Once the FP-tree is built, frequent itemsets are mined recursively:
1. Start from each frequent length-1 pattern as an initial "suffix" pattern.
2. For each suffix pattern, construct its **conditional pattern base** — the set of prefix paths in the FP-tree that co-occur with that suffix.
3. Build a smaller **conditional FP-tree** from this conditional pattern base.
4. Recursively mine this conditional FP-tree, growing longer frequent patterns by concatenating the suffix — this is why the method is called "pattern growth".

This approach transforms the problem of finding long frequent patterns into a search for shorter ones, which are then extended recursively.

### 3.4 Advantages of FP-Growth over Apriori
- **No candidate generation:** unlike Apriori, FP-Growth never generates and tests a huge number of candidate itemsets.
- **Fewer database scans:** the database is compressed into the FP-tree in essentially two scans, whereas Apriori needs one scan per itemset size.
- **Better performance on large/dense datasets:** since mining is performed on the compact tree rather than repeatedly scanning raw transactions, FP-Growth is generally much faster than Apriori.
- **Good selectivity:** the divide-and-conquer strategy focuses only on relevant conditional databases for each pattern, reducing search cost.

### 3.5 Disadvantage of FP-Growth
When the database is very large, it can become unrealistic to build and hold the complete FP-tree entirely in main memory.


## 4. Handling Categorical Attributes

### 4.1 Potential Issues
Association analysis was originally designed for simple transaction (binary/categorical) data, so applying it to real categorical attributes raises some issues:
- **Attributes with many possible values:** e.g., an attribute like "country" may have 200+ possible values, many of which individually have very low support.
  - *Solution:* aggregate the low-support attribute values into broader groups.
- **Highly skewed distribution of attribute values:** e.g., if 95% of visitors have "Buy = No", most discovered itemsets will trivially involve this dominant value and add little insight.
  - *Solution:* drop the highly frequent/dominant items before mining.

### 4.2 Technique for Handling Categorical Attributes
To apply standard association rule mining techniques (designed for binary/transaction data) to categorical attributes, each categorical attribute is transformed into a set of **asymmetric binary variables** — a new "item" is introduced for each distinct attribute-value pair (e.g., "Country = Nepal" becomes its own binary item). This is appropriate when the presence of a value is more meaningful than its absence (the two outcomes of the binary variable are not equally important).


## 5. Sequential and Subgraph Patterns

### 5.1 Sequential Patterns
Sequential pattern mining discovers frequently occurring **ordered** events or subsequences, unlike simple itemsets which have no order. Examples include web browsing sequences, or books issued in a library over time.
- A **sequence** is an ordered list of events, where an item can occur at most once within a single event, but can repeat across different events of the same sequence.
- Given a set of sequences and a minimum support threshold, sequential pattern mining finds all frequent subsequences; a frequent subsequence is also called a **sequential pattern**.
- A sequential pattern of length $l$ is called an **l-pattern**.
- Sequential pattern mining is computationally challenging because it can generate a combinatorially explosive number of candidate subsequences.
- Two common approaches: mining the full set of sequential patterns, or mining only the closed sequential patterns.
- **AprioriAll** is an Apriori-based algorithm for sequential pattern mining — it extends the Apriori principle (that subsets of a frequent itemset must be frequent) to sequences, generating candidate subsequences level by level and pruning those whose sub-sequences are not frequent, similar in spirit to how Apriori generates candidate itemsets.
- Applications: marketing, customer behavior analysis, prediction modeling.

### 5.2 Subgraph Patterns
Subgraph pattern mining extends the same frequent-pattern idea from itemsets/sequences to **graph-structured data**.
- Given a set of labeled graphs, a subgraph is considered **frequent** if its support (number of graphs containing it) is not less than a minimum support threshold.
- Frequent subgraphs are discovered by generating candidate substructures and then checking their frequency, similar to Apriori — this is called the **Apriori-like approach**, where frequent k-subgraphs are used to generate candidate (k+1)-subgraphs.
- **Challenges specific to graphs:**
  - Nodes may contain duplicate labels, complicating matching.
  - Defining support and confidence is less straightforward than for itemsets.
  - Extra structural constraints apply — e.g., it is usually assumed that frequent subgraphs must be connected.
- **Applications:** web mining (e.g., web graph structure), computational chemistry (chemical structures), bioinformatics, and spatial datasets.
