# Serial Interfacing

## 3.1 Serial Data Transmission

In serial data transmission (SDT), data is sent one bit at a time over the transmission channel. It is slower compared to parallel transmission.

### 1. Advantages of Serial over Parallel

- Longer data transmission is possible.
- Voltage loss is not a problem in SDT.
- It requires less number of wires and is cheaper to transmit data.
- Crosstalk is not an issue because there are fewer conductors compared to parallel transmission.

### 2. Synchronous Serial Data Transmission

Data is sent as a continuous stream of bits with no gaps between characters. A **shared clock signal** is used to synchronize the sender and receiver so both know exactly when each bit starts and ends. Data is grouped into **frames or blocks**, and special **sync characters** are sent at the beginning to establish synchronization. Because there are no start/stop bits per character, it is more efficient and faster than asynchronous transmission.

**Advantages:** Faster, less overhead
**Disadvantages:** Complex hardware, expensive, sensitive to clock loss


## 3. Asynchronous Serial Data Transmission

Data is sent **one character at a time** with no shared clock. Instead, each character is framed with a **start bit (0)** at the beginning and one or two **stop bits (1)** at the end. The receiver detects the start bit and uses its own internal clock to sample the remaining bits. Between characters, the line stays in **idle state (logic 1)**. This makes it simple to implement but adds per-character overhead.

Each character frame consists of:
- **1 start bit**
- **7 or 8 data bits**
- **1 parity bit**
- **1 or 2 stop bits**

**Advantages:** Simple, cheap, no clock sync needed
**Disadvantages:** Slower, not ideal for bulk transfer

## 4. Synchronous vs Asynchronous Serial Transmission

| Synchronous | Asynchronous |
|---|---|
| Uses a shared clock signal between sender and receiver | No shared clock; receiver uses its own internal clock |
| Data is sent as a continuous stream in blocks or frames | Data is sent one character at a time |
| Sync characters sent at the start to establish timing | Each character framed with start and stop bits |
| No start/stop bits per character, so less overhead | 1 start bit, 7/8 data bits, 1 parity bit, 1/2 stop bits per character |
| Faster and more efficient for large data transfer | Slower due to per-character framing overhead |
| Complex and expensive hardware required | Simple and cheap to implement |
| Loss of sync corrupts the entire block | Error in one character does not affect others |

## 3.2 SDU and Serialization
### Serial Data Unit (SDU)

A **Serial Data Unit (SDU)** is the framed packet of bits transmitted over a serial communication line. Instead of sending all bits simultaneously (parallel), serial communication sends **one bit at a time** on a single line. Each SDU contains control bits alongside the actual data to ensure the receiver can correctly identify, synchronize, and validate the data.

### Structure of an SDU

```
| Start | D0 | D1 | D2 | D3 | D4 | D5 | D6 | D7 | Parity | Stop |
```

- **Start bit** — always logic 0; signals the receiver that a new data unit is beginning. The line is idle at logic 1, so a HIGH→LOW transition marks the start.
- **Data bits** — the actual data, typically **8 bits** (can be 5–9). Sent **LSB first**.
- **Parity bit** — optional error detection bit. Can be **even parity** (total 1s are even) or **odd parity** (total 1s are odd).
- **Stop bit(s)** — always logic 1; signals the end of the SDU and returns the line to idle. Can be 1, 1.5, or 2 stop bits.

### Transmission Process

**Transmitter side:**

1. The CPU writes a byte of data into the **Transmit Data Register (TDR)**.
2. Data is loaded into the **Transmit Shift Register (TSR)**.
3. The UART appends a **start bit (0)** at the front and **stop bit(s) (1)** at the end, and optionally inserts a parity bit.
4. The TSR **shifts out one bit at a time** on the TX line, synchronized to the **baud rate clock**.
5. Once the last stop bit is sent, the **TDR Empty flag** is set — CPU can load the next byte.

### Receiving Process

**Receiver side:**

1. The RX line is monitored continuously. When a **HIGH→LOW transition** is detected, the start bit has arrived.
2. The receiver waits **half a bit period** and samples the line again to confirm it is a valid start bit (not noise).
3. It then samples the line at the **center of each subsequent bit period** using the baud rate clock to read each data bit into the **Receive Shift Register (RSR)**.
4. After all data bits are received, the **parity bit is checked** (if enabled).
5. The stop bit is verified — if it is not logic 1, a **framing error** is flagged.
6. The assembled byte is transferred to the **Receive Data Register (RDR)** and a **data ready flag** is set to notify the CPU.

### Key Terms

| **Term** | **Meaning** |
|---|---|
| **Baud rate** | Number of signal changes (bits) per second |
| **Framing error** | Stop bit not detected as logic 1 |
| **Parity error** | Received parity does not match computed parity |
| **Overrun error** | New data arrives before CPU reads previous RDR |
| **TDR** | Transmit Data Register — CPU writes byte here |
| **RSR / TSR** | Shift registers that do the actual bit-by-bit shifting |


## 3.3 Errors in Serial Data Transmission

- **Framing Error** — Occurs when the receiver expects a stop bit (logic 1) but receives a 0 instead. This means the receiver's clock has drifted out of sync with the sender, causing the frame boundaries to misalign.

- **Break Error** — Occurs when the transmission line stays at logic 0 (space/low) continuously for longer than a full character frame duration. This is usually caused by a broken wire, disconnected cable, or a deliberate break signal sent by the transmitter.

- **Overrun Error** — Occurs when the sender transmits the next character before the receiver has finished reading the previous one from its buffer. The old data gets overwritten before it is processed, resulting in data loss.

- **Parity Error** — Occurs when the number of 1s in the received data bits does not match the expected parity (odd or even). This indicates that one or more bits were flipped or corrupted during transmission due to noise on the line.

## 3.4 Error Detection in Serial Data Transmission

### 1. Parity Check

A single extra bit (parity bit) is added to each data frame so that the total number of 1s follows a fixed rule — either always even or always odd.

**Mechanism:**
- Sender counts the number of 1s in the data bits.
- For **even parity**: parity bit is set to make total 1s even.
- For **odd parity**: parity bit is set to make total 1s odd.
- Example: data = `1011001` (four 1s) → even parity bit = `1011001 0`, odd parity bit = `1011001 1`.
- Receiver counts 1s in the received data + parity bit.
- If the count violates the agreed parity rule → error detected.
- Limitation: if **two bits** flip, the parity count stays correct and the error goes undetected.


### 2. Checksum

 The sender computes a sum over all data blocks and sends it alongside the data. The receiver recomputes and compares.

**Mechanism:**
- Sender divides the data into fixed-size blocks (e.g., 8-bit or 16-bit segments).
- All blocks are added together using binary addition.
- If the sum overflows, the overflow bits are **wrapped around** and added back (called end-around carry).
- The **one's complement** of the final sum is taken — this is the checksum.
- Checksum is appended to the data and transmitted.
- Receiver adds all received blocks including the checksum.
- If the result is all 1s (i.e., one's complement is 0) → no error.
- If not → error detected.
- Limitation: if errors cancel each other out across blocks, the checksum may still appear valid.

Data blocks (8-bit each):
```
  10110011
+ 01101100
----------
 100011111  → overflow, wrap carry → 00011111 + 1 = 00100000
```
One's complement of `00100000` = **`11011111`** → this is the checksum, appended and sent.

Receiver adds all blocks including checksum:
```
  10110011
+ 01101100
+ 11011111
----------
 111111111 → one's complement = 00000000 ✓ → no error
```
If result ≠ 0 → **error detected**


### 3. Cyclic Redundancy Check (CRC)

The data is treated as a binary polynomial and divided by a predefined **generator polynomial**. The remainder of this division is sent with the data.

**Mechanism:**
- Sender and receiver agree on a **generator polynomial** (e.g., CRC-16, CRC-32).
- Sender appends zeros to the data (equal to the degree of the generator polynomial).
- The data is then **divided by the generator polynomial** using XOR-based binary division (not normal arithmetic division).
- The **remainder** from this division is the CRC value, which replaces the appended zeros.
- The data + CRC is transmitted.
- Receiver performs the same XOR division on the received data using the same generator.
- If remainder = **0** → data is error-free.
- If remainder ≠ **0** → error detected.
- Much stronger than parity or checksum — can detect all single-bit errors, all burst errors up to the length of the generator polynomial, and most longer burst errors.


Data: `1101011011`, Generator: `10011`

- Append 4 zeros (degree of generator = 4): `11010110110000`
- XOR divide `11010110110000` by `10011`:

```
11010110110000 ÷ 10011
-----------------------
11010 XOR 10011 = 01001
 10011 XOR 10011 = 00000
  ...continue until remainder = 1110
```
CRC = `1110` → transmitted: `1101011011 1110`

Receiver divides `11010110111110` by `10011`:
- Remainder = `0000` ✓ → **no error**
- Remainder ≠ 0 → **error detected**

## 3.5 Bit Rate and Baud Rate

**Bit Rate** — The number of **bits transmitted per second** (bps). It measures how much actual data is being sent over the channel.

**Baud Rate** — The number of **signal changes (symbols) per second** (Bd). Each symbol may carry one or more bits depending on the encoding scheme.

**Relationship:**
> Bit Rate = Baud Rate × Number of bits per symbol

- If each symbol carries **1 bit** → Bit Rate = Baud Rate
- If each symbol carries **2 bits** → Bit Rate = 2 × Baud Rate

**Example:**
A modem operates at 2400 baud with 4 signal levels (2 bits per symbol):
Bit Rate = 2400 × 2 = **4800 bps**

**Key Difference:**

| Bit Rate | Baud Rate |
|---|---|
| Measures bits per second | Measures symbols per second |
| Depends on encoding | Property of the physical channel |
| Can be higher than baud rate | Always fixed to signal transitions |


## 3.6 Data Communication Modes

### 1. Simplex
Data flows in **one direction only**. The sender can only send and the receiver can only receive — roles are fixed and cannot be reversed. The entire channel bandwidth is used for transmission in that single direction.
**Example:** TV broadcast, keyboard to computer.


### 2. Half Duplex
Data flows in **both directions, but not simultaneously**. Either side can send, but only one can transmit at a time. When one device is sending, the other must wait. The channel is shared and switches direction as needed.
**Example:** Walkie-talkie — one person talks while the other listens, then they switch.

### 3. Full Duplex
Data flows in **both directions simultaneously**. Both devices can send and receive at the same time. This is achieved either by using two separate channels or by splitting the bandwidth of a single channel into two.
**Example:** Telephone call — both people can speak and hear at the same time.

## 3.7 RS-232C Standard

RS-232C is a serial communication standard defined by the **Electronic Industries Association (EIA)**. It specifies the interface between **DTE (Data Terminal Equipment)** — such as a computer or terminal — and **DCE (Data Communications Equipment)** — such as a modem. It was originally designed for connecting computers to modems for long-distance communication over telephone lines, but has since been widely used for short-distance serial communication between all kinds of devices.

**Features:**
- Supports **full duplex** communication (simultaneous send and receive)
- Uses a **DB-25 (25-pin)** or **DB-9 (9-pin)** connector
- Maximum cable length of approximately **15 meters (50 feet)** at standard speeds
- Uses **single-ended (unbalanced)** signaling — each signal uses one wire referenced to a common ground
- Communication is **asynchronous** — data is framed with start and stop bits
- Normally ±12V levels are used `(+12V for 0)` and `(-12V for 1)`

### Signal Functions

| Signal | Full Name | Direction | Function |
|---|---|---|---|
| **TxD** | Transmitted Data | DTE → DCE | Carries serial data bits from DTE to DCE |
| **RxD** | Received Data | DCE → DTE | Carries serial data bits from DCE to DTE |
| **DTR** | Data Terminal Ready | DTE → DCE | DTE signals it is powered on and ready to communicate |
| **DSR** | Data Set Ready | DCE → DTE | DCE confirms it is powered on and ready to communicate |
| **RTS** | Request to Send | DTE → DCE | DTE requests permission from DCE to begin transmitting |
| **CTS** | Clear to Send | DCE → DTE | DCE grants permission to DTE to start transmitting |
| **DCD** | Data Carrier Detect | DCE → DTE | DCE signals that it has detected a valid carrier from the remote modem |
| **RI** | Ring Indicator | DCE → DTE | DCE signals that the telephone line is ringing (incoming call detected) |
| **DSRD** | Data Signal Rate Detector | DCE → DTE | DCE indicates which of two pre-selected data signaling rates is in use |

### Working Principle
RS-232C uses **negative logic (inverted) voltage levels** for data signals:
- Logic 1 (MARK) → **-3V to -15V**
- Logic 0 (SPACE) → **+3V to +15V**
- The region between -3V and +3V is undefined (transition zone)

For **control signals** (DTR, DSR, RTS, CTS), the logic is positive:
- Signal **asserted (ON)** → **+3V to +15V**
- Signal **deasserted (OFF)** → **-3V to -15V**

The DTE and DCE are connected via the RS-232C interface. Before any data is exchanged, both sides go through a **hardware handshaking sequence** using control lines to confirm readiness. Once both sides confirm they are ready, data is transmitted serially — one bit at a time — over the **TXD** and **RXD** lines at the agreed baud rate.


### Handshaking Sequence
Handshaking ensures both devices are ready before data transfer begins:

1. DTE powers on and asserts **DTR** → signals to DCE: *"I am ready"*
2. DCE powers on and asserts **DSR** → signals to DTE: *"I am also ready"*
3. DTE has data to send, asserts **RTS** → signals to DCE: *"I want to send data"*
4. DCE checks its status, asserts **CTS** → signals to DTE: *"Go ahead and send"*
5. DTE begins transmitting data over **TXD**
6. DCE receives data over its **RXD** line
7. If DCE needs to send data back, the process is mirrored in the reverse direction

# RS-232 Connection Modes — Complete Notes

### 1. Simple (Simplex) Connection

- Data flows in **only one direction** — either from sender to receiver, never both.
- No return data channel exists.
- Used when only one-way communication is needed (e.g., sensor sending data to a logger).

---

#### 1.1 DTE → DCE (Simplex)

The DTE (e.g., a computer) transmits data to the DCE (e.g., a modem). The DCE only listens; it never sends back.

**Pin Connections:**

| DTE Pin | Signal | DCE Pin |
|---------|--------|---------|
| TX (Pin 3) | → | RX (Pin 2) |
| GND (Pin 5) | — | GND (Pin 5) |

**Steps:**
1. Connect DTE **TX** to DCE **RX**.
2. Connect both **GND** pins together.
3. Configure DTE serial port (baud rate, data bits, parity, stop bits).
4. DCE is set to receive-only mode.
5. DTE begins transmitting data frames serially.
6. DCE receives and processes the incoming bits.

> **Note:** No handshake or acknowledgment is used. If DCE is not ready, data is lost.


#### 1.2 DCE → DTE (Simplex)

The DCE transmits data to the DTE. The DTE only receives. Used in broadcast or monitoring scenarios.

**Pin Connections:**

| DCE Pin | Signal | DTE Pin |
|---------|--------|---------|
| TX (Pin 2) | → | RX (Pin 3) |
| GND (Pin 5) | — | GND (Pin 5) |

**Steps:**
1. Connect DCE **TX** to DTE **RX**.
2. Connect both **GND** pins together.
3. Configure DCE to transmit continuously or on trigger.
4. DTE serial port is opened in receive-only mode with matching baud/parity settings.
5. DTE reads incoming data from its RX buffer.

> **Note:** DTE cannot respond or acknowledge. Data integrity is the sender's responsibility.


### 2. Half-Duplex Connection

- Data travels in **both directions**, but **not simultaneously**.
- Only one device transmits at a time; the other listens.
- Direction is controlled manually or by a protocol (e.g., push-to-talk style).
- Uses the same channel/wire for both directions, switched by RTS/CTS.

**Pin Connections:**

| DTE Pin | Signal | DCE Pin |
|---------|--------|---------|
| TX (Pin 3) | ↔ | RX (Pin 2) |
| RX (Pin 2) | ↔ | TX (Pin 3) |
| RTS (Pin 7) | → | CTS (Pin 8) |
| GND (Pin 5) | — | GND (Pin 5) |

**Steps:**
1. Cross-connect TX and RX between DTE and DCE.
2. Connect DTE **RTS** to DCE **CTS** for direction control.
3. Connect both **GND** pins.
4. When DTE wants to transmit:
   - DTE raises **RTS** (Request To Send).
   - DCE responds by raising **CTS** (Clear To Send).
   - DTE transmits data via TX.
5. DTE lowers RTS when transmission is complete.
6. DCE is now free to transmit in the other direction.
7. Devices take turns — only one transmits at any given moment.

> **Note:** A turnaround delay exists between direction changes. Collision avoidance is handled by protocol.

---

### 3. Full-Duplex Connection

- Data travels in **both directions simultaneously**.
- Separate physical lines are used for TX and RX, so there is no collision.
- Most modern RS-232 communication operates in full-duplex mode.
- Handshaking signals (RTS/CTS, DTR/DSR) may be used for flow control.

**Pin Connections:**

| DTE Pin | Signal | DCE Pin |
|---------|--------|---------|
| TX (Pin 3) | → | RX (Pin 2) |
| RX (Pin 2) | ← | TX (Pin 3) |
| RTS (Pin 7) | → | CTS (Pin 8) |
| CTS (Pin 8) | ← | RTS (Pin 7) |
| DTR (Pin 4) | → | DSR (Pin 6) |
| DSR (Pin 6) | ← | DTR (Pin 4) |
| GND (Pin 5) | — | GND (Pin 5) |

**Steps:**
1. Cross-connect TX ↔ RX between DTE and DCE.
2. Connect RTS of each device to CTS of the other (cross).
3. Connect DTR of each device to DSR of the other (cross).
4. Connect GND on both sides.
5. Both devices configure the same baud rate, data bits, parity, and stop bits.
6. DTE asserts **DTR** to signal it is ready; DCE responds with **DSR**.
7. DTE asserts **RTS**; DCE confirms with **CTS** — link is live.
8. Both devices can now **simultaneously** send and receive data.
9. Hardware flow control (RTS/CTS) pauses transmission if the receiver's buffer is full.

> **Note:** Full-duplex is the standard mode for DTE–DCE communication (e.g., PC connected to a modem).


### 4. Null Modem Connection

A **Null Modem** cable connects two **DTE devices directly** (e.g., two computers) **without any DCE** (modem) in between. Because both ends are DTE, the wiring is crossed so each device "thinks" it is talking to a modem.


#### 4.1 Null Modem — Without Handshake

The simplest null modem. Only TX, RX, and GND are connected. No flow control signals are used. The software must handle data flow (software flow control / XON-XOFF, or simply no flow control).

**Pin Connections (DB-9):**

| DTE-A Pin | Signal | DTE-B Pin |
|-----------|--------|-----------|
| TX (Pin 3) | → | RX (Pin 2) |
| RX (Pin 2) | ← | TX (Pin 3) |
| GND (Pin 5) | — | GND (Pin 5) |

> RTS, CTS, DTR, DSR pins are either **left unconnected** or **looped back** on the same device to satisfy software that checks them.

**Optional Loopback (to satisfy handshake-checking software):**

| DTE-A | Looped |
|-------|--------|
| RTS (Pin 7) | → CTS (Pin 8) on same side |
| DTR (Pin 4) | → DSR (Pin 6) on same side |

**Steps:**
1. Cross TX of DTE-A to RX of DTE-B.
2. Cross TX of DTE-B to RX of DTE-A.
3. Connect GND on both sides.
4. (Optional) On each device, loop RTS→CTS and DTR→DSR locally.
5. Set the same baud rate, data bits, parity, and stop bits on both DTE devices.
6. Disable hardware flow control in software settings (use "None" or "XON/XOFF").
7. Both devices can now exchange data directly.

> **Use Case:** Quick file transfer between two computers, serial console connections, debugging.


#### 4.2 Null Modem — With Handshake

**Theory:**
Full null modem with hardware handshaking. The control signals (RTS, CTS, DTR, DSR) are **crossed between the two DTE devices** so that each device's output signal drives the other device's input signal. This allows proper flow control.

**Pin Connections (DB-9):**

| DTE-A Pin | Signal | DTE-B Pin |
|-----------|--------|-----------|
| TX (Pin 3) | → | RX (Pin 2) |
| RX (Pin 2) | ← | TX (Pin 3) |
| RTS (Pin 7) | → | CTS (Pin 8) |
| CTS (Pin 8) | ← | RTS (Pin 7) |
| DTR (Pin 4) | → | DSR (Pin 6) + DCD (Pin 1) |
| DSR (Pin 6) | ← | DTR (Pin 4) |
| DCD (Pin 1) | ← | DTR (Pin 4) |
| GND (Pin 5) | — | GND (Pin 5) |

**Steps:**
1. Cross TX ↔ RX between DTE-A and DTE-B.
2. Cross **RTS of A → CTS of B** and **RTS of B → CTS of A**.
3. Cross **DTR of A → DSR of B** (and DCD of B if needed).
4. Cross **DTR of B → DSR of A** (and DCD of A if needed).
5. Connect GND on both sides.
6. Configure both devices with identical serial parameters (baud, parity, data bits, stop bits).
7. Enable hardware flow control (RTS/CTS) in software on both devices.
8. DTE-A asserts **DTR** → DTE-B sees it on **DSR** (device ready signal received).
9. DTE-A asserts **RTS** → DTE-B sees it on **CTS** (clear to receive).
10. Data transmission begins. If DTE-B's buffer fills up, it drops **RTS**, which DTE-A sees as **CTS low** → DTE-A pauses sending.
11. When DTE-B is ready again, it reasserts **RTS** → DTE-A's **CTS** goes high → transmission resumes.

> **Use Case:** Reliable PC-to-PC serial communication, legacy data transfer, industrial device interfacing.

## RS-232 vs RS-422 vs RS-423 Comparison


### Signaling Method

**RS-232** — Single-ended (unbalanced). One wire per signal measured against a common ground. Any noise on the line corrupts the signal directly. No common-mode rejection.

**RS-422** — Differential (balanced). Two wires per signal (A+ and B−). The receiver measures the voltage difference between the two wires. Noise induced on both wires cancels out. Excellent common-mode rejection.

**RS-423** — Single-ended (improved). Same topology as RS-232 (one wire + GND) but uses lower voltages and sharper signal transitions. Better noise immunity than RS-232 but no differential rejection.


### Voltage Levels

### RS-232
| State | Voltage |
|-------|---------|
| Logic 1 (MARK) | −3V to −15V (max −25V) |
| Logic 0 (SPACE) | +3V to +15V (max +25V) |
| Undefined zone | −3V to +3V |
| Receiver threshold | ±3V |

Uses inverted logic — logic 1 is negative voltage. Wide undefined zone limits speed.

### RS-422
| State | Differential Voltage (A − B) |
|-------|------------------------------|
| Logic 1 | less than −0.2V |
| Logic 0 | greater than +0.2V |
| Max driver output | ±6V |
| Receiver threshold | ±0.2V |
| Common-mode range | −7V to +7V |

Very small threshold (±0.2V) allows fast transitions and high data rates.

### RS-423
| State | Voltage |
|-------|---------|
| Logic 1 | −3.6V to −6V |
| Logic 0 | +3.6V to +6V |
| Max driver output | ±6V |
| Receiver threshold | ±0.2V |

Lower voltages than RS-232, sharper transitions. Still single-ended.


### Maximum Cable Distance

| Standard | Max Distance | Condition |
|----------|-------------|-----------|
| RS-232 | 15 m (50 ft) | At 20 kbps |
| RS-423 | 90 m (300 ft) | At low baud rates |
| RS-422 | 1200 m (4000 ft) | At low baud rates |

RS-232 is limited by high voltage swing and susceptibility to noise over long cables. RS-422 reaches 1200 m because differential signaling rejects ground noise and the small ±0.2V threshold survives heavy signal attenuation.


### Maximum Baud Rate

| Standard | Max Baud Rate (EIA spec) | Practical short-distance |
|----------|--------------------------|--------------------------|
| RS-232 | 20 kbps | ~115.2 kbps |
| RS-423 | 100 kbps | ~100 kbps |
| RS-422 | 10 Mbps | 10 Mbps |

### RS-422 Speed vs Distance Trade-off
| Distance | Max Speed |
|----------|-----------|
| 12 m | 10 Mbps |
| 120 m | 1 Mbps |
| 1200 m | 100 kbps |

Speed drops with distance due to cable capacitance degrading signal rise time.


### Drivers and Receivers per Line

| Standard | Drivers | Receivers | Topology |
|----------|---------|-----------|----------|
| RS-232 | 1 | 1 | Point-to-point only |
| RS-423 | 1 | 10 | Multidrop |
| RS-422 | 1 | 10 | Multidrop |

RS-232 is strictly point-to-point. RS-422 and RS-423 allow one transmitter to drive up to 10 receivers on the same line.


### Differential Signaling Explained (RS-422)

1. Transmitter sends the signal on two wires with inverted polarity — if A = +3V then B = −3V.
2. External noise induces the same voltage on both wires simultaneously (common-mode noise).
3. Receiver computes A − B:
   - Signal: (+3V) − (−3V) = +6V ✓
   - With noise: (+3V + n) − (−3V + n) = +6V (noise cancels) ✓
4. Receiver only responds to differential voltage ≥ ±0.2V, ignoring common-mode components.

Common-mode rejection range is ±7V — meaning ground potential differences of up to 7V between devices are tolerated without data corruption.

### Compatibility

- RS-422 and RS-423 receiver thresholds are both ±0.2V — compatible with each other.
- RS-423 output can drive an RS-232 receiver in some cases (voltage levels partially overlap).
- RS-232 and RS-422 are **not directly compatible** — a converter IC (e.g. MAX490) is required.
- All three use the same asynchronous serial framing (start bit, data bits, parity, stop bits) — only the electrical layer differs.

## 3.9 USB

USB is a standardized serial communication interface designed to replace older inconsistent ports (RS-232, parallel, PS/2) with a single, simple, universal interface. Key features:
- **Hot swappable** — devices can be connected/disconnected without restarting
- **Plug and play** — OS automatically detects and configures devices
- Supports up to **127 devices** simultaneously through hubs
- Provides **power delivery** over the cable for low-power devices
- Uses a **host-controlled** architecture — the host (PC) manages all communication

## USB Standards

#### USB 1.0
The first official release of the USB standard. It introduced two speed modes — Low Speed and Full Speed — but suffered from bugs and compatibility issues and was not widely adopted.
- **Low Speed — 1.5 Mbps** (mice, keyboards)
- **Full Speed — 12 Mbps** (printers, audio)
- Had host controller compatibility issues
- Rarely seen in actual consumer products


#### USB 1.1
A revised and corrected version of USB 1.0 with the same speeds. Fixed bugs and clarified the specification, making it reliable enough for mass adoption.
- Same speeds: **Low Speed 1.5 Mbps, Full Speed 12 Mbps**
- First version to be **widely used in consumer products**
- Backward compatible with USB 1.0


#### USB 2.0 — Hi-Speed
A major upgrade that introduced Hi-Speed mode, 40 times faster than USB 1.1. Used the same connector as USB 1.x and dominated the market for over a decade.
- **480 Mbps** theoretical, ~**60 MB/s** actual throughput
- Same USB-A connector as USB 1.x — fully backward compatible
- Still present on virtually all modern devices


#### USB 3.0 — SuperSpeed
Introduced SuperSpeed mode with a second set of conductors inside the cable alongside USB 2.0 wires, enabling high speed while staying backward compatible.
- **5 Gbps** theoretical, ~**400 MB/s** actual throughput
- Ports marked in **blue** to distinguish from USB 2.0
- Backward compatible with USB 2.0 and 1.x
- Later renamed **USB 3.2 Gen 1x1**


#### USB 3.1 — SuperSpeed+
Doubled the speed of USB 3.0 by introducing improved encoding that reduced protocol overhead.
- **10 Gbps** theoretical, ~**1 GB/s** actual throughput
- Improved **128b/132b encoding** — less overhead than USB 3.0
- Backward compatible with all previous versions
- Later renamed **USB 3.2 Gen 2x1**


#### USB 3.2 — SuperSpeed+ Dual Lane
Achieved 20 Gbps by running two data lanes simultaneously, requiring USB-C to support dual lane operation.
- **20 Gbps** theoretical, ~**2 GB/s** actual throughput
- Uses **two lanes of 10 Gbps** simultaneously
- Requires **USB-C cable** — cannot work over USB-A
- Backward compatible with all previous versions
- Later renamed **USB 3.2 Gen 2x2**

### Speed Comparison Table

| Standard | Official Name | Speed | Actual Throughput | Connector |
|---|---|---|---|---|
| USB 1.0 | Low/Full Speed | 1.5 / 12 Mbps | ~1 MB/s | USB-A |
| USB 1.1 | Low/Full Speed | 1.5 / 12 Mbps | ~1 MB/s | USB-A |
| USB 2.0 | Hi-Speed | 480 Mbps | ~60 MB/s | USB-A |
| USB 3.0 | SuperSpeed (Gen 1x1) | 5 Gbps | ~400 MB/s | USB-A / USB-C |
| USB 3.1 | SuperSpeed+ (Gen 2x1) | 10 Gbps | ~1 GB/s | USB-A / USB-C |
| USB 3.2 | SuperSpeed+ (Gen 2x2) | 20 Gbps | ~2 GB/s | USB-C only |


### Compatibility
- All USB versions are **backward compatible** — a USB 3.0 device plugged into a USB 2.0 port will work at USB 2.0 speed
- A USB 2.0 device plugged into a USB 3.0 port will work at USB 2.0 speed
- USB 3.2 Gen 2x2 (dual lane) **requires USB-C** — cannot work with USB-A connectors
- Speed is always limited by the **slower of the two ends** (device or host port)

### USB Components

### USB Host
The USB host is the **master controller** of the entire USB system. It initiates and manages all communication on the USB bus — no data transfer happens unless the host allows it. Every USB system has exactly **one host controller**.
- Controls all data transfers on the bus
- Detects device connection and disconnection
- Assigns **addresses (1–127)** to each connected device
- Provides **5V power** through the bus to connected devices
- Manages bandwidth allocation across all devices
- Example: PC, laptop, smartphone acting as host


### USB Hub
A USB hub is an **expander** that multiplies one USB port into several. It sits between the host and devices, forwarding data transparently.
- Extends the number of available USB ports
- Transparent to the host — host still manages all devices directly
- Can be **daisy-chained** up to 5 tiers deep
- **Bus-powered** hubs draw power from the USB bus; **self-powered** hubs have their own adapter
- Self-powered hubs supply full 500mA per port; bus-powered hubs have limited power budget
- Maximum **127 devices** total across the entire USB tree

### USB Device
A USB device is any **peripheral** that connects to the USB bus and responds to host requests. It cannot initiate communication on its own.
- Only responds when addressed by the host
- Assigned a unique **address (1–127)** during enumeration
- Describes itself using **descriptors** (type, speed, power needs)
- Can have multiple **endpoints** — logical channels for different data types
- Draws up to **500mA (USB 2.0)** or **900mA (USB 3.0)** from the bus
- Examples: flash drives, keyboards, webcams, printers


### USB Interface Chips
Dedicated ICs that handle the **low-level USB protocol**, signal timing, and electrical interfacing between a microcontroller and the USB bus.

**USB Host Controller Chips** — manage enumeration and data transfer on the host side.
- Example: Intel EHCI, xHCI (built into modern chipsets)

**USB Device Controller Chips** — used in peripherals to handle enumeration and endpoint management.
- Example: Cypress CY7C68013 (USB 2.0 microcontroller)

**USB Hub Controller Chips** — implement hub logic, repeat signals, manage ports.
- Example: SMSC USB2514 (4-port USB 2.0 hub controller)

**USB-to-Serial Bridge Chips** — add USB connectivity to devices that use UART/SPI/I2C internally.
- **FTDI FT232RL** — USB to UART, widely used for microcontroller programming
- **CP2102** — USB to UART, common in ESP32/Arduino boards
- **CH340** — low-cost USB to UART, common in budget Arduino clones

### USB Packets and Protocol

USB communication is based on **packets** — small structured units of data exchanged between the host and devices. All transfers are initiated by the host and consist of a sequence of packets. Every packet begins with a **PID (Packet Identifier)** which identifies the packet type.

### Common Packet Fields

- **SYNC** — Every packet starts with a sync field to allow the receiver to synchronize its clock with the transmitter
- **PID (Packet Identifier)** — Identifies the type of packet (token, data, handshake, SOF). It is sent as 4 bits followed by its 4-bit complement for error checking
- **ADDR (Address)** — 7-bit field identifying the target device (1–127)
- **ENDP (Endpoint)** — 4-bit field identifying which endpoint on the device this packet is directed to
- **DATA** — The actual payload, variable length depending on transfer type
- **CRC (Cyclic Redundancy Check)** — Error detection field appended at the end; CRC5 for token packets, CRC16 for data packets
- **EOP (End of Packet)** — Signals the end of the packet transmission

### Packet Types

#### Token Packet
Sent by the **host** to initiate a transaction. Tells the device what kind of transfer is about to happen and which endpoint to use.
- **IN token** — host requests data from the device
- **OUT token** — host wants to send data to the device
- **SETUP token** — host wants to send a control request to the device
- Fields: SYNC + PID + ADDR + ENDP + CRC5 + EOP

#### Data Packet
Carries the **actual payload** — either from host to device or device to host depending on the direction established by the token packet.
- **DATA0** and **DATA1** alternate with each successful transfer — this is called **data toggling** and is used to detect lost or duplicate packets
- Fields: SYNC + PID + DATA + CRC16 + EOP
- Maximum payload size depends on speed: 8 bytes (Low Speed), 64 bytes (Full Speed), 1024 bytes (High Speed)

#### Handshake Packet
Sent to **acknowledge** the result of a data transfer. Indicates whether the transfer was successful or failed.
- **ACK** — data received successfully, no errors
- **NAK** — device is busy and cannot accept or send data right now; host should retry
- **STALL** — device encountered an error or the request is not supported; host must intervene
- Fields: SYNC + PID + EOP (no address or data — just a response code)

#### SOF Packet (Start of Frame)
Sent by the **host every 1ms (Full Speed) or 125µs (High Speed)** to keep all devices synchronized and to mark the beginning of a new frame.
- Keeps all devices on the bus time-synchronized
- Contains a **frame number** that increments with each SOF
- Devices use SOF to maintain timing for isochronous transfers
- Fields: SYNC + PID + Frame Number + CRC5 + EOP

#### PRE Packet (Preamble)
Sent by the **host** to notify a hub that the next packet is intended for a **Low Speed device** connected downstream. Since the USB bus normally operates at Full Speed, Low Speed devices cannot directly receive Full Speed signals. The PRE packet tells the hub to switch its downstream ports to Low Speed mode for the next transaction.
- Only sent by the host, never by devices
- Used exclusively to enable **Low Speed transactions through a hub**
- After receiving PRE, the hub enables Low Speed signaling on its downstream ports
- Fields: SYNC + PID (PRE) + EOP
- Not used in USB 2.0 Hi-Speed — replaced by the **Split Transaction** mechanism

### Typical Transaction Flow
A single USB transaction consists of three packets in sequence:
```
Host sends TOKEN → Host/Device sends DATA → Device/Host sends HANDSHAKE
```
Example — host reading data from a device:
```
1. Host sends IN token    (which device and endpoint)
2. Device sends DATA0     (the actual data)
3. Host sends ACK         (data received successfully)
```

## 3.9 USB OTG (On-The-Go)

USB OTG is an extension of the USB standard that allows a device to act as **both host and device** depending on the situation. In standard USB, the host is always a PC and the device is always a peripheral. OTG removes this restriction, allowing two peripherals to communicate directly without a PC.

- A single OTG device can switch between **host role** and **device role**
- The device connected to the **micro/mini USB ID pin** determines the initial role — ID pin grounded = host, ID pin floating = device
- Uses **HNP (Host Negotiation Protocol)** to swap host/device roles during a session
- Uses **SRP (Session Request Protocol)** to allow the device to request the host to start a session and provide power
- Commonly used in smartphones, tablets, and cameras
- Allows direct device-to-device communication — e.g. phone reading a USB flash drive, camera transferring photos to a printer
