## 1. Approaches to Knowledge Representation

### 1. Logical Representation
Uses formal logic — propositional logic or FOPL — to represent facts and rules as logical statements. An inference engine applies rules like resolution, forward chaining, or backward chaining to derive new knowledge.

- Example: ∀x: Human(x) → Mortal(x)
- Best for problems requiring formal, precise, and provable reasoning

### 2. Semantic Networks
Represents knowledge as a directed graph where nodes represent objects or concepts and edges represent relationships between them.

- Example: "Dog is an Animal" represented as Dog →(is-a)→ Animal
- Best for representing hierarchical relationships and inheritance

### 3. Frames
Represents knowledge as structured objects called frames, where each frame has slots (attributes) and fillers (values). Similar to classes in OOP.

- Example: Frame for Car — slots: color, brand, speed, owner
- Best for representing objects with many properties and default values

### 4. Production Rules
Represents knowledge as IF-THEN rules. An inference engine matches conditions against known facts and fires applicable rules.

- Example: IF temperature > 100 AND pressure > 50 THEN activate alarm
- Best for expert systems and decision-making problems

### 5. Scripts
Represents stereotypical sequences of events in a structured form. Used for common real-world scenarios.

- Example: Restaurant script — Enter → Sit → Order → Eat → Pay → Leave
- Best for natural language understanding and event-based reasoning

### 6. Conceptual Dependency
Represents meaning of natural language sentences using a small fixed set of primitive actions, making representation language-independent.

- Example: "John ate pizza" represented using primitive action INGEST
- Best for natural language processing systems

### How to Choose an Approach

- Rule-based structured knowledge → **Production Rules or Logical Representation**
- Objects and relationships → **Semantic Networks or Frames**
- Natural language understanding → **Scripts or Conceptual Dependency**
- Formal provable reasoning → **Logical Representation**
- Complex objects with attributes and inheritance → **Frames**
- Simple hierarchical relationships → **Semantic Networks**

## 2. Semantic Network

A Semantic Network is a knowledge representation technique that represents knowledge as a directed graph where **nodes** represent objects, concepts, or events and **edges** represent the relationships between them. It is one of the earliest and most natural ways to represent structured knowledge in AI systems.

The two most common types of edges are:
- **is-a** — represents inheritance (Dog is-a Animal)
- **has-a** — represents property (Dog has-a Tail)

Other edges can represent any relationship like "loves", "owns", "part-of" depending on the domain.

### Advantages / Importance

- **Easy to visualize** — the graph structure makes knowledge easy to understand and present even to non-technical people
- **Inheritance** — properties defined at a higher node are automatically inherited by lower nodes, reducing redundancy. If Animal has-a Heart, then Dog inherits it automatically
- **Natural representation** — closely mirrors how humans think about objects and their relationships
- **Efficient retrieval** — finding related concepts is fast since you just follow edges in the graph
- **Flexible** — new nodes and relationships can be added easily without restructuring the entire network


### Disadvantages / Limitations

- **No standard definition** — there is no universally agreed upon set of relationship types, making different semantic networks incompatible with each other
- **Weak inferential power** — cannot handle complex logical reasoning like quantifiers, negation, or disjunction well
- **Ambiguity** — same relationship can be represented in multiple ways leading to inconsistency
- **Difficult to handle exceptions** — if most dogs are friendly but one specific dog is not, representing this exception cleanly is difficult
- **Does not scale well** — for very large knowledge bases the graph becomes too complex to manage and reason with efficiently


### Example

```
        Animal
        /    \
      is-a   is-a
      /          \
    Bird         Dog
    /  \           \
  is-a  has-a     has-a
  /        \          \
Penguin   Wings      Tail
             |
          can-fly: False (for Penguin)
```

- Animal → is-a → Bird → is-a → Penguin
- Bird → has-a → Wings
- Dog → has-a → Tail
- Dog → is-a → Animal (inherits has-a Heart from Animal)


### Usage

- Used in **natural language processing** to represent word meanings and relationships
- Used in **expert systems** to represent domain knowledge in medical, legal, and engineering fields
- Used in **information retrieval systems** to link related concepts for better search results
- Used in **ontology building** — the foundation of modern knowledge graphs like Google Knowledge Graph
- Used in **question answering systems** where relationships between concepts need to be traversed to find answers

## 3. Frames

A Frame is a knowledge representation technique that represents knowledge as a structured data object called a **frame**, where each frame describes a concept, object, or situation. Each frame consists of **slots** (attributes or properties) and **fillers** (values of those slots). Frames are similar to classes and objects in Object Oriented Programming — a frame defines the structure and an instance frame holds actual values.

Frames can be organized in a hierarchy where child frames **inherit** slots and default values from parent frames, and can override them if needed.

<img width="1280" height="720" alt="image" src="https://github.com/user-attachments/assets/36c3c8df-9b95-4302-bf2f-b6a12feff69a" />


A frame typically looks like:

```
Frame: Dog
  is-a: Animal
  has-legs: 4
  sound: Bark
  owner: [unknown]
  color: [unknown]
```


### Advantages / Importance

- **Natural grouping** — all properties of an object are grouped in one place making knowledge organized and easy to manage
- **Inheritance** — child frames automatically inherit slots and default values from parent frames, reducing redundancy
- **Default values** — slots can have default values that apply unless overridden, making reasoning with incomplete knowledge possible
- **Closely models real world** — objects in the real world naturally have attributes and values, so frames feel intuitive to build and understand
- **Supports object oriented thinking** — easy to map to OOP concepts like classes, instances, and inheritance making implementation straightforward


### Disadvantages / Limitations

- **Weak logical reasoning** — frames do not support formal logical inference like resolution or chaining directly
- **Handling exceptions is messy** — overriding inherited values for exceptions can make the hierarchy inconsistent and hard to maintain
- **No standard format** — different frame systems use different structures making interoperability difficult
- **Rigid structure** — if the domain changes significantly, restructuring the frame hierarchy can be expensive
- **Cannot represent procedural knowledge well** — frames are good for static object descriptions but struggle with representing dynamic processes or events


### Example

```
Frame: Vehicle
  has-wheels: true
  has-engine: true
  used-for: transportation

Frame: Car (is-a: Vehicle)
  has-wheels: 4
  has-doors: 4
  fuel-type: petrol

Frame: ElectricCar (is-a: Car)
  fuel-type: electric        ← overrides Car's default
  has-charging-port: true
```

- ElectricCar inherits has-wheels: 4 and has-doors: 4 from Car
- ElectricCar inherits has-engine: true from Vehicle
- ElectricCar overrides fuel-type to electric

### Usage

- Used in **natural language understanding** to represent typical situations and fill in missing information using default values
- Used in **expert systems** to represent structured domain knowledge like patient records, legal cases, or engineering components
- Used in **object oriented AI systems** where knowledge needs to be modular and reusable
- Used in **vision systems** to represent objects being recognized with their expected properties
- Used as the foundation for modern **ontology languages** like OWL used in the Semantic Web


### How Frames are Useful in Semantic Networks

Semantic networks represent relationships between concepts using nodes and edges, but they lack structure for representing the internal properties of each concept in detail. Frames solve this problem by acting as the internal description of each node in a semantic network.

- Each **node** in a semantic network can be represented as a **frame** with slots describing its properties
- **Edges** in the semantic network (is-a, has-a) correspond to **slots** in the frame
- **Inheritance** in semantic networks is made more powerful and structured through frame hierarchies
- Frames add **default values** to nodes which semantic networks alone cannot represent
- Together they give both the **relational structure** of a semantic network and the **detailed internal description** of frames — making the overall knowledge representation richer and more complete

Example: The node "Dog" in a semantic network becomes a full frame with slots like has-legs: 4, sound: Bark, is-a: Animal — giving far more detail than a simple node with edges could provide.

## 4. Conceptual Dependency

### Description
Conceptual Dependency (CD) is a knowledge representation technique developed by Roger Schank that represents the meaning of natural language sentences using a small fixed set of **primitive actions**. The goal is to represent any sentence from any language in a single **language-independent** canonical form — so that two sentences with the same meaning produce the same representation regardless of how they are worded.

The core idea is that all human actions can be broken down into a small set of primitive acts, and by combining these primitives with actors, objects, directions, and instruments, any sentence can be fully represented.


### Primitives
CD uses a fixed set of primitive actions called **ACTs**:

- **ATRANS** — transfer of abstract relationship (ownership, possession) — Example: give, sell, take
- **PTRANS** — transfer of physical location of an object — Example: go, move, carry
- **PROPEL** — application of physical force — Example: push, kick, throw
- **MOVE** — movement of a body part — Example: wave, kick (body movement)
- **GRASP** — grasping of an object — Example: grab, hold, release
- **INGEST** — taking something inside the body — Example: eat, drink, breathe
- **EXPEL** — expelling something from the body — Example: cry, sweat, spit
- **MTRANS** — transfer of mental information — Example: tell, read, listen
- **MBUILD** — building new mental information — Example: decide, imagine, conclude
- **SPEAK** — producing sound — Example: say, shout, whisper
- **ATTEND** — focusing a sense organ — Example: see, listen, smell


### How Knowledge is Represented Using CD

Every sentence is broken down into a CD structure with the following components:

- **Actor** — who is performing the action
- **ACT** — the primitive action being performed
- **Object** — what the action is performed on
- **Direction** — from where to where (for transfers)
- **Instrument** — what is used to perform the action

Example 1: "John ate pizza"
- Actor: John
- ACT: INGEST
- Object: pizza
- Direction: into John's body

Example 2: "Mary told John the news"
- Actor: Mary
- ACT: MTRANS
- Object: news
- Direction: from Mary to John

Example 3: "Ram gave Sita a book"
- Actor: Ram
- ACT: ATRANS
- Object: book
- Direction: from Ram to Sita

## 6. Scripts

### Description
A Script is a knowledge representation technique developed by Roger Schank and Robert Abelson that represents **stereotypical sequences of events** in a structured form. A script describes what typically happens in a familiar, recurring real-world situation — like going to a restaurant, visiting a doctor, or boarding a flight. Scripts allow an AI system to understand and predict events in common situations by filling in unstated details using prior knowledge.

A script is essentially a **frame for events** — instead of describing properties of an object, it describes the expected sequence of actions in a situation.


### Structure of a Script

A script consists of the following components:

- **Entry Conditions** — conditions that must be true before the script can begin
   - Example: Customer is hungry, Restaurant is open

- **Roles** — participants involved in the script
   - Example: Customer, Waiter, Chef, Cashier

- **Props** — objects used in the script
   - Example: Menu, Food, Table, Bill, Money

- **Track** — variation of the script for different situations
   - Example: Fast food track vs Fine dining track

- **Scenes** — the actual sequence of events broken into stages
   - Example: Entering scene, Ordering scene, Eating scene, Leaving scene

- **Results** — conditions that are true after the script ends
   - Example: Customer is no longer hungry, Restaurant has more money


### Example — Restaurant Script

**Entry Conditions:** Customer is hungry, Customer has money, Restaurant is open

**Roles:** Customer, Waiter, Chef, Cashier

**Props:** Menu, Table, Food, Bill, Money

**Track:** Sit-down restaurant

**Scenes:**

Scene 1 — Entering:
- Customer enters restaurant
- Customer finds a table
- Customer sits down

Scene 2 — Ordering:
- Waiter brings menu
- Customer reads menu
- Customer tells waiter the order
- Waiter gives order to chef
- Chef prepares food

Scene 3 — Eating:
- Waiter brings food to customer
- Customer eats food

Scene 4 — Leaving:
- Waiter brings bill
- Customer pays bill
- Customer leaves restaurant

**Results:** Customer is no longer hungry, Restaurant has received payment
