# Implementation

## 4.3 Mapping Design to Code

### Visibility

| UML | C++ |
|-----|-----|
| `+` | `public:` |
| `-` | `private:` |
| `#` | `protected:` |

### Cardinality

| Multiplicity | C++ |
|---|---|
| `1` or `0..1` | single object → `Student student;` |
| `1..*` or `0..*` | collection → `vector<Student> students;` |

### Relationships

#### Association

Objects are connected but independent.

```cpp
class Student {
    Course course;           // single
    vector<Course> courses;  // many
};
```

#### Aggregation

"Has-a" — object exists independently of parent. Use pointer.

```cpp
class Department {
    Student* student;        // student exists without department
};
```

#### Composition

"Owns" — object dies with parent. Use object directly.

```cpp
class House {
    Room room;               // room dies when house dies
};
```

### Inheritance

"Is-a" — child inherits parent.

```cpp
class Dog : public Animal {
    // Dog inherits all Animal attributes and methods
};
```

### Dependency

"Uses" — one class uses another temporarily (parameter/local var).

```cpp
class Order {
    void process(Payment p); // depends on Payment
};
```

## Many-to-Many

Both classes hold vector of each other — or use a joining class.

```cpp
class Student { vector<Course> courses; };
class Course  { vector<Student> students; };
```

## 4.7 Exception and Error Handling

**Exception** — Unusual Runtime errors due to which normal execution of program is ceased.
**Exception Handling** — A mechanism to detect, catch, and handle runtime errors gracefully without crashing the program.

**Why Exceptions Occur:**

- Division by zero
- File not found
- Invalid memory access
- Array index out of bounds
- Null pointer dereferencing
- Invalid user input etc

**Types of Exceptions:**

**1. Checked Exception** — Predictable exceptions detected at compile time. Compiler forces you to handle it because it knows it might go wrong. e.g. opening a file that might not exist → `FileNotFoundException`

**2. Unchecked Exception** — Unpredictable exceptions detected at runtime. Compiler cannot predict where it might go wrong so handling is optional. e.g. accessing a null object → `NullPointerException`

**3. User-Defined Exception** — Custom exception created by programmer by inheriting base exception class to handle application-specific errors. e.g. `InsufficientFundsException`

**4. System Exception** — Raised by OS or runtime environment, not by the program itself. e.g. division by zero, stack overflow, memory access violation.

**How to Handle (in C++):**
Using `try`, `catch`, `throw` blocks:

```cpp
try {
    // risky code
    throw exception;
}
catch (type e) {
    // handle exception
}
```

- **throw** — raises/signals the exception
- **try** — wraps the code that may cause exception
- **catch** — catches and handles the exception
- **catch(...)** — catches any type of exception

---

**Example:**

```cpp
try {
    if (b == 0)
        throw "Division by zero!";
    cout << a / b;
}
catch (const char* e) {
    cout << "Error: " << e;
}
```
