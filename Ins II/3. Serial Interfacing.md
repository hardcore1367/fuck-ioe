# Serial Interfacing

## 3.1 Serial Data Transmission

In serial data transmission (SDT), data is sent one bit at a time over the transmission channel. It is slower compared to parallel transmission.

### 1. Advantages of Serial over Parallel

- Longer data transmission is possible.
- Voltage loss is not a problem in SDT.
- It requires less number of wires and is cheaper to transmit data.
- Crosstalk is not an issue because there are fewer conductors compared to parallel transmission.

### 2 Synchronous Serial Data Transmission

Data is sent as a continuous stream of bits with no gaps between characters. A **shared clock signal** is used to synchronize the sender and receiver so both know exactly when each bit starts and ends. Data is grouped into **frames or blocks**, and special **sync characters** are sent at the beginning to establish synchronization. Because there are no start/stop bits per character, it is more efficient and faster than asynchronous transmission.

**Advantages:** Faster, less overhead
**Disadvantages:** Complex hardware, expensive, sensitive to clock loss


## 3 Asynchronous Serial Data Transmission

Data is sent **one character at a time** with no shared clock. Instead, each character is framed with a **start bit (0)** at the beginning and one or two **stop bits (1)** at the end. The receiver detects the start bit and uses its own internal clock to sample the remaining bits. Between characters, the line stays in **idle state (logic 1)**. This makes it simple to implement but adds per-character overhead.

Each character frame consists of:
- **1 start bit**
- **7 or 8 data bits**
- **1 parity bit**
- **1 or 2 stop bits**

**Advantages:** Simple, cheap, no clock sync needed
**Disadvantages:** Slower, not ideal for bulk transfer

## 4. Synchronous vs Asynchronous Serial Transmission

| Synchronous | Asynchronous |
|---|---|
| Uses a shared clock signal between sender and receiver | No shared clock; receiver uses its own internal clock |
| Data is sent as a continuous stream in blocks or frames | Data is sent one character at a time |
| Sync characters sent at the start to establish timing | Each character framed with start and stop bits |
| No start/stop bits per character, so less overhead | 1 start bit, 7/8 data bits, 1 parity bit, 1/2 stop bits per character |
| Faster and more efficient for large data transfer | Slower due to per-character framing overhead |
| Complex and expensive hardware required | Simple and cheap to implement |
| Loss of sync corrupts the entire block | Error in one character does not affect others |

## 3.3 Errors in Serial Data Transmission

- **Framing Error** — Occurs when the receiver expects a stop bit (logic 1) but receives a 0 instead. This means the receiver's clock has drifted out of sync with the sender, causing the frame boundaries to misalign.

- **Break Error** — Occurs when the transmission line stays at logic 0 (space/low) continuously for longer than a full character frame duration. This is usually caused by a broken wire, disconnected cable, or a deliberate break signal sent by the transmitter.

- **Overrun Error** — Occurs when the sender transmits the next character before the receiver has finished reading the previous one from its buffer. The old data gets overwritten before it is processed, resulting in data loss.

- **Parity Error** — Occurs when the number of 1s in the received data bits does not match the expected parity (odd or even). This indicates that one or more bits were flipped or corrupted during transmission due to noise on the line.

## 3.4 Error Detection in Serial Data Transmission

### 1. Parity Check

A single extra bit (parity bit) is added to each data frame so that the total number of 1s follows a fixed rule — either always even or always odd.

**Mechanism:**
- Sender counts the number of 1s in the data bits.
- For **even parity**: parity bit is set to make total 1s even.
- For **odd parity**: parity bit is set to make total 1s odd.
- Example: data = `1011001` (four 1s) → even parity bit = `1011001 0`, odd parity bit = `1011001 1`.
- Receiver counts 1s in the received data + parity bit.
- If the count violates the agreed parity rule → error detected.
- Limitation: if **two bits** flip, the parity count stays correct and the error goes undetected.


### 2. Checksum

 The sender computes a sum over all data blocks and sends it alongside the data. The receiver recomputes and compares.

**Mechanism:**
- Sender divides the data into fixed-size blocks (e.g., 8-bit or 16-bit segments).
- All blocks are added together using binary addition.
- If the sum overflows, the overflow bits are **wrapped around** and added back (called end-around carry).
- The **one's complement** of the final sum is taken — this is the checksum.
- Checksum is appended to the data and transmitted.
- Receiver adds all received blocks including the checksum.
- If the result is all 1s (i.e., one's complement is 0) → no error.
- If not → error detected.
- Limitation: if errors cancel each other out across blocks, the checksum may still appear valid.

Data blocks (8-bit each):
```
  10110011
+ 01101100
----------
 100011111  → overflow, wrap carry → 00011111 + 1 = 00100000
```
One's complement of `00100000` = **`11011111`** → this is the checksum, appended and sent.

Receiver adds all blocks including checksum:
```
  10110011
+ 01101100
+ 11011111
----------
 111111111 → one's complement = 00000000 ✓ → no error
```
If result ≠ 0 → **error detected**


### 3. Cyclic Redundancy Check (CRC)

The data is treated as a binary polynomial and divided by a predefined **generator polynomial**. The remainder of this division is sent with the data.

**Mechanism:**
- Sender and receiver agree on a **generator polynomial** (e.g., CRC-16, CRC-32).
- Sender appends zeros to the data (equal to the degree of the generator polynomial).
- The data is then **divided by the generator polynomial** using XOR-based binary division (not normal arithmetic division).
- The **remainder** from this division is the CRC value, which replaces the appended zeros.
- The data + CRC is transmitted.
- Receiver performs the same XOR division on the received data using the same generator.
- If remainder = **0** → data is error-free.
- If remainder ≠ **0** → error detected.
- Much stronger than parity or checksum — can detect all single-bit errors, all burst errors up to the length of the generator polynomial, and most longer burst errors.


Data: `1101011011`, Generator: `10011`

- Append 4 zeros (degree of generator = 4): `11010110110000`
- XOR divide `11010110110000` by `10011`:

```
11010110110000 ÷ 10011
-----------------------
11010 XOR 10011 = 01001
 10011 XOR 10011 = 00000
  ...continue until remainder = 1110
```
CRC = `1110` → transmitted: `1101011011 1110`

Receiver divides `11010110111110` by `10011`:
- Remainder = `0000` ✓ → **no error**
- Remainder ≠ 0 → **error detected**

## 3.5 Bit Rate and Baud Rate

**Bit Rate** — The number of **bits transmitted per second** (bps). It measures how much actual data is being sent over the channel.

**Baud Rate** — The number of **signal changes (symbols) per second** (Bd). Each symbol may carry one or more bits depending on the encoding scheme.

**Relationship:**
> Bit Rate = Baud Rate × Number of bits per symbol

- If each symbol carries **1 bit** → Bit Rate = Baud Rate
- If each symbol carries **2 bits** → Bit Rate = 2 × Baud Rate

**Example:**
A modem operates at 2400 baud with 4 signal levels (2 bits per symbol):
Bit Rate = 2400 × 2 = **4800 bps**

**Key Difference:**

| Bit Rate | Baud Rate |
|---|---|
| Measures bits per second | Measures symbols per second |
| Depends on encoding | Property of the physical channel |
| Can be higher than baud rate | Always fixed to signal transitions |
