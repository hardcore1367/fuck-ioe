Operating System is a system software which works as a interface between a user of a computer and the computer hardware. In other words, It is a program that manages computer hardware.

## 1 Functions of an Operating System
**1. Booting**
- It is a process of loading OS into RAM when computer is powered on
- Bootloader loads the OS kernel into memory and starts the computer to work.

**2. Process Management**
- OS creates, schedules, and terminates processes
- Uses scheduling algorithms (FCFS, Round Robin, SJF) to allocate CPU time
- Handles **process synchronization** and avoids deadlocks

**3. Memory Management**
- Tracks every memory location — which process is using it, which is free
- Allocates memory when process starts, deallocates when it ends

**4. File / Disk Management**
- Organizes data into files and directories on storage devices
- Controls read/write/delete operations
- Manages **file permissions** and maintains directory structure

**5. Loading & Execution**
- Loads executable programs from disk into RAM
- Allocates necessary resources (memory, CPU) and begins execution
- Handles errors during execution (e.g., segmentation fault)

**6. Data Security & Protection**
- Prevents unauthorized access using **authentication** (passwords) and **authorization** (permissions)
- Protects one process from accessing another process's memory
- Provides data backup and access control mechanisms

**7. User Interface**
- **CLI** (Command Line Interface) — user types commands (e.g., Linux terminal)
- **GUI** (Graphical User Interface) — user interacts via icons/windows (e.g., Windows)
- Acts as bridge between user and hardware

**8. Device Management**
- Manages communication between OS and hardware devices (keyboard, printer, disk)
- Uses **device drivers** as translators between OS and hardware
- Handles I/O scheduling and buffering

 ## 2. Types of OS

**1. Batch Operating System**

Similar jobs are grouped into batches and executed one after another without user interaction. The user submits jobs to an operator, the operator groups similar jobs together, and the OS executes them sequentially. There is no direct communication between the user and the computer during execution.

<img width="660" height="330" alt="image" src="https://github.com/user-attachments/assets/4116bcc6-0687-483e-936d-19850e3c04a9" />


- **Advantages:**
  - Efficient for large repetitive tasks
  - CPU stays busy; less idle time
  - Simple to implement
- **Disadvantages:**
  - No interaction during execution
  - If one job fails, rest of batch is delayed
  - Hard to debug

**2. Time-Sharing Operating System**

It is a type of multiprogramming system where each process runs in a round-robin manner. Every task gets a fixed time slice called a quantum. After the quantum ends, the OS switches to the next task, allowing multiple tasks whether from one user or many to run smoothly on a single system.

<img width="1000" height="583" alt="image" src="https://github.com/user-attachments/assets/04b194ec-8dc3-4d29-aab5-32de7002f839" />


- **Advantages:**
  - Multiple users/process can work/run simultaneously
  - Fast response time
  - Better resource utilization
- **Disadvantages:**
  - Security risk between users
  - Heavy overhead due to context switching
  - Complex to manage

**3. Distributed Operating System**

Multiple independent computers (nodes) connected via a network work together and appear as a single system to the user. Tasks are distributed across nodes; if one node fails, others continue working. Communication happens via message passing.

<img width="1000" height="325" alt="image" src="https://github.com/user-attachments/assets/dea51437-ba12-494b-ba3b-b4ab6bb72893" />


- **Advantages:**
  - High fault tolerance
  - Resource sharing across nodes
  - Faster computation via parallel processing
- **Disadvantages:**
  - Complex to design
  - Network failure affects the whole system
  - Security challenges

**4. Network Operating System**

Provides networking features to connect multiple computers so they can share resources like files and printers. Each machine runs its own local OS but also has networking software. Examples: Windows Server, Linux Server.

<img width="800" height="400" alt="image" src="https://github.com/user-attachments/assets/e22cf61a-69c5-4e03-86a5-ec9abb39ea94" />


- **Advantages:**
  - Centralized resource sharing
  - Easy to add new machines
  - Better security via server control
- **Disadvantages:**
  - Expensive server hardware needed
  - If server fails, all clients are affected
  - Requires admin expertise

**5. Real-Time Operating System (RTOS)**

Processes data and responds within a guaranteed time (deadline). Used where timing is critical — medical devices, missile systems, industrial controllers.

**Types:**
- **Hard RTOS** — deadline must be met strictly; missing it causes system failure (e.g., airbag controller, pacemaker)
- **Soft RTOS** — deadline is important but occasionally missing it is acceptable (e.g., video streaming, online games)

- **Advantages:**
  - Guaranteed response time
  - Highly reliable
  - Minimal memory usage
- **Disadvantages:**
  - Very complex to design
  - Limited multitasking
  - Expensive development
 
**6. Multiprogramming Operating System**

Multiple programs are loaded into memory at the same time. When one process is waiting (e.g., for I/O), the CPU switches to another process instead of sitting idle. The goal is to keep the CPU busy at all times. There is only one CPU but it switches between jobs to maximize utilization.

- **Advantages:**
  - CPU is always busy; no idle time
  - Better memory utilization (multiple programs in RAM)
  - Increased overall system throughput
- **Disadvantages:**
  - Complex memory management required
  - If one program crashes, it may affect others
  - No direct user interaction during execution

## 3. OS Structures
**1. Monolithic Structure**

The entire OS runs as a single large program in kernel space. All OS services (memory management, file system, device drivers, process management) are packed together and run in the same address space. Any component can directly call any other component.

<img width="800" height="400" alt="image" src="https://github.com/user-attachments/assets/8604465b-81a0-433c-9608-b50168798613" />

- **Advantages:**
  - Fast — direct communication between components
  - Simple design
  - Low overhead
- **Disadvantages:**
  - If one component crashes, entire OS crashes
  - Hard to maintain and debug
  - Any change requires recompiling the whole kernel

---

**2. Layered Structure**

OS is divided into multiple layers (levels). Each layer only communicates with the layer directly above or below it. Bottom layer (Layer 0) is hardware; top layer is the user interface.

<img width="768" height="576" alt="image" src="https://github.com/user-attachments/assets/e98c233c-09fd-44a5-ad71-7778e3dac1c1" />


- **Advantages:**
  - Easy to debug (test one layer at a time)
  - Each layer is independent and modular
  - Easy to maintain
- **Disadvantages:**
  - Slower — request must pass through every layer
  - Difficult to define clean boundaries between layers
  - Less efficient than monolithic

---

**3. Microkernel Structure**

Only the most essential functions stay in the kernel (IPC, basic scheduling, basic memory management). Everything else (file system, device drivers, networking) runs in **user space** as separate processes.

<img width="474" height="229" alt="image" src="https://github.com/user-attachments/assets/ce85d27c-f471-44cb-810b-2e35ed4ae05d" />


- **Advantages:**
  - Very stable — a crashed service doesn't crash the kernel
  - Easy to extend or port to new hardware
  - Better security and isolation
- **Disadvantages:**
  - Slower — user space and kernel space must communicate via IPC (message passing)
  - More complex design
  - Higher overhead

---

**4. Virtual Machine (VM) Structure**

Creates an illusion that each user has their own dedicated hardware. A **hypervisor** (VMM — Virtual Machine Monitor) sits between hardware and OS and manages multiple virtual machines, each running its own OS.

<img width="527" height="318" alt="image" src="https://github.com/user-attachments/assets/aca8a5ee-b431-435b-9c6a-d0e953659b17" />


- **Advantages:**
  - Multiple OSes can run on the same hardware
  - Complete isolation — one VM crash doesn't affect others
  - Great for testing and development
- **Disadvantages:**
  - High resource overhead (each VM needs its own RAM, CPU)
  - Complex to implement
  - Performance slower than running OS directly on hardware

---

**5. Client-Server Structure**

OS functionality is split into **servers** (provide services) and **clients** (request services). Servers run in user space; kernel just handles communication. Similar to microkernel but emphasizes the client-server model explicitly.

```
Client Process → requests service
      ↓
   Kernel (message passing)
      ↓
Server Process → provides service (file, print, memory)
```

- **Advantages:**
  - Highly modular and flexible
  - Servers can run on different machines (suits distributed systems)
  - Crash of one server doesn't affect others
- **Disadvantages:**
  - Slow due to message passing overhead
  - Complex communication management
  - Server failure makes that service unavailable

## 4. Shell and Kernel 
**Kernel**

The kernel is the **core of the OS** — the part that directly interacts with hardware. It is the first program loaded into memory during booting and stays in memory until the system shuts down. It runs in **privileged mode (kernel mode)** meaning it has full access to all hardware resources.

What it does:
- Manages CPU (process scheduling)
- Manages RAM (memory allocation)
- Manages devices (via drivers)
- Handles system calls from user programs
- Controls file system access

---

**Shell**

The shell is the **interface between the user and the kernel**. The user types commands into the shell; the shell interprets them and passes them to the kernel for execution. It runs in **user mode**.

Types of Shell:
- **CLI Shell** — user types commands (e.g., Bash, Zsh on Linux)
- **GUI Shell** — user interacts via icons/windows (e.g., Windows Explorer)

What it does:
- Reads user commands
- Interprets and parses them
- Sends request to kernel via system calls
- Displays output back to user



**Shell vs Kernel**

| Feature | Shell | Kernel |
|---|---|---|
| Role | Interface between user and kernel | Core of OS; interacts with hardware |
| Mode | User mode | Kernel (privileged) mode |
| Function | Interprets user commands | Manages CPU, memory, devices |
| Type | CLI or GUI | Monolithic, Micro, etc. |
| Loaded | After kernel during boot | First thing loaded at boot |
| Direct HW access | No | Yes |
| Example | Bash, Zsh, PowerShell | Linux kernel, Windows NT kernel |


**Relationship:**
```
User → Shell → Kernel → Hardware
```
User talks to shell, shell talks to kernel, kernel talks to hardware.

## 5. System Calls

A system call is a **programmatic way for a user program to request a service from the kernel**. Since user programs run in user mode and cannot directly access hardware, they use system calls to ask the OS to do it on their behalf.


**Purpose**
- Provides controlled access to hardware and OS resources
- Acts as the interface between user programs and the kernel
- Ensures security — user programs can't directly mess with hardware


**How it Works**

```
User Program
    ↓  calls system call (e.g., read())
Library (e.g., glibc)
    ↓  triggers software interrupt / trap
Kernel Mode (switches from user mode)
    ↓  OS executes the requested service
    ↓  result returned to user program
User Program resumes
```

1. User program calls a function (e.g., `read()`)
2. A **trap/interrupt** is generated — CPU switches to kernel mode
3. Kernel identifies which system call was requested (via **system call number**)
4. Kernel executes the service
5. Result is returned; CPU switches back to user mode
6. Program continues execution

---

**Types of System Calls**

**1. Process Control**
- Create, terminate, load, execute processes
- e.g., `fork()`, `exit()`, `exec()`

**2. File Management**
- Create, open, read, write, close, delete files
- e.g., `open()`, `read()`, `write()`, `close()`

**3. Device Management**
- Request/release devices, read/write from devices
- e.g., `ioctl()`, `read()`, `write()` on device files

**4. Information Maintenance**
- Get/set system info — time, date, process attributes
- e.g., `getpid()`, `alarm()`, `sleep()`

**5. Communication**
- Create connections, send/receive messages between processes
- e.g., `pipe()`, `socket()`, `send()`, `recv()`

**6. Protection**
- Control access to resources, set permissions
- e.g., `chmod()`, `chown()`, `umask()`


**Key Point to Remember:**
> System calls are the only legal way for a user program to enter kernel mode.

## 6. Examples of OS
**Unix**

One of the oldest and most influential operating systems, developed at **Bell Labs (AT&T) in 1969**. It was the foundation for most modern OSes. Designed for multi-user, multitasking environments. Written in **C language**, making it portable across hardware.

- Used in: servers, workstations, mainframes
- Key features: everything is a file, shell scripting, pipes, modular tools
- Examples: Solaris, HP-UX, AIX, macOS (Unix-based)


**Linux**

An **open-source OS kernel** created by **Linus Torvalds in 1991**, inspired by Unix but written from scratch. Anyone can view, modify, and distribute it. Combined with GNU tools it forms a complete OS (GNU/Linux).

- Used in: servers, smartphones (Android), embedded systems, supercomputers
- Key features: open source, highly customizable, stable, free
- Examples (distros):
  - **Ubuntu** — beginner friendly, desktop use
  - **Debian** — stable, used in servers
  - **Kali Linux** — security and penetration testing
  - **Red Hat / CentOS** — enterprise servers
  - **Android** — Linux kernel based, mobile


**Windows**

Developed by **Microsoft**, first released in **1985**. Most widely used desktop OS in the world. Closed-source and commercial. Known for its GUI and compatibility with most software and hardware.

- Used in: personal computers, laptops, enterprise systems, gaming
- Key features: GUI-based, plug and play, wide software support, Active Directory for enterprise
- Versions: Windows XP → 7 → 8 → 10 → 11

## 7. Handheld Operating System
**Handheld OS** — lightweight OS designed for portable devices like smartphones and tablets. Optimized for touch input, low power, and wireless connectivity.

**Examples:**
- **Android** — Google, Linux-based, open source, most widely used
- **iOS** — Apple, Unix-based, closed source, iPhone/iPad only
- **HarmonyOS** — Huawei, multi-device support
- **KaiOS** — feature phones, low-cost devices

## Questions
1. Explain why exo-kernel doesn't require re-mapping of resources?
> In a traditional OS, the kernel **abstracts hardware** — it maps physical resources (actual RAM addresses, disk blocks) into virtual abstractions (virtual memory, file system). Every access requires the kernel to **re-map** virtual → physical, adding overhead.
>
> In an **Exokernel**, there is **no abstraction layer**. The kernel exposes **physical resources directly** to applications — actual physical memory addresses, disk blocks, CPU time slots. Applications (via **libOS — library OS**) manage these resources themselves.
>
> Since the application already holds the **actual physical resource name/address**, there is nothing to re-map — it accesses the resource directly.
> 
> 
> Traditional OS:
> App → virtual address → kernel remaps → physical address → hardware
> 
> Exokernel:
> App → physical address directly → hardware
> 
> **In short:** Re-mapping only exists because of abstraction. Exokernel removes abstraction, so re-mapping becomes unnecessary.
