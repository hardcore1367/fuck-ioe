## PID Tuning

**PID Tuning** means adjusting the three gains — **Kp, Ki, Kd** — of a PID controller so the system responds in the best way possible (fast, stable, no big overshoot).

---

**What each gain does when you increase it:**

| Gain | Effect |
|------|--------|
| **Kp** (Proportional) | Faster response, but too much → oscillation |
| **Ki** (Integral) | Removes steady-state error, but too much → overshoot |
| **Kd** (Derivative) | Reduces overshoot, but too much → sensitive to noise |

---

**Common Tuning Method — Trial & Error (Manual Tuning):**
1. Set Ki = 0, Kd = 0
2. Increase **Kp** until system responds fast but starts oscillating
3. Increase **Kd** to reduce oscillation
4. Increase **Ki** to remove remaining steady-state error
5. Fine-tune until response is satisfactory

---

**Goal of tuning:** Minimize **rise time**, **overshoot**, and **steady-state error** at the same time.

## P Controller (Proportional Controller)

A **P Controller** is the simplest type of controller. It looks at the current error (difference between desired value and actual value) and produces an output that is directly proportional to that error. This means — bigger the error, bigger the correction.

**Output:** `u(t) = Kp × e(t)`

The **Kp** (proportional gain) controls how aggressively the system reacts. If Kp is too small, the system responds slowly. If Kp is too large, the system overshoots and starts oscillating. The main problem with a P controller is that it **always leaves a steady-state error** — it can never bring the error to exactly zero because once the error becomes small, the correction also becomes too small to fully fix it.

- Fast and simple to implement
- Higher Kp → faster response but causes oscillation
- Always has residual **steady-state error**
- Best for systems where small error is acceptable

---

## PI Controller (Proportional + Integral)

A **PI Controller** adds an **Integral term** on top of the proportional term. The integral part continuously adds up (accumulates) the error over time. Even if the error is very small, given enough time, the integral will keep growing until the controller pushes the output enough to completely eliminate the error. This is why PI controllers can achieve **zero steady-state error**.

**Output:** `u(t) = Kp·e(t) + Ki·∫e(t)dt`

The **Ki** (integral gain) controls how fast the accumulated error is acted upon. However, this comes at a cost — the system becomes **slower** and more likely to **overshoot**. Another problem is **integral windup**, where the integral keeps growing very large when the system is saturated (e.g., motor at max speed), causing the controller to lose control temporarily.

- Eliminates steady-state error completely
- Slower response compared to P alone
- Risk of overshoot and integral windup
- Best for systems that need accuracy but can tolerate some sluggishness

---

## PD Controller (Proportional + Derivative)

A **PD Controller** adds a **Derivative term** to the proportional term. The derivative looks at how fast the error is changing. If the error is decreasing rapidly, the controller reduces its output early to prevent overshooting the target. In simple words — it **predicts** where the system is going and acts in advance to slow it down.

**Output:** `u(t) = Kp·e(t) + Kd·(de/dt)`

The **Kd** (derivative gain) controls the braking effect. A higher Kd gives more damping and reduces overshoot. However, the derivative term is very **sensitive to noise** — any small sudden spike in the sensor signal looks like a rapid change in error, causing the controller to react unnecessarily. Also, since there is no integral term, PD controllers **cannot remove steady-state error**.

- Reduces overshoot and improves stability
- Faster and more stable than P alone
- Cannot eliminate steady-state error
- Very sensitive to measurement noise
- Best for systems with low noise and where overshoot is a big concern

---

## PID Controller (Proportional + Integral + Derivative)

A **PID Controller** combines all three terms together — proportional, integral, and derivative. It is the most commonly used controller in real-world systems. The proportional part gives a fast initial response, the integral part eliminates the steady-state error over time, and the derivative part prevents the system from overshooting by detecting rapid changes in error and applying a braking effect.

**Output:** `u(t) = Kp·e(t) + Ki·∫e(t)dt + Kd·(de/dt)`

All three gains **(Kp, Ki, Kd)** must be properly **tuned** for the controller to work well. If any gain is set incorrectly, the system can become unstable, too slow, or noisy. Methods like **Ziegler-Nichols** or manual trial-and-error are used for tuning. PID controllers are used in temperature control, motor speed control, robotics, drones, and almost every industrial control system.

- Combines benefits of all three controllers
- Eliminates steady-state error (from I) and reduces overshoot (from D)
- Requires careful tuning of all three gains
- Best overall performance among all controller types
- Widely used in real-world industrial and embedded systems

---
| Feature | P | PI | PD | PID |
|---|---|---|---|---|
| **Terms Used** | Proportional | Proportional + Integral | Proportional + Derivative | All three |
| **Steady-State Error** | Always present | Eliminated | Always present | Eliminated |
| **Overshoot** | Moderate | High | Low | Low |
| **Noise Sensitivity** | Low | Low | High | Moderate |
| **Complexity** | Simple | Moderate | Moderate | Complex |
| **Main Drawback** | Can't remove steady-state error | Integral windup, overshoot | Can't remove steady-state error | Needs careful tuning |


## PID Controller – Algorithm & Pseudocode

**Algorithm:**
1. Read setpoint (desired value) and actual output
2. Calculate error = setpoint − actual
3. Calculate proportional term = Kp × error
4. Add error to integral sum, calculate integral term = Ki × integral_sum
5. Calculate derivative term = Kd × (error − previous_error)
6. Output = P + I + D
7. Save current error as previous_error
8. Apply output to the system
9. Repeat

---

**Pseudocode:**

```
SET Kp, Ki, Kd
SET integral = 0
SET prev_error = 0

LOOP:
    error = setpoint - actual_output

    proportional = Kp * error

    integral = integral + error
    integral_term = Ki * integral

    derivative = Kd * (error - prev_error)

    output = proportional + integral_term + derivative

    apply(output)

    prev_error = error

    wait(dt)
END LOOP
```

---

**Key Variables:**
- `error` → current difference between desired and actual
- `integral` → accumulated error over time
- `prev_error` → error from last iteration (used for derivative)
- `output` → final control signal sent to the system

---

## Open Loop Automobile Cruise Controller

**Open Loop** means the controller sends a fixed output based on input only — it does **not** check the actual speed to correct itself.

---

**How it works:**

The driver sets a desired speed. The controller calculates how much throttle is needed to reach that speed based on a **predefined model** and applies it. It does not measure actual speed — so if road slope or wind changes the speed, it does nothing to fix it.

---

**Block Diagram:**

```
[Desired Speed] → [Controller] → [Throttle] → [Engine] → [Vehicle Speed]
```
No feedback path — output is never measured or corrected.

---

**Algorithm:**
1. Driver sets desired speed (setpoint)
2. Controller calculates required throttle using a fixed formula
3. Throttle signal is sent to the engine
4. Vehicle accelerates to expected speed
5. No correction is made after that

---

**Pseudocode:**

```
SET desired_speed
SET Kp = throttle gain constant

throttle_output = Kp * desired_speed

apply_throttle(throttle_output)

// No feedback, no correction
// System runs with fixed throttle
```

---

**Limitations of Open Loop Cruise Controller:**

- If road goes uphill → speed drops, controller does nothing
- If wind resistance increases → speed drops, no correction
- Engine wear or load changes → output is inaccurate
- Not reliable for real-world use

**That is why real cruise controllers are closed loop (use actual speed feedback to continuously correct the throttle).**

---
## Closed Loop Automobile Cruise Controller

**Closed Loop** means the controller continuously **measures the actual speed**, compares it with desired speed, and corrects the throttle automatically.

---

**How it works:**

The driver sets a desired speed. The controller reads the actual speed from a sensor, calculates the error, and adjusts the throttle to reduce that error. This keeps repeating — so even if road slope or wind changes the speed, the controller detects it and corrects it.

---

**Block Diagram:**

```
[Desired Speed] → [+] → [Controller] → [Throttle] → [Engine] → [Actual Speed]
                   ↑                                                    |
                   └──────────────[Speed Sensor]───────────────────────┘
                                  (Feedback Path)
```

---

**Algorithm:**
1. Driver sets desired speed (setpoint)
2. Sensor measures actual vehicle speed
3. Error = desired speed − actual speed
4. Controller calculates throttle based on error
5. Throttle is applied to engine
6. Vehicle speed changes
7. Sensor measures new speed again
8. Repeat continuously

---

**Pseudocode:**

```
SET desired_speed
SET Kp, Ki, Kd
SET integral = 0, prev_error = 0

LOOP:
    actual_speed = read_speed_sensor()

    error = desired_speed - actual_speed

    proportional = Kp * error

    integral = integral + error
    integral_term = Ki * integral

    derivative = Kd * (error - prev_error)

    throttle_output = proportional + integral_term + derivative

    apply_throttle(throttle_output)

    prev_error = error

    wait(dt)
END LOOP
```

---

**Advantages over Open Loop:**

| Open Loop | Closed Loop |
|---|---|
| No feedback | Continuously measures actual speed |
| Cannot correct disturbances | Automatically corrects speed changes |
| Inaccurate on hills or wind | Maintains speed accurately |
| Simple but unreliable | Slightly complex but reliable |

**This is how real cruise control systems work in cars — using a PID closed loop controller.**


## Key Terms in a Control System

**Controller:**
The brain of the system. It takes the error (difference between desired and actual output) and calculates what action needs to be taken. It decides "how much correction is needed."
- Example: PID algorithm running on a microcontroller

---

**Plant:**
The actual physical system being controlled. It receives the control signal and produces the output. It is what we are trying to control.
- Example: A car engine, a motor, a heater

---

**Actuator:**
The device that converts the controller's signal into a physical action. It sits between the controller and the plant.
- Example: Throttle valve, motor driver, relay switch

---

**Sensor:**
Measures the actual output of the plant and sends it back to the controller as feedback.
- Example: Speed sensor, temperature sensor, encoder

---

**Setpoint (Reference Input):**
The desired value that we want the system to reach and maintain.
- Example: Desired speed = 60 km/h in cruise control

---

**Error:**
Difference between the setpoint and the actual output.
`Error = Setpoint − Actual Output`

---

**Feedback:**
The signal sent from the sensor back to the controller so it can compare actual output with desired output and make corrections.

---

### How They All Connect:

```
[Setpoint] → [Controller] → [Actuator] → [Plant] → [Output]
                  ↑                                      |
                  └──────────────[Sensor]────────────────┘
                                (Feedback)
```
