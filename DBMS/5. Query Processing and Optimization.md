## 1. Query Processing
Query processing refers to the series of steps taken to execute a database request, which includes parsing, optimizing, and evaluating the query to retrieve or manipulate data efficiently

### Steps of Query Processing
1. Parsing and translation
2. Optimization
3. Evaluation

**Steps:**

**1. Parsing**
- Query is broken down into tokens and a **parse tree** is built
- Three checks performed:

  - **Syntax Check** — verifies query follows correct SQL grammar rules
    - e.g., missing keyword, wrong clause order → syntax error thrown
    - `SELCT * FROM Student` → syntax error (typo in SELECT)

  - **Semantic Check** — verifies query makes logical sense against the database schema
    - Checks if referenced tables, columns, and data types actually exist
    - e.g., `SELECT grade FROM Student` → error if column `grade` doesn't exist in Student table

  - **Shared Pool Check** — checks if the **same query has been executed before**
    - Database maintains a shared pool (cache) of previously parsed and optimized queries
    - If found (**soft parse**) → reuse existing execution plan, skip optimization
    - If not found (**hard parse**) → proceed to full optimization step
    - Saves significant time for repeated queries

**2. Translation**
- Valid query is translated into a **relational algebra expression**
- Internal representation used for optimization

**3. Optimization**
- Multiple equivalent execution plans generated
- **Query optimizer** selects the cheapest plan based on estimated cost (I/O, CPU, memory)
- Uses catalog statistics (table size, indexes) for cost estimation

**4. Execution**
- Selected plan is executed by the **query execution engine**
- Data fetched from disk, operations performed, result returned to user

## 2. Evaluation of Expressions

The process of executing a relational algebra expression (query plan) by computing each operation in the expression tree to produce the final result.

Each query is broken into a tree of operations (select, join, project etc.) — the question is **how to execute these operations together efficiently.**

**1. Materialization**
- Each operation is evaluated **one at a time**, bottom to top in the expression tree
- Result of each operation is stored in a **temporary relation on disk**
- Next operation reads from that temporary relation
- Process repeats until final result is produced

**How it works:**
```
Expression: σ(age>20) (Student ⋈ Marks)

Step 1: Compute Student ⋈ Marks → store result in temp1 (written to disk)
Step 2: Compute σ(age>20) on temp1 → store result in temp2 (written to disk)
Step 3: Return temp2 as final result
```

**Efficiency:**
- Simple and straightforward to implement
- Each intermediate result fully computed before next step
- **Very expensive** — multiple disk writes and reads for each intermediate result
- High I/O cost, high memory usage for large intermediate relations


**2. Pipelining**
- Multiple operations are evaluated **simultaneously** by passing results directly from one operation to the next **without writing to disk**
- Output tuples of one operation are fed as input to the next operation on the fly
- No temporary relations stored on disk

**How it works:**
```
Expression: σ(age>20) (Student ⋈ Marks)

Step 1: Join starts producing tuples one by one
Step 2: Each tuple immediately passed to σ(age>20)
Step 3: Filtered tuples directly returned as result
→ No disk write at any step
```

**Efficiency:**
- Eliminates costly disk writes for intermediate results
- Operations overlap in execution → much faster
- Significantly reduces I/O cost
- Not always possible — some operations like **sort and hash-join** require full input before producing output (called **blocking operations**)

**Materialization vs Pipelining:**

| Materialization | Pipelining |
|---|---|
| Results written to disk after each step | Results passed directly, no disk write |
| High I/O cost | Low I/O cost |
| Slower due to multiple disk accesses | Faster, operations overlap |
| Simple to implement | More complex to implement |
| Always applicable | Not applicable for blocking operations |
| High memory for temp relations | Low memory usage |


## 3. Query Optimization

The process of selecting the most efficient execution plan among multiple equivalent plans for a given query, minimizing time and resource usage.


**1. Heuristic-Based Optimization (Rule-Based)**
- Uses a set of **predefined rules** to transform the query into a more efficient form
- No cost estimation — just applies transformations that are **generally known to be better**
- Done during the **translation/algebraic optimization** phase

**Common Heuristic Rules:**
- **Perform selection early** — apply σ (select) as early as possible to reduce tuple count before joins
- **Perform projection early** — apply π (project) early to eliminate unnecessary columns
- **Most restrictive selection first** — apply the condition that filters the most rows first
- **Avoid cartesian product** — replace with joins wherever possible
- **Combine cascaded selections** — merge multiple σ into one to reduce passes

**Example:**
```
Original: π(name) (σ(age>20) (Student ⋈ Marks))

Heuristic: push σ down before join
Optimized: π(name) (σ(age>20)(Student) ⋈ Marks)
→ Join is done on a smaller filtered Student table
```

**Advantage** — fast, no overhead of cost estimation
**Disadvantage** — rules may not always produce the optimal plan, no awareness of actual data size or distribution



**2. Cost-Based Optimization**
- Generates **multiple equivalent execution plans** and estimates the cost of each
- Selects the plan with the **lowest estimated cost**
- Cost is measured in terms of **disk I/O, CPU usage, memory consumption**

**How it works:**
- **Catalog statistics** are maintained — table size, number of tuples, index info, column distribution
- For each operation (join, select, sort), cost is estimated using these statistics
- All possible orderings and methods are considered (e.g., nested loop join vs hash join vs merge join)
- Plan with minimum total cost is selected

**Example:**
```
Query: Student ⋈ Marks ⋈ Courses

Plan 1: (Student ⋈ Marks) ⋈ Courses  → cost = 500
Plan 2: Student ⋈ (Marks ⋈ Courses)  → cost = 300
Plan 3: (Student ⋈ Courses) ⋈ Marks  → cost = 450

→ Optimizer selects Plan 2 (lowest cost)
```

**Cost factors considered:**
- Size of intermediate relations
- Available indexes on join/filter columns
- Memory buffer size
- Sorting and hashing costs

**Advantage** — produces genuinely optimal or near-optimal plan
**Disadvantage** — expensive process itself, large number of plans to evaluate for complex queries


| Heuristic | Cost-Based |
|---|---|
| Rule-based transformations | Estimates actual execution cost |
| No statistics needed | Uses catalog statistics |
| Fast optimization process | Slower optimization process |
| May not find optimal plan | Finds near-optimal plan |
| Good for simple queries | Better for complex queries |

## 4. Query Decomposition

The process of breaking down a high-level query (SQL) into a sequence of lower-level relational algebra operations that can be executed by the system.

**4 Steps:**

**1. Analysis**
- Query is analyzed for **syntax and semantic correctness**
- Syntax check — verifies correct SQL grammar
- Semantic check — verifies tables, columns, and data types exist in schema
- Invalid queries are rejected with appropriate error messages

**2. Normalization**
- Query is converted into a **standardized/canonical form**
- Eliminates redundancies and simplifies the query structure
- Converts conditions into **conjunctive normal form (CNF)** — AND of OR conditions
- Makes further processing easier and consistent

**3. Redundancy Elimination**
- Checks if query is **semantically meaningful** beyond just schema validation
- Detects and removes **redundant predicates and conditions**
- e.g., `WHERE age > 20 AND age > 30` → simplified to `WHERE age > 30`
- Queries that are always false or produce empty results are detected and rejected early

**4. Simplification / Rewriting**
- Query is **rewritten into relational algebra expression**
- Heuristic transformations applied:
  - Push selections down (apply filters early)
  - Push projections down (eliminate unused columns early)
  - Remove unnecessary joins
- Produces an optimized relational algebra expression ready for the optimizer

---

```
SQL Query
   ↓
Step 1: Analysis → Syntax + Semantic Check
   ↓
Step 2: Normalization → Convert to canonical form (CNF)
   ↓
Step 3: Redundancy Elimination → Simplify conditions
   ↓
Step 4: Rewriting → Relational Algebra Expression
   ↓
Query Optimizer
```
