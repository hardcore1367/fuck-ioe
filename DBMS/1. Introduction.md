# Introduction  [3 hours] [4 marks]

Database: A database is a collection of related data which represents some aspect of the real world.
DBMS: Database Management System is a software for storing and retrieving user's data while considering appropriate security measures.

## Advantages of DBMS

- **Data sharing** — multiple users access data simultaneously
- **Reduced redundancy** — centralized storage avoids duplicate data
- **Data integrity** — constraints enforce accuracy and consistency
- **Data security** — access control restricts unauthorized users
- **Backup & recovery** — restores data automatically after failure
- **Data independence** — structural changes don't affect applications

## Drawbacks of File System

- **Data redundancy** — same data duplicated across files causing inconsistency
- **No concurrency** — poor handling of simultaneous access
- **No security** — lacks proper access control
- **No atomicity** — partial failures leave data corrupted
- **Data isolation** — files in different formats, hard to query together

## DBMS vs File System

| File System | DBMS |
|---|---|
| Redundancy is high | Redundancy is minimal |
| Consistency is hard to maintain | Consistency enforced via constraints |
| No concurrency support | Concurrency handled with locking |
| Recovery is manual | Recovery is automatic |
| Security is minimal | Role-based access control exists |

## Data Abstraction in DBMS

Hides internal complexity from users by showing only relevant details. Achieved through **3 levels**:

**Physical Level** (lowest)

- Describes *how* data is actually stored on hardware
- Deals with storage structures, file formats, indexes, access paths
- Managed by the OS and storage engine — users never see this

**Logical Level** (middle)

- Describes *what* data is stored and relationships between them
- Defines tables, fields, data types, constraints
- Programmers and DBAs work at this level — no concern about physical storage

**View Level** (highest)

- Shows only a *relevant portion* of the database to a specific user
- Different users see different views of the same data
- Simplifies interaction and adds a layer of security by hiding sensitive data

## Data Independence

Ability to change the schema at one level without affecting the schema at the next higher level.

**Two types:**

**Physical Data Independence**

- Change physical storage (file format, indexes, storage device) without affecting the logical schema
- The logical level doesn't need to know *where* or *how* data is stored physically
- DBA can reorganize storage, add indexes, or change file structures to improve performance — applications remain unaffected
- e.g., moving data from HDD to SSD, or changing from sequential to hashed file organization doesn't change table definitions

**Logical Data Independence**

- Change logical schema (add/remove a table or field) without affecting user views or application programs
- Views and external applications are insulated from structural changes in the logical layer
- e.g., adding a new column to a table or splitting a table into two doesn't break existing user views
- Harder to achieve than physical independence because applications are more tightly coupled to logical structure

**Importance:**

- Applications don't break when database internals change
- Reduces maintenance cost and effort
- Allows database to evolve without rewriting application code
- Provides flexibility to optimize storage independently
- Supports multiple user views without structural conflicts

## Schema vs Instance

**Schema** — the overall design/structure of the database defined at the time of creation. It describes table names, field names, data types, and constraints. Changes rarely.

**Instance** — the actual data stored in the database at a particular moment in time. Changes frequently as data is inserted, updated, or deleted.

| Schema | Instance |
|---|---|
| Structure of the database | Actual data at a point in time |
| Defined once, changes rarely | Changes with every insert/update/delete |
| Like a blueprint of a house | Like the people living in it at a moment |
| e.g., `Student(id, name, age)` | e.g., `(1, "Ram", 20)` |

## SQL Commands & Types

**DDL (Data Definition Language)**

- Defines and modifies the structure/schema of the database
- Changes made by DDL are auto-committed — cannot be rolled back
- `CREATE` — creates a new table, database, or index
- `ALTER` — adds, modifies, or drops columns in an existing table
- `DROP` — permanently deletes a table or database along with all its data
- `TRUNCATE` — removes all rows from a table but keeps the structure intact

**DML (Data Manipulation Language)**

- Used to manipulate the actual data stored in tables
- Changes can be rolled back (not auto-committed)
- `INSERT` — adds new records into a table
- `UPDATE` — modifies values of existing records based on a condition
- `DELETE` — removes specific records based on a condition (unlike TRUNCATE, it can target specific rows)

**DQL (Data Query Language)**

- Used purely to retrieve/query data from the database
- Does not modify any data
- `SELECT` — fetches data from one or more tables, supports filtering (`WHERE`), sorting (`ORDER BY`), grouping (`GROUP BY`)
- e.g., `SELECT name, age FROM Student WHERE age > 20;`

**DCL (Data Control Language)**

- Controls user access and permissions on database objects
- Managed by the DBA (Database Administrator)
- `GRANT` — gives specific privileges (SELECT, INSERT, etc.) to a user
- `REVOKE` — withdraws previously granted privileges from a user
- e.g., `GRANT SELECT ON Student TO user1;`
