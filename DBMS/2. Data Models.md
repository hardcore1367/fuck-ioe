# Data Models  [7 hours] [12 marks]

## 1. Data Models

A collection of tools and concepts used to describe structure, relationships, constraints, and semantics of data in a database. Provides an abstraction layer between physical storage and how users see data.

**Types:**

**Conceptual (High-level)** — describes *what* data exists and what relationships exist, independent of any DBMS. Closest to how end users think. Tool: **ER Diagram**. e.g., "Student enrolls in Course"

**Logical (Middle-level)** — describes *how* data is organized into tables, fields, keys. DBMS-type dependent but storage-independent. Tool: **Relational schema**. e.g., `Student(id, name, age)`

**Physical (Low-level)** — describes *how* data is actually stored on hardware. Deals with file formats, indexes, access paths. Managed by DBA/storage engine. e.g., B-tree index on `Student.id`

| | Conceptual | Logical | Physical |
|---|---|---|---|
| Focus | What data exists | How structured | How stored |
| Tool | ER Diagram | Relational schema | Indexes, files |
| DBMS dependent | No | Partially | Yes |

## 2. ER Model & ER Diagram

**ER Model** — high-level conceptual model representing real-world entities and relationships for database design.

**ER Diagram** — visual representation using standard symbols (rectangles, diamonds, ellipses).

**Purpose:**
- Blueprint for database design before implementation
- Maps real-world concepts into database schema
- Communicates structure between designers, developers, clients

## 3. Entities

**Strong Entity** — exists independently, has its own primary key. Represented by **single rectangle**. e.g., Student, Employee

**Weak Entity** — cannot exist without a parent strong entity. Has no primary key — identified by **partial key + parent's primary key**. Represented by **double rectangle**, its relationship by **double diamond**. e.g., Dependent of Employee

| Strong Entity | Weak Entity |
|---|---|
| Exists independently | Depends on strong entity |
| Has primary key | Has only partial key |
| Single rectangle | Double rectangle |
| e.g., Employee | e.g., Dependent |

## 4. Attributes

Properties/characteristics of an entity — represented by **ellipse**

- **Simple** — atomic, indivisible. e.g., age, gender
- **Composite** — divisible into sub-parts. e.g., Name → First, Last
- **Multi-valued** — can hold multiple values. e.g., phone numbers — **double ellipse**
- **Derived** — computed from another attribute, not stored. e.g., Age from DOB — **dashed ellipse**
- **Key attribute** — uniquely identifies entity — value **underlined** inside ellipse

## 5. Keys

- **Super key** — any set of attributes that uniquely identifies a tuple (may have redundant attributes)
- **Candidate key** — minimal super key — no subset of it can uniquely identify a tuple
- **Primary key** — selected candidate key — **underlined** in ER diagram
- **Foreign key** — references primary key of another table, enforces referential integrity
- **Partial key** — partially identifies weak entity combined with parent's PK — **dashed underline**

## 6. Relationship

Association between two or more entities — represented by **diamond**. e.g., Student *enrolls in* Course

**6.1 Degree** — number of entity types participating in a relationship
- **Unary (degree 1)** — entity relates to itself. e.g., Employee *manages* Employee
- **Binary (degree 2)** — two entities, most common. e.g., Student *enrolls in* Course
- **Ternary (degree 3)** — three entities. e.g., Doctor *prescribes* Medicine *to* Patient

**6.2 Cardinality** — number of instances of one entity associated with instances of another
- **1:1** — e.g., Person — Passport
- **1:N** — e.g., Department — Employees
- **M:N** — e.g., Student — Course

| Degree | Cardinality |
|---|---|
| Number of entity *types* in relationship | Number of entity *instances* participating |
| Structural property | Quantitative constraint |
| e.g., binary, ternary | e.g., 1:1, 1:N, M:N |

## 7. Participation

**Total Participation** — every instance *must* participate — **double line**. e.g., every Employee must belong to a Department

**Partial Participation** — some instances may not participate — **single line**. e.g., some Employees may not manage any project

## 8. Specialization & Generalization

**Specialization** — **top-down** process of dividing a general entity into specialized sub-entities. Sub-entities inherit parent attributes + have their own. Represented by **ISA triangle**. e.g., Employee → Manager, Engineer, Clerk

**Generalization** — **bottom-up** process of combining multiple entities with common attributes into one general entity. e.g., Car + Truck + Bike → Vehicle

**Importance:**
- Reduces redundancy — common attributes defined once at parent level
- Models real-world hierarchies naturally
- Supports **attribute inheritance** — sub-entities don't redefine common attributes

## 9. Aggregation

Treats a **relationship set as a higher-level entity** so it can participate in another relationship. Used when a relationship itself must relate to another entity. Represented by **rectangle drawn around the relationship**. e.g., (Employee *works on* Project) relationship is *managed by* Department

**Importance:**
- Models complex scenarios where relationships participate in other relationships
- Avoids duplicating relationship data
- Makes ER diagram cleaner and more expressive

| Aggregation | Generalization |
|---|---|
| Treats relationship as entity | Combines entities into general one |
| When relationship must relate to another entity | To reduce redundancy via common attributes |
| Abstraction of relationships | Abstraction of entities |
| e.g., (works-on) managed by Department | e.g., Car + Bike → Vehicle |
