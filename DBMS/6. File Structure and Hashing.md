## 1. Basic Terminology

**File** — A named collection of related records stored on secondary storage (disk). Represents a table in physical storage.

**Record** — A collection of related fields representing a single entity instance. e.g., one row in a Student table.

**Field** — smallest named unit of data in a record. Has a name, data type, and value. e.g., `student_id`, `name`, `age`

The database is stored as a collection of files. Each file is a sequence of records. A record is a sequence of fields.

```
File
 └── Records (rows)
       └── Fields (columns/attributes)
```

## 2. Types of Records

### 1. Fixed-Length Records
Every record occupies the **same number of bytes** — all fields have predefined fixed sizes.

**Structure:**
```
Record size = sum of all field sizes (always constant)

Student record: | id(4B) | name(20B) | age(4B) | = 28 bytes always

Record 1: [0001][Ram_________________][0020]
Record 2: [0002][Sita________________][0022]
Record 3: [0003][Hari________________][0021]
```
Unused space in fields is **padded** with spaces or null bytes.

**Locating any record directly:**
```
Record i starts at byte offset = (i-1) × record_size
Record 3 → offset = 2 × 28 = 56 bytes from start
```

**Adv:**
- Simple and fast — direct offset calculation
- Easy to reuse deleted slots via free list

**Disadv:**
- Wastes space — short values still occupy full allocated bytes
- Not suitable for varying-length data like names, descriptions

### 2. Variable-Length Records
Records have **different sizes** — some fields vary in length (VARCHAR, TEXT, repeated fields).

**Adv:**
- Space efficient — stores only actual data, no padding
- Handles real-world data naturally (names, descriptions)

**Disadv:**
- Slower access — no direct offset calculation
- Fragmentation over time requires periodic compaction

## Fixed vs Variable Length

| | Fixed-Length | Variable-Length |
|---|---|---|
| Record size | Always constant | Differs per record |
| Space usage | Wasteful (padding) | Efficient |
| Direct access | Yes — offset formula | No — must read header |
| Insertion | Simple — fixed slot | Complex — need right-sized slot |
| Deletion | Mark + free list | Mark + compaction needed |
| Search | Faster — fixed steps | Slower — variable steps |
| Example fields | INT, CHAR(n), DATE | VARCHAR, TEXT, BLOB |

## 3. Record Representation

**Byte String Representation**

- Each record is stored as a **continuous string of bytes** on disk
- A special **end-of-record (EOR) marker** (e.g., `$`) is placed at the end of each record to indicate where it ends
- Fields within the record are separated by **delimiter characters**
- Simple but hard to parse — must scan byte by byte to find field boundaries
- e.g., `Ram$20$CS$` where `$` marks field/record end

**Fixed-Length Representation**

- Each field is allocated a **fixed number of bytes** regardless of actual value size
- Record size = sum of all fixed field sizes — always constant
- Any unused space in a field is **padded** (with spaces or null bytes)
- Easy to locate any record or field using arithmetic — no need to scan
- e.g., `name` always occupies 20 bytes — "Ram" stored as `Ram_________________` (padded)

```
Record: | id (4B) | name (20B) | age (4B) |
Offset:   0         4            24
→ Field 'age' always at byte offset 24
```

| Byte String | Fixed-Length |
|---|---|
| Continuous bytes with delimiters | Fixed bytes per field |
| Uses EOR marker to find record end | Record size always constant |
| Space efficient | May waste space due to padding |
| Hard to locate fields directly | Easy random access via offset |

## 4. Record Organization in a File

### 5.1 Heap File Organization

Records inserted in **any available free space** — no particular order.

**Insertion** — scan for free slot, insert there. If none, append at end.

```
Initial: [Ram|20] [---deleted---] [Sita|22] [---deleted---] [Hari|21]
Insert "Gita|23" → fills first deleted slot
Result:  [Ram|20] [Gita|23] [Sita|22] [---deleted---] [Hari|21]
```

**Deletion** — record marked with a **deletion marker**, space reused later. Not physically removed immediately.

```
Delete Ram → [***deleted***] [Gita|23] [Sita|22] [---deleted---] [Hari|21]
```

**Search** — no order, must scan every record one by one until found

```
Search "Hari" → scan [Gita] → [Sita] → [Hari] ✓ (linear scan)
```

**Adv:**

- Very fast insertion — no sorting needed
- Simple to implement

**Disadv:**

- Slow search — full file scan required
- Deleted space accumulates, needs periodic compaction

### 5.2 Sequential File Organization

#### Unordered File Method

Records stored in **insertion order** — appended at end as they arrive.

**Insertion** — always append at end, no ordering maintained

```
[Ram|20] [Sita|22] [Hari|21] [Gita|23]  ← insertion order
Insert "Mohan|24" → [Ram|20] [Sita|22] [Hari|21] [Gita|23] [Mohan|24]
```

**Deletion** — mark as deleted

```
Delete Sita → [Ram|20] [***] [Hari|21] [Gita|23] [Mohan|24]
```

**Search** — linear scan from beginning

```
Search "Gita" → scan Ram → *** → Hari → Gita ✓
```

**Adv:**

- Fastest insertion (just append)
- Good for write-heavy, log-type data

**Disadv:**

- Slow search — no order to exploit
- Wasted space from deleted records

#### Sorted File Method (Ordered Sequential)

Records maintained in **sorted order** by a search key at all times.

**Insertion** — find correct position, shift all subsequent records to make room

```
Sorted by id: [1|Ram] [3|Sita] [5|Hari] [7|Gita]
Insert id=4  → shift 5,7 right
Result:        [1|Ram] [3|Sita] [4|New] [5|Hari] [7|Gita]
```

To avoid costly shifting — **overflow block** used temporarily, merged periodically

```
Main:     [1|Ram] [3|Sita] [5|Hari] [7|Gita]
Overflow: [4|New] [6|Xyz]
→ periodically merged and re-sorted
```

**Deletion** — mark as deleted, periodic reorganization removes gaps

```
Delete id=3 → [1|Ram] [***] [4|New] [5|Hari] [7|Gita]
→ after reorg: [1|Ram] [4|New] [5|Hari] [7|Gita]
```

**Search** — **binary search** possible since records are sorted

```
Search id=5 in [1,3,4,5,7,9,11,13]
Mid=5 ✓ → found in O(log n)
```

**Adv:**

- Fast search via binary search — O(log n)
- Excellent for range queries (e.g., id between 3 and 7)

**Disadv:**

- Slow insertion — shifting required
- Periodic reorganization needed for overflow blocks

---

### 5.3 Hash File Organization

**Hash function h(key)** computes bucket address where record is stored. File divided into fixed number of **buckets**.

**Insertion** — apply hash function, go to that bucket, insert

```
h(id) = id % 5      (5 buckets: 0,1,2,3,4)

Insert id=10 → h(10)=0 → Bucket 0: [10|Ram]
Insert id=23 → h(23)=3 → Bucket 3: [23|Sita]
Insert id=15 → h(15)=0 → Bucket 0: [10|Ram][15|Hari]  ← same bucket
Insert id=25 → h(25)=0 → Bucket 0 full → overflow bucket linked
```

```
Bucket 0: [10|Ram][15|Hari] → [25|Gita]  ← overflow chain
Bucket 1: []
Bucket 2: []
Bucket 3: [23|Sita]
Bucket 4: []
```

**Deletion** — compute h(key), locate record in bucket, remove it

```
Delete id=15 → h(15)=0 → scan Bucket 0 → find [15|Hari] → delete
Bucket 0: [10|Ram][***] → [25|Gita]
```

**Search** — compute h(key), go directly to bucket, scan only that bucket

```
Search id=23 → h(23)=3 → Bucket 3 → [23|Sita] ✓
Only 1 bucket scanned regardless of file size → O(1)
```

**Adv:**

- Extremely fast exact-match search — O(1)
- Fast insertion and deletion

**Disadv:**

- Cannot do range queries (e.g., id between 10 and 20) — no order maintained
- Overflow chains degrade performance when many collisions occur

---

### 5.4 B+ Tree File Organization

Records stored in **sorted leaf nodes** of a B+ tree. Leaf nodes linked as a **doubly linked list**. Internal nodes store only keys for routing.

**Structure:**

```
              [30 | 60]                 ← internal node (keys only)
             /    |     \
      [10|20]  [30|50]  [60|80]        ← leaf nodes (actual records)
          ↔        ↔        ↔          ← linked list between leaves
```

**Insertion** — find correct leaf using tree traversal, insert in sorted position

- If leaf has space → insert directly
- If leaf is full → **split leaf**, push middle key up to parent

```
Insert 40 into [30|50] → [30|40|50] → full → split
→ [30|40] and [50], push 50 up to parent
Parent: [30 | 50 | 60]
```

**Deletion** — find record in leaf, remove it

- If leaf still has enough entries → done
- If **underflow** → borrow from sibling or **merge** with sibling, remove key from parent

```
Delete 10 from [10|20] → [20] → underflow if min=2
→ merge with sibling or borrow from [30|50]
```

**Search** — traverse from root to correct leaf using keys

```
Search 50:
Root [30|60] → 50 > 30, 50 < 60 → go middle child
[30|50] → found 50 ✓
O(log n) — tree always balanced
```

**Range query** — find start leaf, then follow linked list

```
Range 30 to 60:
→ find leaf with 30 → [30|40] → follow link → [50] → follow link → [60] ✓
```

**Adv:**

- Always balanced — O(log n) for search, insert, delete
- Excellent range queries via leaf linked list

**Disadv:**

- Complex implementation — splits and merges add overhead
- Extra space used by internal nodes storing duplicate keys

---

### Summary

| | Heap | Pile | Sorted Sequential | Hash | B+ Tree |
|---|---|---|---|---|---|
| Insert | Fast | Fastest | Slow | Fast | Moderate |
| Delete | Fast | Fast | Slow | Fast | Moderate |
| Exact Search | Slow O(n) | Slow O(n) | Fast O(log n) | Fastest O(1) | Fast O(log n) |
| Range Query | Poor | Poor | Good | Poor | Excellent |
| Ordered | No | No | Yes | No | Yes |

## 5. RAID (Redundant Array of Independent Disks)

**Definition** — a storage technology that combines multiple physical disks into one logical unit to improve **performance, reliability, or both** through data distribution and redundancy.

**Two key concepts:**
- **Striping** — splitting data across multiple disks for speed
- **Mirroring** — copying same data on multiple disks for redundancy
- **Parity** — extra computed data stored to reconstruct lost data on failure

---

### RAID Levels

### RAID 0 — Striping (No Redundancy)
- Data split into **blocks and distributed** across all disks in round-robin
- No redundancy — no mirror, no parity
```
Data: [A1][A2][A3][A4]
Disk 1: [A1][A3]
Disk 2: [A2][A4]
```
- **Read/Write** — both very fast, all disks work in parallel
- **Failure** — one disk fails → **all data lost**, no recovery possible
- **Use case** — video editing, gaming, temp data — where speed matters, data loss acceptable

**Adv:** Maximum performance, full disk capacity used
**Disadv:** Zero fault tolerance — most risky RAID level

---

### RAID 1 — Mirroring
- Every disk has an **exact mirror copy** — data written to both simultaneously
```
Disk 1: [A1][A2][A3]   (original)
Disk 2: [A1][A2][A3]   (mirror)
```
- **Read** — fast, can read from either disk
- **Write** — slower, must write to both disks
- **Failure** — one disk fails → other disk has full copy, no data lost
- **Storage efficiency** — 50% — half the total capacity wasted on mirror
- **Use case** — OS drives, critical databases, boot drives

**Adv:** High reliability, fast read
**Disadv:** Expensive — 50% storage wasted

---

### RAID 5 — Striping with Distributed Parity
- Data and **parity blocks distributed** across all disks — no dedicated parity disk
- Parity stored on a different disk for each stripe
- Minimum **3 disks** required
```
Disk 1: [A1][B1][C1][P_D]
Disk 2: [A2][B2][P_C][D2]
Disk 3: [A3][P_B][C3][D3]
Disk 4: [P_A][B4][C4][D4]
P = parity block for that stripe
```
- **Parity calculation:** `P = A1 XOR A2 XOR A3` — if one block lost, XOR remaining to recover
- **Read** — fast, parallel reads across all disks
- **Write** — slower — must read old data + old parity, compute new parity, write data + parity (4 I/O operations per write)
- **Failure** — tolerates **1 disk failure** — rebuild from parity
- **Storage efficiency** — `(n-1)/n` — e.g., 4 disks → 75% usable
- **Use case** — general purpose servers, file servers

**Adv:** Good balance of performance, storage efficiency, fault tolerance
**Disadv:** Only 1 disk failure tolerated — risky during rebuild

---

### RAID 6 — Striping with Double Distributed Parity
- Same as RAID 5 but with **two independent parity blocks** per stripe (P and Q)
- Uses two different parity algorithms (XOR + Galois Field math)
- Minimum **4 disks** required
```
Disk 1: [A1][B1][P_A][Q_A]
Disk 2: [A2][B2][P_B][Q_B]
Disk 3: [P1][B3][A3][Q_C]
Disk 4: [Q1][P2][A4][B4]
Two parity blocks per stripe stored on different disks
```
- **Failure** — tolerates **2 simultaneous disk failures** — recovers using two parities
- **Read** — same as RAID 5 — fast
- **Write** — slower than RAID 5 — must compute and write **two** parity blocks
- **Storage efficiency** — `(n-2)/n` — e.g., 6 disks → 66% usable
- **Use case** — large storage arrays, enterprise storage, critical data

**Adv:** Survives 2 simultaneous failures, safer during long rebuilds
**Disadv:** Higher write overhead, slightly less storage efficiency than RAID 5

---

### RAID 10 (1+0) — Mirroring + Striping
- Combines RAID 1 (mirror) and RAID 0 (stripe) — minimum **4 disks**
- First mirror pairs of disks, then stripe across mirrors
```
Disk 1: [A1][A2]  ┐ mirrored pair 1
Disk 2: [A1][A2]  ┘
Disk 3: [A3][A4]  ┐ mirrored pair 2
Disk 4: [A3][A4]  ┘
Striped across pair 1 and pair 2
```
- **Failure** — can survive **one disk failure per mirror pair** (potentially 2 total if different pairs)
- **Read/Write** — very fast — benefits of both striping and mirroring
- **Storage efficiency** — 50% — same as RAID 1
- **Use case** — high-performance databases, transactional systems

**Adv:** Best performance + redundancy combination
**Disadv:** Expensive — 50% storage wasted

---

## Why RAID 6 is Better than RAID 5

| | RAID 5 | RAID 6 |
|---|---|---|
| Parity blocks | 1 per stripe | 2 per stripe |
| Disk failures tolerated | 1 | 2 |
| Rebuild risk | High | Low |
| Write performance | Faster | Slightly slower |
| Storage efficiency | (n-1)/n | (n-2)/n |
| Minimum disks | 3 | 4 |

**Key reason — Rebuild Window Risk:**
- Modern disks are very large (4TB, 8TB, 16TB) — rebuilding a failed RAID 5 array takes **hours to days**
- During rebuild, remaining disks are under heavy stress — probability of **second disk failure is high**
- RAID 5 cannot survive a second failure → **total data loss**
- RAID 6 tolerates a second failure even during rebuild → data is safe throughout the entire rebuild process
- For large arrays, RAID 6 is now considered the **minimum safe standard**

---

## How to Choose RAID Type

| Requirement | Best RAID |
|---|---|
| Maximum speed, no redundancy needed | RAID 0 |
| Simple redundancy, small setup | RAID 1 |
| Balance of speed + redundancy + efficiency | RAID 5 |
| High reliability, large arrays, critical data | RAID 6 |
| Maximum performance + redundancy, budget available | RAID 10 |
| Large capacity with fault tolerance | RAID 6 |

**Key factors to consider:**
- **Fault tolerance needed** — how many disk failures must be survived
- **Budget** — more redundancy = more disks = more cost
- **Read/Write ratio** — write-heavy workloads suffer more on RAID 5/6
- **Data criticality** — financial/medical data → RAID 6 or RAID 10
- **Rebuild time concern** — large disks → RAID 6 over RAID 5


## Indexing

**Definition** — a data structure that provides **fast access to records** in a file based on the value of a search key, without scanning the entire file.

**Structure of an Index Entry:**
```
Index file:
| Search Key | Pointer to Record |
|------------|-------------------|
| 10         | → Block 1, offset 0  |
| 20         | → Block 1, offset 28 |
| 30         | → Block 2, offset 0  |
```
- **Search key** — attribute(s) used to look up records
- **Pointer** — points to the actual record in the data file
- Index file is much smaller than data file — fits in fewer disk blocks


## 6. Types of Indices

### 1. Primary Index (Clustering Index)
- Built on **ordered file** where data file is **physically sorted** by the search key
- Search key = ordering key of the file (usually primary key)
- One index entry per **block** of data file (not per record)
- Index entry points to **first record of each block**
```
Data file (sorted by id):
Block 1: [1|Ram] [2|Sita] [3|Hari]
Block 2: [4|Gita] [5|Mohan] [6|Lal]
Block 3: [7|Dev] [8|Kiran] [9|Nita]

Primary Index:
| Key | Pointer |
|  1  | → Block 1 |
|  4  | → Block 2 |
|  7  | → Block 3 |
```
- **Search** — binary search on index → find block → scan block
- Index is always **sparse** (one entry per block, not per record)

**Adv:** Small index size, fast search
**Disadv:** Data file must be physically sorted — insertion is costly

---

### 2. Secondary Index
- Built on a **non-ordering field** — data file may not be sorted by this key
- Can be built on **any attribute** (non-key or key)
- One index entry **per record** (dense) since data is not sorted — cannot skip records
- Points directly to the record containing that search key value
```
Data file (sorted by id, NOT by dept):
[1|Ram|CS] [2|Sita|EE] [3|Hari|CS] [4|Gita|EE]

Secondary Index on dept:
| Key | Pointer      |
| CS  | → [1|Ram|CS] |
| CS  | → [3|Hari|CS]|
| EE  | → [2|Sita|EE]|
| EE  | → [4|Gita|EE]|
```
- Multiple records can have same search key → **bucket pointers** used (pointer to list of matching records)
- Always **dense**

**Adv:** Enables fast search on non-primary attributes
**Disadv:** Large index size (one entry per record), expensive to maintain on updates

---

### 3. Clustered vs Non-Clustered Index

**Clustered** — physical order of records on disk **matches** the index order
- Records with close key values stored close together physically
- Only **one** clustered index per table (can't sort physically two ways)
- e.g., primary index on `id` — records stored in id order on disk

**Non-Clustered** — physical order of records **does not match** index order
- Index is separate structure with pointers back to records
- Multiple non-clustered indexes allowed per table
- e.g., secondary index on `dept` — records not sorted by dept on disk

| Clustered | Non-Clustered |
|---|---|
| Physical order matches index | Physical order independent |
| One per table | Multiple per table |
| Faster range queries | Slower range queries |
| Usually primary index | Usually secondary index |

---

### 4. Dense vs Sparse Index

**Dense Index**
- **One index entry for every record** in the data file
- Index contains search key + pointer for each and every record
- Can directly check if a record exists just from the index

```
Data file:
Record 1: [1|Ram]
Record 2: [2|Sita]
Record 3: [3|Hari]
Record 4: [4|Gita]

Dense Index:
| 1 | → Record 1 |
| 2 | → Record 2 |
| 3 | → Record 3 |
| 4 | → Record 4 |
```
Search id=3:
```
Scan index → find key=3 → follow pointer → directly reach [3|Hari] ✓
No need to touch data file until exact location known
```

**Adv:** Faster search — record found directly from index
**Disadv:** Large index — one entry per record, more storage, more I/O to load index

---

**Sparse Index**
- **One index entry per block** (not per record) of the data file
- Only works if data file is **physically sorted** by search key
- Index entry points to **first record of each block**
- To find a record → find largest key ≤ search key in index → go to that block → scan block

```
Data file (sorted, 3 records per block):
Block 1: [1|Ram] [2|Sita] [3|Hari]
Block 2: [4|Gita] [5|Mohan] [6|Lal]
Block 3: [7|Dev] [8|Kiran] [9|Nita]

Sparse Index:
| 1 | → Block 1 |
| 4 | → Block 2 |
| 7 | → Block 3 |
```
Search id=5:
```
Scan index → largest key ≤ 5 is 4 → go to Block 2
Scan Block 2: [4|Gita] [5|Mohan] ✓ → found
```
Search id=8:
```
Largest key ≤ 8 is 7 → go to Block 3
Scan Block 3: [7|Dev] [8|Kiran] ✓ → found
```

**Adv:** Much smaller index — one entry per block, less storage, fits in memory easily
**Disadv:** Slower than dense — must scan within the block after reaching it. Requires sorted data file

---

## Dense vs Sparse Summary

| | Dense Index | Sparse Index |
|---|---|---|
| Entries | One per record | One per block |
| Index size | Large | Small |
| Search speed | Faster — direct pointer | Slightly slower — block scan needed |
| Requires sorted file | No | Yes |
| Used with | Secondary index, any file | Primary index, sorted file |
| Memory fit | Harder | Easier |


## Hashing

**Definition** — a technique to directly compute the **storage location (bucket)** of a record using a **hash function** on the search key, enabling O(1) average access time.

**Hash Function:** `h(key) → bucket address`
```
h(student_id) = student_id % 5
id=23 → bucket 3
id=10 → bucket 0
```

**Bucket** — a unit of storage (disk block or page) that holds one or more records. Multiple records can map to same bucket → **collision** → handled by **overflow chaining**

```
Bucket 0: [10|Ram] → [15|Hari] → [25|Gita]  ← overflow chain
Bucket 1: [11|Sita]
Bucket 2: [22|Dev]
Bucket 3: [23|Mohan]
Bucket 4: [19|Kiran]
```

---

## 1. Static Hashing

**Definition** — fixed number of buckets allocated at the start. Hash function always maps to the **same fixed set of buckets** — bucket count never changes.

**How it works:**
- Define `n` buckets at creation time
- `h(key) = key % n` always
- All insertions, deletions, searches use same fixed `n`

**Insertion:**
```
n=5 buckets
Insert id=10 → h(10)=0 → Bucket 0: [10|Ram]
Insert id=15 → h(15)=0 → Bucket 0: [10|Ram][15|Hari]
Insert id=25 → h(25)=0 → Bucket 0 full → overflow bucket linked
Bucket 0: [10|Ram][15|Hari] → [25|Gita][overflow]
```

**Search:**
```
Search id=15 → h(15)=0 → go Bucket 0
Scan: [10|Ram] → [15|Hari] ✓
```

**Deletion:**
```
Delete id=10 → h(10)=0 → scan Bucket 0 → remove [10|Ram]
Bucket 0: [15|Hari] → [25|Gita]
```

**Problem — Overflow:**
- Too many records hash to same bucket → long overflow chains
- Performance degrades from O(1) to O(n) in worst case
- Solution: **periodic rehashing** — create new larger table, re-insert all records
  - Expensive — entire file must be reorganized

**Adv:**
- Very simple to implement and understand
- O(1) average search, insert, delete
- Predictable bucket locations — no metadata overhead
- Works well when data size is known in advance

**Disadv:**
- Fixed bucket count — cannot grow/shrink with data
- Overflow chains degrade performance as data grows
- Periodic rehashing is very expensive — entire file reorganized
- Wasted space if buckets allocated too many, overflow if too few

---

## 2. Dynamic Hashing (Extendible Hashing)

**Definition** — hash table that **grows and shrinks dynamically** as data increases or decreases. No full rehashing — only the affected bucket is split.

**Key idea:**
- Uses a **directory** (array of pointers to buckets)
- Directory has a **global depth** — number of bits of hash value used to index directory
- Each bucket has a **local depth** — number of bits that define which records it holds
- When bucket overflows → **split only that bucket**, double directory if needed

**Structure:**
```
Global depth = 2 (use 2 bits of hash)
Directory:
00 → Bucket A (local depth=2): [records with hash 00...]
01 → Bucket B (local depth=2): [records with hash 01...]
10 → Bucket C (local depth=2): [records with hash 10...]
11 → Bucket D (local depth=2): [records with hash 11...]
```

**Insertion — bucket split example:**
```
Insert record, maps to Bucket A (00) → Bucket A is full
Split Bucket A:
→ Bucket A keeps records with hash 000...
→ New Bucket A' gets records with hash 001...
If local depth < global depth → just update directory pointer
If local depth = global depth → double the directory first
```

**Before split (global depth=1):**
```
Directory:
0 → Bucket A: [h=0 records]   (local depth=1)
1 → Bucket B: [h=1 records]   (local depth=1)
```

**After Bucket A overflows — double directory (global depth=2):**
```
Directory:
00 → Bucket A:  [h=00 records]  (local depth=2)
01 → Bucket A': [h=01 records]  (local depth=2)
10 → Bucket B                   (local depth=1, both point to B)
11 → Bucket B
```

**Search:**
```
Search key k → compute h(k) → take first (global depth) bits
→ index into directory → follow pointer → scan bucket
```

**Deletion:**
- Remove record from bucket
- If bucket becomes underfull → **merge** with sibling bucket
- If directory becomes half-empty → **halve** the directory
- Much less common than splits in practice

**Adv:**
- Grows and shrinks dynamically — no wasted space as data changes
- Only overflowed bucket is split — no full file reorganization
- No long overflow chains — performance stays near O(1)
- Space-efficient — directory doubles only when necessary

**Disadv:**
- Directory can grow very large if data is skewed (many records hash to same prefix)
- Two disk accesses needed — one for directory, one for bucket
- More complex to implement than static hashing
- Directory doubling can still be expensive for large directories

---

## Static vs Dynamic Hashing

| | Static Hashing | Dynamic Hashing |
|---|---|---|
| Bucket count | Fixed at creation | Grows/shrinks dynamically |
| Overflow handling | Overflow chains | Bucket splitting |
| Rehashing | Full file rehash needed | Only split bucket rehashed |
| Performance over time | Degrades with overflow | Stays near O(1) |
| Implementation | Simple | Complex |
| Space usage | Wasteful or insufficient | Efficient |
| Best for | Known, fixed data size | Unknown or growing data |
