# 8. Transaction and Concurrency Control

A **transaction** is a sequence of operations treated as one logical unit of work. Concurrency improves performance by allowing several transactions to run together, but a concurrency-control protocol must prevent their conflicting operations from producing an incorrect result.

## 1. Transactions

### 1.1 ACID Properties

1. **Atomicity:** Either every operation of the transaction completes, or none of its effects remains.
2. **Consistency:** The transaction moves data from one valid state to another while preserving system rules.
3. **Isolation:** Concurrent transactions behave as if they executed separately in some serial order.
4. **Durability:** After commit, the result survives process or system failure.

### 1.2 Flat, Nested, and Distributed Transactions

#### 1.2.1 Flat Transaction

A **flat transaction** contains one sequence of operations with one final commit or abort. It has no internal subtransactions. If any operation fails, the whole transaction aborts.

#### 1.2.2 Nested Transaction

A **nested transaction** is organized as a tree with one top-level transaction and several child subtransactions. Children may execute concurrently and may themselves contain more children.

A child commit is **provisional**: its result becomes permanent only when the top-level transaction commits. A failed child can often be retried or replaced without immediately aborting every independent child, but aborting the top-level transaction cancels all its results.

#### 1.2.3 Distributed Transaction

A **distributed transaction** accesses resources managed at more than one network site. Each participating server executes a local part, while an atomic commit protocol ensures that every site reaches the same final commit or abort decision.

### 1.3 Transaction-Type Comparison

| S.N. | Flat transaction | Nested transaction | Distributed transaction |
|---:|---|---|---|
| 1 | It consists of one sequence of operations without internal subtransactions. | It consists of a hierarchy of parent and child subtransactions. | It consists of operations performed by resource managers at multiple sites. |
| 2 | It has one final commit or abort point. | Child commits are provisional until the top-level transaction commits. | All participating sites must agree on one commit or abort result. |
| 3 | Failure of any operation normally aborts the whole transaction. | A failed child may be retried or replaced without immediately cancelling independent children. | Failure of one site or coordinator may delay or abort the global transaction. |
| 4 | It provides little internal concurrency or failure isolation. | Independent child transactions may execute concurrently and isolate some failures. | Work may execute concurrently at different sites but requires network coordination. |
| 5 | It is suitable for a simple local unit of work. | It is suitable for complex work naturally divided into smaller tasks. | It is suitable when one logical operation updates resources on different machines. |

## 2. Concurrency-Control Methods

**Concurrency control** coordinates conflicting reads and writes so that the result is equivalent to a correct serial execution. Without it, transactions may cause lost updates, dirty reads, inconsistent retrievals, and cascading aborts.

### 2.1 Locks

A **lock** gives a transaction controlled access to a data item.

- A **shared lock (S)** permits reading. Several transactions may hold shared locks on the same item.
- An **exclusive lock (X)** permits writing. No other shared or exclusive lock may exist on that item.

Two shared locks are compatible. Every pair involving an exclusive lock is incompatible.

Locking prevents conflicting access, but transactions may wait for each other and form a deadlock.

### 2.2 Two-Phase Locking

**Two-Phase Locking (2PL)** divides each transaction's locking activity into two phases:

1. **Growing phase:** The transaction may acquire locks but cannot release any lock.
2. **Shrinking phase:** The transaction may release locks but cannot acquire a new lock.

The point at which a transaction holds its maximum set of locks is its **lock point**. Ordering transactions by their lock points makes a 2PL schedule conflict-serializable.

**Limitations**

- Transactions can deadlock while waiting for one another's locks.
- A transaction may release a write lock before commit. Another transaction can then read that uncommitted value, causing a cascading abort if the writer later aborts.

### 2.3 Strict Two-Phase Locking

**Strict 2PL** follows ordinary 2PL but holds every exclusive lock until the transaction commits or aborts.

Because no other transaction can read or overwrite an uncommitted written value, strict 2PL prevents dirty reads and cascading aborts. It provides conflict serializability and recoverability, but longer lock holding may reduce concurrency.

### 2.4 Two-Phase Locking Comparison

| S.N. | Two-Phase Locking | Strict Two-Phase Locking |
|---:|---|---|
| 1 | A transaction releases locks during its shrinking phase. | A transaction holds all exclusive locks until commit or abort. |
| 2 | Another transaction may read a value written by an uncommitted transaction after its lock is released. | Another transaction cannot read or overwrite an uncommitted written value. |
| 3 | Cascading aborts are possible. | Cascading aborts caused by dirty reads are prevented. |
| 4 | It guarantees conflict serializability. | It guarantees conflict serializability and produces a recoverable, cascadeless schedule. |
| 5 | Earlier lock release can provide more concurrency. | Longer exclusive-lock holding may reduce concurrency. |

### 2.5 Optimistic Concurrency Control

**Optimistic Concurrency Control (OCC)** assumes conflicts are rare. Transactions execute without locks and are checked only before commit.

#### 2.5.1 Phases

1. **Read phase:** The transaction reads committed data, performs its work, and stores proposed writes in a private workspace.
2. **Validation phase:** The system checks whether committing the transaction would conflict with overlapping transactions and violate serializability.
3. **Write phase:** If validation succeeds, private changes are applied atomically. If it fails, the transaction aborts and restarts.

#### 2.5.2 Advantages and Limitations

**Advantages**

- Transactions do not wait for locks, so lock deadlock cannot occur.
- Read-heavy and low-conflict workloads obtain high concurrency.
- Read operations have little coordination overhead.

**Limitations**

- A conflict discovered late wastes all work performed by the aborted transaction.
- High-contention workloads may cause repeated aborts and starvation.
- Validation and atomic installation are harder across several distributed sites.

### 2.6 Timestamp Ordering

**Timestamp-ordering concurrency control** gives each transaction $T$ a unique timestamp $TS(T)$. Conflicting operations must occur in timestamp order, where a smaller timestamp represents an older transaction.

For every data item $X$, the system stores:

- $RTS(X)$: the largest timestamp of any transaction that successfully read $X$.
- $WTS(X)$: the timestamp of the transaction that most recently wrote $X$.

#### 2.6.1 Read Rule

When transaction $T$ requests **read($X$)**:

1. If $TS(T) < WTS(X)$, the requested version has already been overwritten by a newer transaction, so $T$ aborts and restarts.
2. Otherwise, the read is allowed and $RTS(X)$ becomes $\max(RTS(X), TS(T))$.

#### 2.6.2 Write Rule

When transaction $T$ requests **write($X$)**:

1. If $TS(T) < RTS(X)$, a newer transaction has already read the old value, so allowing the write would violate timestamp order. Transaction $T$ aborts.
2. If $TS(T) < WTS(X)$, a newer transaction has already written $X$. The system either aborts $T$ or ignores this obsolete write under **Thomas's write rule**.
3. Otherwise, the write is performed and $WTS(X)=TS(T)$.

Timestamp ordering is deadlock-free because transactions do not wait, but repeated restarts may starve a transaction.

### 2.7 Concurrency-Control Comparison

| S.N. | Lock-based control | Optimistic control | Timestamp ordering |
|---:|---|---|---|
| 1 | A transaction controls conflicts by acquiring shared and exclusive locks. | A transaction executes freely and validates its work before commit. | A transaction follows the order defined by fixed timestamps. |
| 2 | Conflicting transactions wait for locks. | Conflicting transactions are normally detected during validation. | An operation violating timestamp order causes an abort or an obsolete write is ignored. |
| 3 | Deadlock is possible because transactions may wait in a cycle. | Lock deadlock is impossible because transactions do not wait for locks. | Deadlock is impossible because operations are rejected instead of waiting. |
| 4 | It performs well when conflicts are common and waiting is cheaper than restarting. | It performs well when conflicts are rare and transactions are mainly reads. | It provides a clear global ordering but may restart transactions frequently. |
| 5 | Its main costs are lock management, blocking, and deadlock handling. | Its main costs are validation and wasted work after a late conflict. | Its main costs are timestamp metadata and possible starvation. |

## 3. Distributed Transactions and Atomic Commit

A **distributed transaction** has one coordinator and several participants at different sites. Atomic commitment requires all participants to commit or all to abort, even when processes communicate only through messages.

### 3.1 Two-Phase Commit Protocol

**Two-Phase Commit (2PC)** uses a voting phase followed by a decision phase.

#### 3.1.1 Phase 1: Voting

1. The coordinator writes the start information to stable storage and sends **VOTE-REQUEST** to all participants.
2. Each participant checks whether it can commit its local work.
3. A participant that can commit records its prepared state, keeps the required resources, and replies **VOTE-COMMIT**.
4. A participant that cannot commit records an abort and replies **VOTE-ABORT**.

#### 3.1.2 Phase 2: Decision

1. If every participant votes to commit, the coordinator records **GLOBAL-COMMIT** and sends it to all participants.
2. If any participant votes to abort or fails to vote before timeout, the coordinator records and sends **GLOBAL-ABORT**.
3. Each participant records the decision, commits or rolls back locally, releases its resources, and sends an acknowledgement.

### 3.2 Main Problem of Two-Phase Commit

After voting **VOTE-COMMIT**, a participant enters the **READY** or **uncertain** state. It cannot decide alone because the coordinator may already have selected either global commit or global abort.

If the coordinator crashes before the participant learns the decision, the participant may remain blocked while holding locks until the coordinator recovers or another participant supplies the recorded decision. Thus, 2PC guarantees atomicity but is a **blocking protocol**.

### 3.3 Three-Phase Commit Protocol

**Three-Phase Commit (3PC)** adds a **pre-commit phase** between voting and final commit. Its goal is to prevent participants from remaining indefinitely in an uncertain state after a fail-stop coordinator crash.

#### 3.3.1 Phase 1: Can Commit

1. The coordinator sends **VOTE-REQUEST**.
2. Participants reply **VOTE-COMMIT** or **VOTE-ABORT**.
3. Any abort vote causes **GLOBAL-ABORT**.

#### 3.3.2 Phase 2: Pre-Commit

1. If every vote is commit, the coordinator records **PRE-COMMIT** and sends **PRE-COMMIT** to all participants.
2. Each participant records the pre-commit state, replies **ACK**, and knows that every participant voted to commit.

#### 3.3.3 Phase 3: Do Commit

1. After receiving the required acknowledgements, the coordinator records and sends **GLOBAL-COMMIT**.
2. Each participant commits, releases its resources, and acknowledges the decision.

### 3.4 How 3PC Reduces the 2PC Blocking Problem

- A participant in the initial state can safely abort after timeout because no commit decision has become possible.
- A participant in the pre-commit state knows that all participants voted to commit and can cooperate with the others to complete the transaction.
- No single non-final state has direct transitions to both commit and abort, reducing uncertainty after coordinator failure.

3PC is non-blocking only under stronger assumptions such as bounded communication delay, reliable failure detection, and fail-stop failures. Network partitions can violate these assumptions, so 3PC is less common in practice.

## 4. Distributed Deadlocks

A **distributed deadlock** occurs when transactions at different sites wait in a cycle and none can proceed. No single site may hold the complete wait-for information, making detection and prevention harder than in one system.

For example, $T_1$ may hold a lock at site $S_1$ and wait for $T_2$ at $S_2$, while $T_2$ holds the required lock at $S_2$ and waits for $T_1$ at $S_1$.

### 4.1 Deadlock Avoidance and Prevention Approaches

#### 4.1.1 Global Resource Ordering

All transactions request resources in one predefined global order. A circular wait cannot form because every wait follows the same direction. This is simple but may reduce concurrency and requires advance ordering of resources.

#### 4.1.2 Wait-Die Scheme

Wait-die uses transaction timestamps:

- If the requester is older than the lock holder, the requester waits.
- If the requester is younger, it aborts or “dies” and later restarts with its original timestamp.

Wait edges always go from older to younger transactions, so a cycle cannot form.

#### 4.1.3 Wound-Wait Scheme

Wound-wait reverses the waiting rule:

- If the requester is older, it forces the younger holder to abort or “wounds” it.
- If the requester is younger, it waits for the older holder.

Wait edges always go from younger to older transactions, preventing a cycle.

#### 4.1.4 Timeout

A transaction that waits longer than a fixed limit is aborted and restarted. Timeout is easy to implement, but a slow transaction may be aborted even when no deadlock exists.

### 4.2 Phantom Deadlock

A **phantom deadlock** is a deadlock reported from stale distributed wait-for information even though the real wait cycle has already disappeared or never existed at one instant.

It occurs because dependency messages travel at different speeds. A detector may combine an old wait edge from one site with a new edge from another and construct a false cycle. The resulting unnecessary abort can be reduced by using timestamps, sequence numbers, consistent snapshots, or confirmation before selecting a victim.

## 5. Cascading Aborts

A **cascading abort** occurs when one transaction reads data written by another uncommitted transaction. If the writer aborts, the reader has used an invalid value and must also abort. Any transaction that used the reader's result may then abort as well.

**Example:** $T_1$ writes $X$, $T_2$ reads that uncommitted $X$ and writes $Y$, and then $T_1$ aborts. Transaction $T_2$ must abort because its result depends on a value that no longer exists.

### 5.1 Prevention

- **Strict 2PL:** Hold exclusive locks until commit or abort so no other transaction reads uncommitted writes.
- **Cascadeless schedules:** Allow a transaction to read a value only after the writer commits.
- **Recoverable schedules:** Do not allow a reader to commit before the transaction whose value it read.
- **Private workspaces:** Optimistic control keeps tentative writes private until validation and commit.
