An **Operating System (OS)** is system software that manages hardware resources (CPU, memory, storage, I/O) and provides services to applications and users. It acts as an intermediary between hardware and software.

## 1. OS Architecture — Monolithic vs Microkernel

| **S.N.** | **Monolithic OS** | **Microkernel OS** |
|---|---|---|
| 1 | The entire operating system runs as a single large program in kernel space. | Only essential services like IPC and basic scheduling run in the kernel; the rest run in user space. |
| 2 | All services such as memory management, file system, and device drivers are tightly coupled together. | Services like file systems and device drivers run as separate user-level processes, making them loosely coupled. |
| 3 | It is faster in performance because there are no mode switches between components. | It is slightly slower because communication between components requires message passing. |
| 4 | A bug in any one service can crash the entire operating system. | A bug in one service does not affect the kernel or other services, making it more reliable. |
| 5 | It is difficult to maintain, extend, or update because everything is packed into one block. | It is easy to extend and maintain because individual services can be added or updated independently. |
| 6 | Examples include early UNIX and Linux, which follow the monolithic structure. | Examples include Mach, Minix, and QNX, which follow the microkernel structure. |

## 2. Network OS vs Distributed OS

| **S.N.** | **Network OS** | **Distributed OS** |
|---|---|---|
| 1 | Each machine has its own local operating system and manages its own resources independently. | A single unified operating system manages all the machines and their resources collectively. |
| 2 | Users are aware of the different machines and must explicitly log in or access remote resources. | Users are not aware of the underlying machines as the system appears as a single unified machine. |
| 3 | Resource sharing is possible but the user must know the location of the resource to access it. | Resources are shared transparently without the user needing to know their physical location. |
| 4 | It is easier to design and implement as each machine runs its own independent OS. | It is more complex to design as it requires coordination and communication across all machines. |
| 5 | Failure of one machine does not affect other machines since each operates independently. | Failure of one machine can potentially affect the overall system depending on the design. |
| 6 | Examples include Novell NetWare and Windows Server, which are common network operating systems. | Examples include Amoeba and Plan 9, which are true distributed operating systems. |

**Why NOS is Preferred over DOS:**

1. NOS is easier to design and implement because each machine runs its own independent OS without requiring global coordination.
2. NOS is commercially mature and widely available, whereas true DOS is mostly experimental and limited.
3. Each machine in NOS can be managed independently, making administration simpler.
4. NOS can be deployed over existing operating systems without replacing them on all machines.
5. Adding or removing a machine does not disrupt the rest of the system, making NOS more scalable.
6. NOS gives users and administrators more direct control over individual machines and their local resources.

## 3. Distributed OS — Definition and Characteristics

A **Distributed OS (DOS)** is a single unified operating system that manages a collection of independent machines connected over a network, making them appear and behave as one coherent system to the user. Unlike a Network OS where each machine runs its own local OS, a DOS spans all machines and controls their resources collectively through a single system image.

**Characteristics of a Distributed OS:**

1. **Single System Image** — the entire collection of machines appears as one computer to the user; the distribution is completely hidden.
2. **Transparency** — users and applications are unaware of which physical machine is running their process, where data is stored, or how many copies exist.
3. **Resource Sharing** — CPU, memory, storage, and other resources across all machines are managed and shared by the OS as a unified pool.
4. **Fault Tolerance** — the OS detects failures and automatically migrates processes or data to other nodes, keeping the system operational.
5. **Process Migration** — the OS can move a running process from one machine to another for load balancing or fault recovery without user intervention.
6. **Scalability** — new machines can be added to the system and the OS integrates them into the shared resource pool without disruption.
7. **Concurrency** — multiple processes run simultaneously across different machines, coordinated by the unified OS.

## 4. Processes and Threads

| **S.N.** | **Process** | **Thread** |
|---|---|---|
| 1 | A process is an independent program in execution with its own memory space and resources. | A thread is a lightweight unit of execution that shares memory and resources with other threads within the same process. |
| 2 | Creating and managing a process is expensive in terms of time and system resources. | Creating and managing a thread is much cheaper and faster compared to a process. |
| 3 | Processes do not share memory, so communication between them requires IPC mechanisms like sockets or pipes. | Threads share the same memory space, making communication between them faster and simpler. |
| 4 | A process is isolated, so failure of one process does not directly affect another. | A failure in one thread can potentially crash all other threads within the same process. |
| 5 | Context switching between processes is slow due to the overhead of saving and restoring separate memory spaces. | Context switching between threads is fast because they share the same address space. |
| 6 | In distributed systems, processes run on different machines and communicate over the network. | Threads are used within a single machine to handle multiple tasks concurrently, improving responsiveness. |

**Why Threads are Important in Distributed Systems:**

1. Threads allow a server to handle multiple client requests simultaneously without creating a new process for each request.
2. They improve performance by reducing the overhead of process creation and context switching.
3. Threads make it easier to overlap computation with communication, which is critical in distributed systems.
