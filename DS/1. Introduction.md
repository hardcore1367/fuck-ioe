# 1. Introduction to Distributed Systems

## 1. Distributed System and Examples

A **distributed system** is a collection of independent components (hardware/software) located on networked computers that communicate and coordinate by **passing messages** to achieve a common goal, while appearing as a single coherent system to the user.

- **Internet** — millions of networked machines share resources and services globally. It is called a **network of networks** because independently managed networks communicate through common protocols such as TCP/IP. Services such as the Web, email, file transfer, and cloud computing are distributed among many servers, routers, and client machines. If one server or route fails, other parts of the Internet can usually continue working or use an alternative route.
- **BitTorrent** — peers share file chunks without transferring the complete file through one central server. The file is divided into smaller pieces, and a tracker or distributed hash table helps a peer discover other peers that possess those pieces. A peer downloads different pieces from several peers simultaneously while also uploading its available pieces to others. This distributes storage and network load, removes the central download-server bottleneck, and can make popular files faster to download as more peers join.
- **Blockchain** — nodes maintain replicated copies of a shared, decentralized ledger. When a user submits a transaction, participating nodes verify it according to the network rules and use a consensus mechanism to agree on the next valid block. The accepted block is linked to earlier blocks and distributed to the nodes, making unauthorized alteration easy to detect. Since no single database server controls the complete record, the system can remain available even when some nodes fail.
- **SSO (Single Sign-On)** — one authentication service is accessed by many independent applications. The user first proves their identity to an identity provider, which issues a digitally protected token containing the authentication result. Participating applications validate this token instead of separately requesting the user's password. This allows one login to provide controlled access to several distributed services while centralizing authentication and reducing repeated password handling.

## 2. Advantages of DS

1. Fault tolerant — failure of one node does not crash the whole system.
2. Scalable — new nodes can be added without redesigning the system.
3. Resource sharing — CPU, storage, and data can be shared across nodes.
4. Better performance — tasks run in parallel across multiple machines.
5. Geographically distributed users can be served efficiently with low latency.

## 3. Disadvantages of DS

1. Complex to design, debug, and maintain due to concurrency and partial failures.
2. Data consistency is hard to maintain across replicated nodes.
3. Network dependency — any network failure can cause partial or complete service disruption.
4. Security is harder to enforce across multiple nodes and communication channels.
5. Higher overhead — message passing and coordination consume extra resources.

## 4. Design Goals and Challenges of Distributed System

### 4.1 Transparency

A distributed system should hide its distributed nature and appear as a single coherent system to the user. Users should not know where a resource is located, how many copies exist, or whether a node has failed and recovered. The web is a good example — a user types a URL without knowing which server in which country is responding.

Fully achieving transparency is extremely hard in practice. Network latency, partial failures, and replication lag all break the illusion of a single system. Access, location, migration, replication, concurrency, failure, performance, and scaling transparency each require separate mechanisms to implement.

#### 4.1.1 Types of Transparency

1. **Access Transparency** — hides differences in data representation and resource access mechanisms so that local and remote resources are accessed using identical operations. For example, the same file operations can open and read a local file or a remote file. Middleware performs communication, data conversion, and error handling behind the common interface.
2. **Location Transparency** — hides the physical or network location of a resource; a user accesses a resource by its name without knowing where it actually resides. Resource names therefore remain valid even when the server address is unknown to the user. A naming service translates the logical resource name into its current network location.
3. **Migration/Mobility Transparency** — hides the fact that a resource or process can be moved to a different location; access remains unaffected even if the resource has migrated. For example, a mobile user can move between network cells without restarting an active service. The system updates location information or forwards requests so existing users do not need to reconnect manually.
4. **Replication Transparency** — hides the fact that multiple copies of a resource exist across nodes; the user interacts with a single logical resource without knowing how many physical replicas are maintained. The system selects a suitable replica and keeps the copies consistent. It can direct a request to a nearby or healthy copy to improve response time and availability.
5. **Concurrency Transparency** — hides the sharing of a resource by multiple users; each user appears to have exclusive access even when many users are accessing the same resource simultaneously. Synchronization prevents their concurrent operations from corrupting the shared state. Locks, transactions, or operation ordering ensure that the final result remains correct.
6. **Failure Transparency** — hides the occurrence of a fault and its subsequent recovery; the system continues to function from the user's perspective even when a component has failed. The system may retry a lost request or redirect it to a healthy replica. For example, a replicated web service can move a client request to another server when the original server crashes.
7. **Performance Transparency** — hides the actions taken to reconfigure the system to improve performance under varying load; users experience consistent response times regardless of system load. The system may redistribute work, add replicas, or move resources without changing how clients use the service. Load balancing and caching are common mechanisms for providing this transparency.

### 4.2 Openness

A distributed system should offer services through standard, publicly defined protocols and interfaces so that components from different vendors can interoperate and new services can be added without breaking existing ones. The web is built on open standards like HTTP, HTML, and DNS so any browser or server can participate.

Maintaining openness while allowing the system to evolve is difficult. Changing a standard interface can break all components that depend on it. Balancing backward compatibility with extensibility requires very careful interface design.

### 4.3 Scalability

The system should handle growth in users, data, and resources without degrading performance. Scalability can be in size (more users), geography (more locations), or administration (more organizations). Cloud platforms like AWS scale by adding nodes on demand.

As more nodes are added, maintaining data consistency becomes harder, coordination overhead increases, and bottlenecks shift to unexpected places. Designing algorithms and protocols that remain efficient at large scale is a non-trivial engineering problem.

### 4.4 Fault Tolerance

The system should continue operating correctly even when some nodes or communication links fail. This is achieved through replication, redundancy, and automatic recovery. A blockchain network, for example, continues operating even if hundreds of nodes go offline simultaneously.

In a DS, nodes can fail silently without the rest of the system knowing — unlike a centralized system where failure is total and obvious. Detecting partial failures, deciding whether a node has crashed or is just slow, and recovering gracefully without corrupting state are all very difficult problems.

### 4.5 Concurrency

Multiple users and processes should be able to access and modify shared resources simultaneously without conflict. The system must handle synchronization so that concurrent operations produce consistent results. A distributed database must allow thousands of users to read and write simultaneously.

Without careful design, concurrency leads to race conditions, deadlocks, and inconsistent state. Designing correct synchronization mechanisms across nodes that do not share memory and communicate only through messages is significantly harder than in a single-machine system.

### 4.6 Security

Data and resources must be protected from unauthorized access across all nodes and communication channels. This includes authentication (verifying identity), authorization (controlling access), and encryption (protecting data in transit). TLS/SSL is used to secure communication between distributed components.

Because data travels over open networks and is stored across many nodes, the attack surface is far larger than a centralized system. Ensuring end-to-end security, preventing spoofing and eavesdropping, and managing access control consistently across all nodes is a major ongoing challenge.

### 4.7 Heterogeneity

The system should work seamlessly across different hardware, operating systems, programming languages, and network types. Middleware and standard protocols abstract the underlying differences so a Java client on Windows can communicate with a Python server on Linux without any special handling.

Every layer of the system — hardware, OS, language, network, database — can differ between nodes. Building middleware that correctly handles all these differences adds significant complexity. Security also becomes harder since different platforms may have different vulnerability profiles.

### 4.8 Performance

Tasks should be distributed across nodes to maximize throughput and minimize response time. Load balancing ensures no single node becomes a bottleneck. Distributed systems like Google's search infrastructure process billions of requests per day by spreading work across thousands of machines.

Achieving good performance in a DS is hard because network communication is orders of magnitude slower than local memory access. Minimizing the number of messages, reducing latency, and balancing load dynamically without central coordination requires careful design of algorithms and data placement strategies.

### 4.9 Resource Sharing

Hardware and software resources such as CPU, storage, files, and services should be efficiently shared among all users. This avoids duplication, reduces cost, and allows users to access resources they do not physically own. Cloud storage like Google Drive allows users to share files stored on remote servers.

Managing shared resources across nodes requires coordination to prevent conflicts, enforce access control, and maintain consistency. When multiple nodes try to access or modify the same resource simultaneously, the system must arbitrate correctly without becoming a bottleneck itself.

## 6. Architectural Models of Distributed System

An **architectural model** defines how components of a distributed system are structured, how they interact, and how responsibilities are distributed across nodes. It answers the question: _who talks to whom, and in what role?_

### 6.1 Client-Server Model

The most fundamental DS architecture. The system is divided into two distinct roles — a **client** that requests services and a **server** that provides them. The server runs continuously, listening for incoming requests. The client initiates communication, sends a request, waits for a response, and uses the result. A single server can handle multiple clients simultaneously. The server and client can run on different machines connected over a network. This model is simple, easy to manage, and centralizes control on the server side. However, the server can become a bottleneck and a single point of failure if not replicated. Examples: web servers (HTTP), email servers (SMTP), database servers.

### 6.2 Multi-Tier Architecture

An evolution of client-server where the system is split into multiple distinct layers, each with a specific responsibility. This improves modularity, maintainability, and scalability.

- **1-Tier** — All three concerns — user interface, business logic, and data storage — reside on a single machine. There is no distribution at all. Simple to build but impossible to scale. Example: a standalone desktop application like MS Paint or a local text editor.

- **2-Tier** — The system is split into a **client** (handles UI and sometimes some logic) and a **server** (handles data storage and processing). The client communicates directly with the database or backend server. Easy to implement but the client becomes heavy and the server is still a bottleneck. Example: a desktop billing application directly querying a remote database.

- **3-Tier** — The most widely used architecture in modern systems. Three separate layers: **Presentation Layer** (browser or client app handles UI), **Application/Business Logic Layer** (middle-tier server processes requests, applies rules), and **Data Layer** (database server stores and retrieves data). The client never talks directly to the database — all requests go through the middle tier. This separation makes the system more secure, maintainable, and scalable. Example: a web application where the browser talks to a Node.js/Django server, which talks to a PostgreSQL database.

### 6.3 Peer-to-Peer (P2P) Model

In the P2P model there is no dedicated server or client — every node in the network is called a **peer** and acts as both simultaneously. A peer can request resources from others and also serve resources to others at the same time. There is no central authority or single point of control, making the system fully decentralized. Because all peers share the load, the system is highly fault tolerant — removing any single peer does not bring down the network. It also scales naturally as adding more peers increases the system's total capacity rather than adding load. The challenge is that coordination, consistency, and security are harder to enforce without a central authority. Examples: BitTorrent (file sharing), Blockchain (ledger), Skype (early versions).

## 7. Fundamental Models of Distributed System

A **fundamental model** describes the essential properties and behaviors of a distributed system at a theoretical level. Unlike architectural models which describe structure, fundamental models describe _how the system behaves_ in terms of communication, failure, and security.

### 7.1 Interaction Model

In a distributed system, processes on different nodes communicate by passing messages over a network. The **interaction model** describes the nature and constraints of this communication.

Two key issues arise: **performance** and **timing**. Message delivery is not instant — there is always a **communication delay** caused by network latency, congestion, and processing time. Because of this, it is impossible to have a single global clock that all nodes agree on, making **time synchronization** a fundamental challenge.

The interaction model has two variants:

- **Synchronous Model** — there are known bounds on execution time, message delivery time, and clock drift. Every process completes within a finite known time. Easier to reason about but hard to guarantee in practice.
- **Asynchronous Model** — there are no bounds on timing. A message may take any amount of time to arrive, and a process may pause for any duration. This is the realistic model of the internet and most real DS.

### 7.2 Failure Model

In a distributed system, both processes and communication channels can fail. The **failure model** classifies the types of failures that can occur so that the system can be designed to handle them.

**Process Failures:**

- **Crash Failure** — a process stops executing and does not recover. Other processes may not immediately know it has crashed. This is the most common and simplest type of failure.
- **Omission Failure** — a process fails to respond to some requests but continues running otherwise.
- **Byzantine Failure** — the most severe type. A process continues running but sends incorrect, inconsistent, or malicious responses. Very hard to detect and handle.

**Communication Failures:**

- **Omission Failure** — a message is sent but never received, or a reply is never sent back. Caused by network congestion or packet loss.
- **Arbitrary/Byzantine Failure** — corrupted or fabricated messages are delivered. Rare but dangerous.

The system uses techniques like **timeouts**, **acknowledgements**, **replication**, and **redundancy** to detect and recover from failures.

### 7.3 Security Model

The **security model** defines the threats that a distributed system faces and the mechanisms used to protect it. Since DS components communicate over open networks, they are exposed to a wide range of attacks.

**Threats:**

- **Threats to processes** — an enemy process may impersonate a legitimate process (spoofing), tamper with messages, or replay old messages to cause incorrect behavior.
- **Threats to communication channels** — messages can be intercepted (**eavesdropping**), modified (**tampering**), or blocked (**denial of service**).
- **Denial of Service (DoS)** — an attacker floods the system with requests, exhausting resources and preventing legitimate users from being served.

**Security Mechanisms:**

- **Encryption** — messages are encrypted so that intercepted data cannot be read by unauthorized parties. Ensures confidentiality.
- **Authentication** — verifies the identity of a process or user before granting access. Ensures the communicating party is who they claim to be.
- **Authorization** — controls what an authenticated user or process is allowed to do. Ensures access rights are enforced.
- **Secure Channels** — communication channels that provide both encryption and authentication together, such as TLS/SSL.

## 8. Heterogeneity

**Heterogeneity** in DS means the system is composed of components that differ in hardware, operating systems, programming languages, networks, and databases — yet must all work together seamlessly.

For example, a Java client on Windows must be able to communicate with a Python server on Linux over different network types. This is handled using **middleware**, **standard protocols** (HTTP, TCP/IP, JSON), and **virtual machines/containers** that abstract the underlying differences.

## 9. How DS is organized as middleware

**Middleware** is a software layer that sits between the OS and distributed applications, hiding the complexity of the underlying network and hardware and making the distributed system appear as a single coherent system.

```
[ Application ]
[ Middleware  ]   ← the "glue" layer
[ OS / Network ]
```

**How DS is organized as middleware:**

**1. Hides Heterogeneity** — different machines run different OS, hardware, and languages. Middleware abstracts all of this using standard protocols (HTTP, RPC, TCP/IP) so components communicate without caring about each other's platform.

**2. Provides Communication Services** — middleware offers RPC and RMI so processes on different machines can call each other's functions/methods as if they were local. The programmer never writes raw socket code.

**3. Naming and Location Transparency** — middleware maintains a naming service so clients can locate resources by name, not by IP. The actual location of a service can change without the client knowing.

**4. Manages Replication and Consistency** — middleware handles data replication across nodes and ensures consistency, shielding the application from the complexity of keeping multiple copies in sync.

**5. Handles Fault Tolerance** — middleware detects failures, retries requests, and reroutes to healthy nodes automatically. The application sees a reliable service even when underlying nodes crash.

**6. Security and Access Control** — middleware enforces authentication and authorization uniformly across all nodes, so individual applications don't need to implement their own security.

**Examples of middleware:** CORBA, Java RMI, Web Services (SOAP/REST), message queues (RabbitMQ).

## 10. Distributed Computing Systems

A **distributed computing system** combines the processing and storage resources of multiple networked computers to solve computational tasks. Depending on how closely the machines are connected and how resources are provided, it may be organized as a cluster, grid, or cloud.

### 10.1 Cluster Computing

A **cluster** is a group of closely connected computers that work together and appear as one computing system. The machines are normally located in the same organization, connected through a high-speed LAN, and often use similar hardware and operating systems.

Clusters divide a large computation among several nodes to improve performance and can continue working when a failed node is replaced by another node. They are commonly used for scientific computing, parallel processing, web-server farms, and high-availability services.

- In a **symmetric cluster**, all nodes have equal roles and can perform application work.
- In an **asymmetric cluster**, one head node manages or schedules work for the remaining worker nodes.

A cluster scheduler divides a job into smaller tasks, assigns them to available nodes, and combines their results. Nodes may use shared storage and exchange heartbeat messages so that a failed machine can be detected and its work reassigned. The close, high-speed connection provides fast coordination, but a cluster is normally limited to one administrative organization or physical site.

### 10.2 Grid Computing

A **grid** combines computing, storage, and other resources belonging to different machines or organizations so that they can complete a common task. Unlike a cluster, grid nodes may be geographically separated and may use different hardware, operating systems, and network technologies.

Grid middleware discovers available resources, authenticates users, schedules tasks, and combines unused processing capacity into a virtual organization. Its architecture contains:

1. **Fabric layer:** Provides direct access to local processors, storage, networks, and other resources.
2. **Connectivity layer:** Supplies communication and security protocols for accessing remote resources.
3. **Resource layer:** Manages and controls an individual resource.
4. **Collective layer:** Coordinates multiple resources through discovery, scheduling, allocation, and replication services.
5. **Application layer:** Contains applications that use the shared grid resources.

A user submits a large job to the grid middleware rather than selecting each computer manually. The middleware locates suitable resources, divides or schedules the work, transfers the required data, and returns the combined result. Grids are useful for scientific simulations and research projects that need large processing capacity, but heterogeneous machines, changing availability, security, and cross-organization scheduling make coordination difficult.

### 10.3 Cloud Computing

**Cloud computing** provides computing power, storage, databases, platforms, and applications over a network on demand. Users obtain resources when required and normally pay only for what they consume instead of purchasing and maintaining the physical infrastructure.

Cloud providers use virtualization and containers to divide large data-centre resources safely among many customers. Users request, configure, and release these resources through web interfaces or APIs, while the provider manages the physical machines, networking, and basic availability.

Its main characteristics are:

- **On-demand self-service:** Users provision resources without direct interaction with the provider.
- **Broad network access:** Services are available through standard network connections and client devices.
- **Resource pooling:** Shared infrastructure serves several users while keeping their data isolated.
- **Rapid elasticity:** Resources can expand or shrink according to demand.
- **Measured service:** Usage is monitored for control and pay-as-you-go billing.

**Cloud service models**

- **Software as a Service (SaaS):** The provider supplies a complete application that users access through the network, such as Google Workspace.
- **Platform as a Service (PaaS):** The provider supplies tools and a managed platform for developing and deploying applications, such as Google App Engine.
- **Infrastructure as a Service (IaaS):** The provider supplies virtual machines, storage, and networking on which users install their own software, such as Amazon EC2.

**Cloud deployment models**

- **Public cloud:** A provider offers shared cloud resources to customers over the Internet.
- **Private cloud:** One organization operates cloud infrastructure for its own authorized users.
- **Hybrid cloud:** Public and private clouds are connected so workloads and data can use both environments.

**Advantages**

- Resources can be deployed rapidly and scaled according to demand.
- Users can access services and data from different locations and devices.
- Managed backup, recovery, and redundancy can improve service availability.
- Pay-as-you-go charging avoids purchasing maximum hardware capacity in advance.

**Limitations**

- Poor Internet connectivity can prevent access to cloud resources.
- Moving an application between providers may be difficult because of vendor lock-in.
- Customers have less control over the provider's underlying infrastructure.
- Storing sensitive data with a third party introduces privacy and security concerns.
