# 7. Replication

**Replication** is the process of maintaining multiple copies of the same data, object, or service on different machines in a distributed system. Although several physical copies exist, clients should see them as one logical resource.

Each copy is called a **replica**, and the process that manages a replica is called a **replica manager**. When one replica is updated, the replication protocol decides when and how that update reaches the other replicas.

Replication is mainly used to improve **performance**, **availability**, and **fault tolerance**. Its main difficulty is keeping all copies sufficiently consistent.

## 1. Reasons for Replication

### 1.1 Performance Improvement

Replicas can be placed close to the users that access them. A client can then read data from a nearby replica instead of contacting a distant server, which reduces communication delay.

Replication also distributes requests among multiple servers. Independent read requests can be processed in parallel, reducing the load on each server and increasing the total throughput of the system.

**Example:** A content delivery network stores copies of web content in different geographical regions so users receive the content from a nearby server.

### 1.2 Increased Availability

A service is **available** when a client can obtain a response from it. If one replica crashes or becomes unreachable, the client can contact another replica and continue using the service.

If each of `n` replicas fails independently with probability `p`, the probability that at least one replica remains available is:

$$
A = 1-p^n
$$

For example, if two independent servers each have a failure probability of `0.05`, then:

$$
A=1-(0.05)^2=0.9975=99.75\%
$$

This improvement assumes that the replicas do not fail because of the same shared cause, such as a common power supply or network link.

### 1.3 Fault Tolerance

**Fault tolerance** is the ability of a service to continue producing correct results even when some components fail. Replication provides redundant copies that can replace a failed component or outvote an incorrect result.

- A backup can take over when the current server crashes.
- Several replicas can execute the same request so that a failure of one replica does not stop the operation.
- With a suitable voting protocol and enough independent replicas, incorrect or malicious results can be detected and masked.

Availability only means that some response can be obtained. Fault tolerance additionally requires the response to remain correct.

### 1.4 Main Challenges of Replication

| Challenge | Explanation |
|---|---|
| **Replica placement** | The system must decide how many replicas to create and where to place them. Poor placement may give little performance or failure-independence benefit. |
| **Update propagation** | A change accepted by one replica must reach other replicas through push-based or pull-based communication. |
| **Consistency** | During propagation, different replicas may temporarily contain different values. The system must define which differences are allowed. |
| **Coordination cost** | Stronger agreement among replicas requires more messages, waiting, and synchronization. |
| **Failure handling** | The system must detect failed replicas, redirect clients, recover replicas, and avoid processing retried requests twice. |

#### 1.4.1 Replica Placement

| Replica type | Description | Example |
|---|---|---|
| **Permanent replica** | A fixed server that always stores a copy of the data. | Primary database server or mirror site |
| **Server-initiated replica** | A server dynamically creates a copy where demand is high. | CDN edge server |
| **Client-initiated replica** | A client stores a temporary local copy to reduce access delay. | Browser cache |

#### 1.4.2 Update Propagation

- **Push-based propagation:** The replica that receives an update sends it to other replicas. It gives fresher copies but creates communication even when another replica does not currently need the data.
- **Pull-based propagation:** A replica periodically requests newer data from another replica. It reduces unnecessary messages but may return an older value before the next pull.

## 2. Object Replication

**Object replication** creates several copies of a distributed object, including its state and the operations that can be performed on it. A client invokes one logical object while the replication mechanism selects or coordinates the physical copies.

A **front end** receives the client request and hides the locations and number of replicas. The replica managers execute operations and maintain the replicated state. This property is called **replication transparency**.

### 2.1 Application-Controlled Replication

In application-controlled replication, the application creates the copies, sends requests to them, propagates updates, and resolves inconsistencies. It gives the application full control but makes the program more complex and tightly coupled to the replication method.

### 2.2 System-Managed Replication

In system-managed replication, middleware performs replica placement, request routing, update propagation, and failure recovery. The application accesses one logical object and does not manage individual copies.

| Basis | Application-controlled | System-managed |
|---|---|---|
| Responsibility | Application code manages replicas | Middleware manages replicas |
| Flexibility | Easy to apply object-specific rules | Limited to policies offered by middleware |
| Application complexity | Higher | Lower |
| Transparency | Low | High |
| Consistency handling | Implemented by application | Implemented by replication protocol |

## 3. Replication as a Scaling Technique

Replication improves scalability by placing data near the processes that use it and by allowing several replicas to serve requests at the same time.

For read-heavy workloads, adding replicas provides two important benefits:

1. **Lower latency:** A client reads from a nearby copy.
2. **Higher throughput:** Read requests are divided among several servers.

However, every update may need to reach several replicas. As the number of replicas increases, the communication and agreement required for a write also increases. Strictly synchronizing every replica before completing each update can therefore become a scalability bottleneck.

The central trade-off is:

| Stronger consistency | Better scalability and availability |
|---|---|
| Replicas coordinate before exposing an update | Replicas may accept operations independently |
| Clients observe more uniform results | Clients may temporarily observe stale or different results |
| Higher communication delay | Faster local response |
| More difficult during a network partition | Service can continue in more partitions |

A system chooses a consistency model according to how much temporary inconsistency the application can tolerate.

## 4. Consistency Models

A **consistency model** is a contract between a replicated data store and its clients. It specifies the order and values that read and write operations are allowed to observe.

### 4.1 Strict Consistency

Under **strict consistency**, every read must return the value written by the most recent write in absolute global time.

Strict consistency is mostly a theoretical model for distributed systems because there is no perfectly synchronized global clock and messages take time to travel between machines.

### 4.2 Linearizability

Under **linearizability**, every operation appears to occur instantaneously at one point between its invocation and its response. If operation `A` finishes before operation `B` begins in real time, every process must observe `A` before `B`.

Linearizability provides behavior close to a single correct copy, but it requires coordination between replicas and may reduce availability during a network partition.

### 4.3 Sequential Consistency

A system is **sequentially consistent** when the result is the same as if all operations were executed in one sequential order, while the operations issued by each individual process remain in that process's program order.

Unlike linearizability, the common order does not have to respect the real-time order of operations performed by different processes.

### 4.4 Causal Consistency

Two operations are **causally related** when one operation can influence the other. Under causal consistency:

- All processes observe causally related writes in the same order.
- Concurrent writes that cannot influence each other may be observed in different orders by different processes.

**Example:** If a user publishes a post and another user writes a reply after reading it, every process must observe the post before the reply. Two unrelated posts may appear in different orders.

### 4.5 Eventual Consistency

Under **eventual consistency**, replicas may temporarily contain different values, but if no new updates occur, all replicas eventually converge to the same value.

It provides high availability and low coordination cost, making it useful for caches, directory information, and geographically distributed services. Applications may need conflict detection and resolution when different replicas accept concurrent updates.

### 4.6 Comparison of Consistency Models

| Model | Ordering guarantee | Real-time order required? | Main trade-off |
|---|---|---:|---|
| **Strict** | Read returns the absolutely latest write | Yes, using absolute global time | Impractical in a distributed system |
| **Linearizable** | One global order consistent with observed real time | Yes | Strong behavior but expensive coordination |
| **Sequential** | One global order preserving each process's program order | No | Easier than linearizability but still requires agreement |
| **Causal** | Causally related writes have one order; concurrent writes may differ | No | Preserves cause and effect with less coordination |
| **Eventual** | Replicas converge when updates stop | No | High availability with temporary inconsistency |

## 5. Fault-Tolerant Services Through Replication

A replicated service remains fault tolerant only when the replicas are coordinated correctly. Merely storing several copies is not enough.

The replication protocol should provide:

- **Single-copy correctness:** Client-visible behavior should be equivalent to one correct, non-replicated service under the promised linearizable or sequential ordering guarantee.
- **Liveness within the fault bound:** The service should continue accepting and completing requests while the number and type of failures remain within the protocol's stated assumptions.
- **Request identification:** Every request has a unique identifier so a retry is not executed twice.
- **Ordering:** Replicas process conflicting requests in a compatible order.
- **State transfer:** A recovering or newly added replica receives a current copy of the service state.
- **Failure detection:** The system identifies an unavailable replica and changes membership or selects a replacement.
- **Deterministic execution:** Replicas given the same state and request should produce the same new state and result.

Two major methods are **passive replication** and **active replication**.

### 5.1 Passive Replication

**Passive replication**, also called **primary-backup replication**, uses one primary replica and one or more backup replicas. The front end sends operations to the primary. The primary executes each operation and transfers the resulting state or update to the backups.

If the primary fails, the system promotes an up-to-date backup to become the new primary.

#### 5.1.1 Operation of Passive Replication

1. **Request:** The front end gives the client request a unique identifier and sends it to the primary.
2. **Coordination:** The primary checks for duplicate identifiers and places the request in its execution order.
3. **Execution:** The primary performs the operation and computes the new state and result.
4. **Agreement:** For an update operation, the primary sends the updated state, the unique request identifier, and the stored response to the backups, then waits for the acknowledgements required by the protocol. This lets a promoted backup recognize a retried request and return its earlier response without executing the operation twice.
5. **Response:** The primary returns the result to the front end, which forwards it to the client.
6. **Failover:** If the primary fails, a sufficiently up-to-date backup is selected as the new primary, and an uncompleted client request can be retried using the same identifier.

#### 5.1.2 Advantages of Passive Replication

- **Lower execution cost:** Only the primary executes each client operation; backups apply state updates.
- **Simple client interaction:** The front end normally communicates with one replica.
- **Suitable for non-deterministic services:** The primary can make a non-deterministic choice and send the resulting state to backups.
- **Efficient response handling:** The front end receives only one normal response.

#### 5.1.3 Disadvantages of Passive Replication

- **Failover delay:** Service may pause while a new primary is selected and clients are redirected.
- **Primary bottleneck:** All requests normally pass through one primary.
- **State-transfer overhead:** The primary must keep the backups sufficiently up to date.
- **Limited fault masking:** Basic primary-backup replication handles crash failures but cannot by itself mask an incorrect or malicious primary.

### 5.2 Active Replication

In **active replication**, all replica managers have equal roles. The front end sends every request to the complete replica group using reliable, totally ordered multicast. Every correct replica receives requests in the same order and executes them independently.

Active replication is also called the **state-machine approach** because each replica behaves as the same deterministic state machine.

> [!NOTE]
> Insert a diagram of **active replication** here. It should show the client, front end, totally ordered multicast, several equal replica managers executing the same request, and their responses returning to the front end.

#### 5.2.1 Operation of Active Replication

1. **Request:** The front end attaches a unique identifier and multicasts the request to all replica managers.
2. **Coordination:** The group communication system delivers requests reliably and in the same total order at every correct replica.
3. **Execution:** Every replica executes the request. Deterministic replicas starting in the same state therefore produce the same new state and result.
4. **Agreement:** No separate state-update phase is required because the ordered multicast and identical execution keep replicas coordinated.
5. **Response:** Replicas send results to the front end. For crash faults, the first correct response may be used; to mask arbitrary faults, the front end requires an appropriate matching majority.

#### 5.2.2 Advantages of Active Replication

- **Immediate crash masking:** Other replicas continue processing when one replica crashes, so no primary promotion is required.
- **No single primary bottleneck:** All replicas have equal roles.
- **Transparent recovery:** A client can continue receiving responses while a faulty replica is removed or repaired.
- **Arbitrary-fault masking:** With voting, independent failure assumptions, and enough replicas, inconsistent results can be detected and masked.

#### 5.2.3 Disadvantages of Active Replication

- **High resource use:** Every replica executes every request.
- **Communication cost:** Requests require reliable total-order multicast.
- **Determinism requirement:** Non-deterministic actions such as local time or random choices must be controlled so replicas do not diverge.
- **Complex group management:** Membership changes and recovering replicas must be coordinated safely.

### 5.3 Active vs Passive Replication

| Basis | Active replication | Passive replication |
|---|---|---|
| Replica roles | All replicas are equal and active | One primary; remaining replicas are backups |
| Request delivery | Request is multicast to all replicas | Request is sent to the primary |
| Execution | Every replica executes every request | Only the primary executes the request |
| State synchronization | Same ordered requests produce the same state | Primary sends state changes to backups |
| Main communication requirement | Reliable total-order multicast | Reliable primary-to-backup update transfer |
| Crash recovery | Remaining replicas continue immediately | A backup must be promoted |
| Failover delay | Very low | Higher because of primary election/promotion |
| Resource consumption | High | Lower execution cost |
| Determinism | Required | Not strictly required for the operation itself |
| Basic fault coverage | Crash faults; arbitrary faults with voting and sufficient replicas | Primarily crash faults |
| Main weakness | Expensive execution and ordering | Primary bottleneck and failover delay |

## 6. High-Availability Services

A **high-availability service** is designed to remain accessible for as much time as possible, including during server crashes, temporary disconnections, and network partitions.

Strongly fault-tolerant replication commonly propagates updates eagerly and coordinates replicas before replying. A high-availability system may instead allow an available replica to accept a request immediately and exchange the update later. This reduces response time and lets more network partitions continue operating, but replicas may temporarily disagree.

| Eager replication | Lazy replication |
|---|---|
| Updates are sent to replicas before an operation completes | Updates are propagated later in the background |
| Provides stronger consistency | Provides weaker or eventual consistency |
| Requires more coordination | Requires less immediate coordination |
| May block during a partition | Can remain available in more partitions |
| Common in tightly coordinated fault-tolerant services | Common in gossip-based and geographically distributed services |

Examples of high-availability designs include Gossip, Bayou, and Coda. They allow useful work to continue during disconnection and reconcile updates when communication is restored.

## 7. Gossip Architecture

The **gossip architecture** is a lazy replication method in which replica managers periodically exchange information about the updates they have received. It is called gossip because each replica gradually spreads its knowledge to other replicas, much like information spreading through a group of people.

A client front end may send a request to any reachable replica manager. The system supports two basic operations:

- **Query:** Reads the current state without changing it.
- **Update:** Changes the replicated state and is later propagated to other replicas.

### 7.1 Processing Queries and Updates

1. **Request:** The front end selects a reachable replica and sends a query or update. It may choose another replica if the normal one is unavailable or overloaded.
2. **Early update response:** For an update, the receiving replica may acknowledge acceptance before every other replica receives it, providing high availability.
3. **Coordination:** Before applying an operation, the replica checks whether its causal dependencies have already been applied. Missing dependencies can be obtained through gossip.
4. **Execution:** The replica executes the query or update when the required ordering condition is satisfied.
5. **Query response:** For a query, the replica returns its result after execution.
6. **Agreement by gossip:** Replica managers periodically exchange batches of known updates. Each receiver applies the missing updates in an allowed order until the replicas converge.

### 7.2 Ordering and Consistency

Gossip systems commonly attach logical or vector timestamps to updates. These timestamps help a replica determine:

- whether it has already received an update;
- which earlier updates an operation depends on;
- whether two updates are causally related or concurrent; and
- which updates must be requested from another replica.

Gossip naturally supports eventual consistency. It can also preserve causal ordering by delaying an update until all of its causal predecessors have been applied.

### 7.3 Advantages of Gossip Architecture

- **High availability:** Any reachable replica can accept a request.
- **Partition tolerance:** Separate network partitions can continue serving local clients.
- **Scalability:** Replicas exchange updates without requiring one central coordinator.
- **Load distribution:** Queries and updates can be spread among several replicas.
- **Batch propagation:** Several updates can be sent in one gossip exchange, reducing immediate coordination.

### 7.4 Limitations of Gossip Architecture

- **Temporary inconsistency:** Different clients may observe different values before updates spread.
- **Delayed convergence:** An update may take several gossip rounds to reach every replica.
- **Conflict resolution:** Concurrent updates may require application-specific or automatic conflict handling.
- **Metadata overhead:** Replicas must keep update histories, timestamps, or version vectors.
- **Unsuitable for strict operations:** Applications requiring an immediately current global value may need stronger coordination.
