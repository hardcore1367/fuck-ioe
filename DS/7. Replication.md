# 7. Replication

**Replication** maintains multiple copies of the same data, object, or service on different machines. Each copy is a **replica**, and the process managing it is a **replica manager**. Clients should see the replicas as one logical resource.

## 1. Reasons for Replication

### 1.1 Main Reasons

1. **Better performance:** A client can read from a nearby replica, reducing communication delay.
2. **Higher throughput:** Several replicas can process independent requests at the same time, distributing the server load.
3. **Increased availability:** If one replica crashes or becomes unreachable, another replica can continue serving clients.
4. **Fault tolerance:** Redundant replicas allow the service to continue producing correct results within its assumed failure limit.
5. **Geographical access:** Copies placed in different regions reduce latency for widely distributed users.

Replication introduces an important difficulty: an update accepted by one replica must reach the others. Strong consistency requires more coordination, while weak consistency permits faster responses but may temporarily expose different values.

### 1.2 Replication as a Scaling Technique

Replication improves scalability mainly for **read-heavy workloads**:

- **Lower latency:** A client reads from a nearby copy.
- **Load distribution:** Requests are divided among several servers.
- **Parallel service:** Several replicas answer different clients simultaneously.

However, every write may need to update several replicas. Increasing the number of replicas can therefore increase communication, ordering, and agreement costs. A scalable design balances fast local access against the required consistency level.

## 2. Object Replication

**Object replication** creates multiple copies of a distributed object's state and operations. A client invokes one logical object through a **front end**, while replica managers execute operations and maintain the copies. Hiding the number and locations of these copies is called **replication transparency**.

The two main object-replication techniques are **active replication** and **passive replication**.

### 2.1 Active Replication

In **active replication**, all replicas have equal roles. The front end sends every request to the complete replica group using reliable, totally ordered multicast. Every correct replica executes requests in the same order.

It is also called the **state-machine approach** because deterministic replicas starting from the same state produce the same new state and result.

#### 2.1.1 Implementation Steps

1. The front end attaches a unique ID to the client request.
2. It reliably multicasts the request to every replica manager.
3. Total-order multicast delivers requests in the same order at all correct replicas.
4. Every replica executes the request deterministically.
5. The replicas send their results to the front end.
6. For crash faults, the front end may accept the first correct response. To mask arbitrary faults, it requires a suitable matching majority.

#### 2.1.2 Advantages

- **Immediate crash masking:** Other replicas continue processing if one replica crashes.
- **Very low failover delay:** No backup promotion is required.
- **No primary bottleneck:** All replicas have equal roles.
- **Fault masking:** With enough independent replicas and voting, inconsistent results can be detected and masked.

#### 2.1.3 Disadvantages

- **High resource cost:** Every replica executes every request.
- **Ordering overhead:** Requests require reliable total-order multicast.
- **Determinism requirement:** Random values, local time, and other non-deterministic operations must be controlled.
- **Complex membership:** Adding, removing, and recovering replicas must be coordinated safely.

### 2.2 Passive Replication

**Passive replication**, also called **primary-backup replication**, uses one primary and one or more backups. The primary executes each operation and sends the resulting state change to the backups.

If the primary fails, an up-to-date backup is promoted as the new primary.

> [!NOTE]
> Insert a diagram of **passive or primary-backup replication architecture** here. It should show the client, front end, one primary replica, multiple backups, requests and responses through the primary, state updates sent to backups, and backup promotion after primary failure.

#### 2.2.1 Implementation Steps

1. The front end gives the request a unique ID and sends it to the primary.
2. The primary checks whether the request is a retry and places it in execution order.
3. The primary executes the operation and computes the result and new state.
4. It sends the state change and request ID to the backups.
5. After receiving the acknowledgements required by the protocol, the primary returns the result.
6. If the primary fails, the system promotes a sufficiently up-to-date backup and retries any unfinished request using the same ID.

Unique request IDs prevent the new primary from executing the same retried operation twice.

#### 2.2.2 Advantages

- **Lower execution cost:** Only the primary executes each client operation.
- **Simple interaction:** The front end normally communicates with one replica.
- **Supports non-determinism:** The primary can make a choice and send its resulting state to the backups.
- **Lower response duplication:** Only the primary normally sends a result.

#### 2.2.3 Disadvantages

- **Failover delay:** Service may pause while a backup is selected and promoted.
- **Primary bottleneck:** All normal requests pass through one server.
- **State-transfer cost:** Backups must be kept sufficiently up to date.
- **Limited fault masking:** Basic primary-backup replication handles crash failure but cannot mask a malicious or incorrect primary by itself.

### 2.3 Active and Passive Replication Comparison

| S.N. | Active replication | Passive replication |
|---:|---|---|
| 1 | All replicas have equal and active roles. | One replica is the primary, and the others are backups. |
| 2 | The front end multicasts every request to all replicas. | The front end sends each request to the primary. |
| 3 | Every replica executes every request. | Only the primary executes the client operation. |
| 4 | Total-order delivery and deterministic execution keep replica states equal. | The primary sends state changes to the backups. |
| 5 | The system requires reliable total-order multicast. | The system requires reliable primary-to-backup update transfer. |
| 6 | Remaining replicas continue immediately after one replica crashes. | A backup must be selected and promoted after primary failure. |
| 7 | Failover delay is very low because no promotion is required. | Failover delay is higher because primary replacement is required. |
| 8 | Resource consumption is high because all replicas execute every operation. | Execution cost is lower because backups mainly apply state updates. |
| 9 | Replica operations must be deterministic. | The primary may execute non-deterministic operations and copy their results. |
| 10 | Its main weaknesses are repeated execution and ordering cost. | Its main weaknesses are the primary bottleneck and promotion delay. |

## 3. Fault-Tolerant Services Through Replication

A **fault-tolerant service** continues producing correct results despite a permitted number and type of failures. Multiple copies alone are insufficient; the replication protocol must coordinate them correctly.

Important requirements are:

- **Single-copy behavior:** Clients should observe behavior equivalent to one correct service under the promised consistency model.
- **Request identification:** A unique request ID prevents a retry from being executed twice.
- **Compatible ordering:** Replicas process conflicting operations in a safe order.
- **State transfer:** A recovering or new replica receives a current state before serving clients.
- **Failure recovery:** The system redirects requests, changes membership, or selects a replacement after failure.
- **Deterministic execution for active replication:** Equal states and requests must produce equal results.

Passive replication tolerates primary crash by promoting a backup. Active replication masks a replica crash immediately because the other replicas continue executing the ordered requests.

## 4. High-Availability Services

A **high-availability service** remains accessible for as much time as possible, even during server crashes, disconnections, or network partitions.

High availability may conflict with strong consistency. Waiting for every required replica provides fresher and more uniform data but may block during failure. Allowing an available replica to respond locally keeps the service running but may temporarily return stale data.

### 4.1 Eager and Lazy Replication

| S.N. | Eager replication | Lazy replication |
|---:|---|---|
| 1 | Updates reach the required replicas before the operation completes. | Updates propagate to other replicas later in the background. |
| 2 | Clients observe stronger consistency. | Clients may temporarily observe different replica states. |
| 3 | Each update requires more immediate coordination. | Each update requires less immediate coordination. |
| 4 | The service may block when required replicas are unreachable. | An available replica can continue serving more network partitions. |
| 5 | It is suitable for tightly coordinated fault-tolerant services. | It is suitable for gossip-based and geographically distributed services. |

## 5. Consistency Models

A **consistency model** defines which values and operation orders clients are allowed to observe from replicated data.

### 5.1 Strict Consistency

Every read returns the value of the most recent write in absolute global time. This model is mostly theoretical because distributed systems have no perfectly synchronized global clock.

### 5.2 Linearizability

Every operation appears to occur at one instant between its invocation and response. If one operation completes before another starts, all processes observe them in that real-time order.

### 5.3 Sequential Consistency

All operations appear in one common sequential order, and each process's operations remain in its program order. The common order does not have to preserve real-time order between different processes.

### 5.4 Causal Consistency

All processes observe causally related writes in the same order. Concurrent writes that cannot influence each other may be observed in different orders.

For example, every process must observe a post before a reply written after reading that post.

### 5.5 Eventual Consistency

Replicas may temporarily differ, but if no new updates occur, they eventually converge to the same value. It provides high availability and low coordination cost but may require conflict resolution.

## 6. Gossip Architecture

**Gossip architecture** is a lazy replication technique in which replica managers periodically exchange the updates they know. Knowledge spreads gradually until all reachable replicas converge.

A client can send a **query** or **update** to any reachable replica, which makes the service highly available.

### 6.1 Working

1. The front end selects a reachable replica and sends a query or update.
2. The receiving replica may acknowledge an update before every other replica receives it.
3. The replica records the update with logical or vector timestamp information.
4. Replica managers periodically exchange batches of known updates.
5. A receiver requests and applies its missing updates in an allowed causal order.
6. If communication continues and no new updates occur, all replicas converge.

### 6.2 Advantages and Limitations

**Advantages**

- Any reachable replica can serve a client, providing high availability.
- Separate network partitions can continue serving local users.
- Updates spread without a central coordinator.
- Batching several updates reduces immediate communication cost.

**Limitations**

- Different clients may temporarily observe different values.
- An update may require several gossip rounds to reach every replica.
- Concurrent updates may require conflict resolution.
- Timestamps and update histories create metadata overhead.
