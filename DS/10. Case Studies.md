# 10. Case Studies

## 1. MACH

**MACH** is a microkernel operating system that keeps only essential mechanisms in the kernel and runs operating-system personalities, such as UNIX emulation, in user space. Its kernel provides five main abstractions: **processes, threads, memory objects, ports, and messages**. A process supplies an address space and resources, while one or more threads provide the units of execution. Protected ports and message passing form the basis of inter-process communication, including communication between processes on different machines. MACH also supports external memory managers and distributed shared memory, making it portable and suitable as a base for heterogeneous distributed operating systems.

## 2. JINI

**JINI** is a Java-based distributed-system technology that lets devices and software services form a dynamic federation. A service discovers a **lookup service**, joins it, and registers a Java proxy object describing how clients can use the service. A client discovers the lookup service, searches for the required service, downloads its proxy, and communicates through the proxy's Java interface. **Leasing** gives each registration a limited lifetime, so failed or disconnected services disappear unless they renew their leases. JINI also provides distributed events, transactions, JavaSpaces, and security mechanisms for coordinating changing groups of services.

## 3. CORBA

**CORBA** is OMG middleware that enables objects written in different languages and running on different platforms to communicate. Interfaces are defined in **IDL**, and an IDL compiler generates client stubs and server skeletons that marshal and unmarshal parameters. The **Object Request Broker (ORB)** locates the target object, sends the request, and returns the result while hiding network and location details. Static invocation uses generated interfaces known at compile time, whereas dynamic invocation discovers interface information and constructs requests at runtime. Object adapters connect servants to the ORB, while naming, event, security, transaction, interface-repository, and implementation-repository services support complete distributed applications.

## 4. TIB/Rendezvous

**TIB/Rendezvous** is message-oriented middleware for exchanging data among distributed applications across a network. Each application uses the Rendezvous API and a local daemon, which hides transport details and connects the process to other Rendezvous daemons. Its subject-based publish–subscribe model lets senders publish to a subject without knowing the receivers, while interested applications subscribe to that subject. The daemon filters subject addresses, routes messages, handles acknowledgements and retransmission when required, and dispatches incoming data to the correct application. Support for several programming-language interfaces and hardware platforms makes it suitable for loosely coupled, heterogeneous distributed applications.
