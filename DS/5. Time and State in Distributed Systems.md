# 5. Time and State in Distributed Systems

## 1. Physical and Logical Time

### 1.1 Physical Clocks

A **physical clock** estimates real or wall-clock time by counting ticks from a hardware oscillator. Different machines cannot maintain exactly the same time because their oscillators run at slightly different rates and network delays vary.

- **Clock skew** is the difference between two clock readings at the same real instant.
- **Clock drift** is the difference between the rates at which clocks run.

Therefore, distributed clocks must be synchronized periodically. **External synchronization** keeps clocks close to an external source such as UTC, whereas **internal synchronization** keeps the clocks close to one another. A system normally corrects an advanced clock by slowing it temporarily so that time does not move backward; this is called **clock slewing**.

### 1.2 Physical Clock vs Logical Clock

| S.N. | Physical clock | Logical clock |
|---:|---|---|
| 1 | A physical clock estimates real or wall-clock time. | A logical clock represents the order and causal relationship of events. |
| 2 | It is maintained using a hardware oscillator and an operating-system clock. | It is maintained using software counters or timestamp vectors. |
| 3 | It may be synchronized with UTC or another physical clock. | It is updated from local events and exchanged messages without using UTC. |
| 4 | Its accuracy is affected by drift, skew, and uncertain network delay. | It does not measure elapsed real time and may represent only a partial order. |
| 5 | Its timestamp estimates when an event occurred. | Its timestamp represents an event's logical position or causal history. |
| 6 | Cristian's algorithm, Berkeley's algorithm, and NTP synchronize physical clocks. | Lamport clocks and vector clocks are examples of logical clocks. |

## 2. Physical Clock Synchronization Algorithms

### 2.1 Cristian's Algorithm

**Cristian's algorithm** externally synchronizes a client with a time server connected to an accurate UTC source. It assumes that request and reply delays are approximately equal.

1. The client sends a request at local time $T_0$.
2. The server returns its timestamp $S$ as close as possible to sending the reply.
3. The client receives the reply at $T_1$ and calculates:

   $$
   RTT=T_1-T_0
   $$

4. It estimates the one-way delay as $RTT/2$ and sets or slews its clock toward:

   $$
   C_{new}=S+\frac{RTT}{2}
   $$

The client normally repeats the request and selects the sample with the smallest round-trip time. The estimate becomes inaccurate when request and reply delays are unequal, and one time server can become a bottleneck or single point of failure.

### 2.2 Berkeley Algorithm

The **Berkeley algorithm** provides internal synchronization when no machine needs direct UTC time. An elected **master or time daemon** brings all clocks close to the group average.

1. The master polls every slave for its current clock value.
2. It estimates each current value while allowing for network delay.
3. It rejects faulty or extreme readings.
4. It calculates the average offset of the valid clocks, including its own clock.
5. It sends each machine the positive or negative adjustment it must make.
6. Each machine slews its clock, and the process is repeated periodically.

**Example:** If the master and two slaves have offsets $0$, $-10$, and $+25$ minutes, the average offset is:

$$
\frac{0-10+25}{3}=+5\text{ minutes}
$$

The new group time is five minutes ahead of the master's old time. Berkeley synchronization does not require UTC, but the result can differ from real time, and master failure requires a new election.

### 2.3 Network Time Protocol

The **Network Time Protocol (NTP)** synchronizes clocks with UTC across large networks. It uses a hierarchy of redundant servers called **strata**.

| Level | Meaning |
|---|---|
| **Stratum 0** | It contains accurate reference devices such as atomic, radio, or GPS clocks. |
| **Stratum 1** | It contains primary servers connected directly to stratum-0 devices. |
| **Stratum 2 and above** | It contains secondary servers synchronized through lower-stratum servers. |

A client normally consults several servers, filters inaccurate observations, and prefers samples with smaller delay. For an NTP exchange:

- $T_1$ is the time at which the client sends the request.
- $T_2$ is the time at which the server receives the request.
- $T_3$ is the time at which the server sends the reply.
- $T_4$ is the time at which the client receives the reply.

The server's estimated offset relative to the client is:

$$
\theta=\frac{(T_2-T_1)+(T_3-T_4)}{2}
$$

The estimated network delay is:

$$
\delta=(T_4-T_1)-(T_3-T_2)
$$

NTP is scalable and fault tolerant because it uses several servers, but asymmetric routes and variable network delay limit its accuracy.

## 3. Logical Clocks and Event Ordering

### 3.1 Happened-Before Relation

A **logical clock** orders events according to causality instead of real time. Lamport's **happened-before relation**, written $a\rightarrow b$, is defined by three rules:

1. **Process order:** If $a$ and $b$ occur in the same process and $a$ occurs first, then $a\rightarrow b$.
2. **Message order:** If $a$ sends a message that $b$ receives, then $a\rightarrow b$.
3. **Transitivity:** If $a\rightarrow b$ and $b\rightarrow c$, then $a\rightarrow c$.

If neither $a\rightarrow b$ nor $b\rightarrow a$, the events are **concurrent**, written $a\parallel b$.

### 3.2 Lamport's Logical Clock

Lamport's logical clock assigns one increasing integer timestamp to every event. Each process $P_i$ maintains a scalar counter $L_i$.

**Implementation rules:**

1. Initially, every process sets $L_i=0$.
2. Before every internal or send event, process $P_i$ increments its clock:

   $$
   L_i=L_i+1
   $$

3. A sent message carries the current timestamp $t=L_i$.
4. Before timestamping the receipt of a message carrying $t$, process $P_j$ updates:

   $$
   L_j=\max(L_j,t)+1
   $$

These rules satisfy the **clock condition**:

$$
a\rightarrow b \implies L(a)<L(b)
$$

For a total order, events can be ordered lexicographically using $(L(e),\text{process ID})$. The process ID breaks ties between equal timestamps.

**Example:**

| Step | Event | Timestamp calculation |
|---:|---|---|
| 1 | $P_1$ performs event $a$. | $L(a)=1$ |
| 2 | $P_1$ sends a message at event $b$. | $L(b)=2$ |
| 3 | $P_2$ independently performs event $c$. | $L(c)=1$ |
| 4 | $P_2$ receives the message carrying timestamp 2 at event $d$. | $L(d)=\max(1,2)+1=3$ |

Events $b$ and $c$ are concurrent even though $L(c)<L(b)$. This demonstrates that timestamp order alone does not prove causality.

**Limitations:**

- $L(a)<L(b)$ does not imply $a\rightarrow b$; the converse of the clock condition is false.
- Scalar timestamps cannot determine whether two events are concurrent.
- Lamport clocks do not measure real time or elapsed duration.
- A total order arbitrarily orders concurrent events and loses concurrency information.

### 3.3 Vector Clocks

A **vector clock** extends Lamport's clock so that both causality and concurrency can be identified. In a system of $N$ processes, process $P_i$ maintains vector $V_i[1\ldots N]$, where $V_i[k]$ records the number of events of $P_k$ known to $P_i$.

**Implementation rules:**

1. Initially, every component is zero.
2. Before an internal or send event, $P_i$ increments its own component:

   $$
   V_i[i]=V_i[i]+1
   $$

3. A sent message carries a copy of $V_i$.
4. When $P_i$ receives vector $T$, it first merges every component:

   $$
   V_i[k]=\max(V_i[k],T[k])
   $$

5. It then increments its own component to timestamp the receive event.

For vectors $X$ and $Y$, $X<Y$ when every component of $X$ is less than or equal to the corresponding component of $Y$, and at least one component is smaller. Therefore:

- $a\rightarrow b$ if and only if $V(a)<V(b)$.
- $a\parallel b$ when neither vector is smaller; the vectors are incomparable.

**Example:** If $P_2$ has vector $[0,1]$ and receives a message carrying $[2,0]$, it first merges them to obtain $[2,1]$ and then increments its own component. The receive event therefore gets timestamp $[2,2]$.

### 3.4 Lamport Clock vs Vector Clock

| S.N. | Lamport clock | Vector clock |
|---:|---|---|
| 1 | It represents each timestamp using one scalar integer. | It represents each timestamp using an array of $N$ integers. |
| 2 | It guarantees that causality implies timestamp order. | It guarantees that timestamp order is equivalent to causality. |
| 3 | It cannot determine concurrency from timestamps alone. | It detects concurrency using incomparable vectors. |
| 4 | It naturally produces a scalar order and can form a total order using a process-ID tie-breaker. | It naturally produces a causal partial order. |
| 5 | It requires $O(1)$ timestamp space in each message. | It requires $O(N)$ timestamp space in each message. |
| 6 | It is suitable for simple event ordering and mutual-exclusion algorithms. | It is suitable for causal delivery, version tracking, and conflict detection. |

The vector clock solves Lamport clock's inability to infer causality, but its storage and communication overhead grow with the number of processes.

## 4. Causal Ordering of Messages

### 4.1 Causal Message Order

Messages have **causal order** when every process delivers causally related messages in an order consistent with happened-before. If:

$$
send(m_1)\rightarrow send(m_2)
$$

then every process that delivers both messages must deliver $m_1$ before $m_2$. Concurrent messages may be delivered in either order.

**Example:** $P_1$ sends update $m_1$. After receiving $m_1$, process $P_2$ sends dependent update $m_2$. If $P_3$ receives $m_2$ first, it buffers $m_2$ until $m_1$ arrives.

### 4.2 Vector-Clock Delivery Rule

For causal multicast, $V_j[k]$ can count how many messages from $P_k$ process $P_j$ has delivered. A message from $P_i$ carrying vector $T$ is delivered by $P_j$ only when:

$$
T[i]=V_j[i]+1
$$

and

$$
T[k]\leq V_j[k] \quad \text{for every } k\neq i
$$

The first condition ensures that the next message from the sender is received in order. The second ensures that all causal dependencies from other senders have already been delivered. A message that fails either condition is buffered and checked again after later deliveries.

## 5. Global State and State Recording

### 5.1 Global State

The **global state** of a distributed system consists of the local state of every process and the state of every communication channel, represented by messages in transit. It is used for deadlock or termination detection, checkpointing, recovery, debugging, and evaluating global conditions.

A **cut** is a boundary across all process timelines. Events before the boundary are included in the recorded state, while later events are excluded.

### 5.2 Cut Types

| S.N. | Consistent cut | Inconsistent cut | Strongly consistent cut |
|---:|---|---|---|
| 1 | It contains every cause of every included event. | It includes an event but excludes one of its causes. | It is a consistent cut with no messages in transit. |
| 2 | It may include a send while excluding its later receive, so the message is recorded in the channel state. | It includes a receive while excluding the corresponding send, which records an effect without its cause. | It includes both the send and receive of every recorded message. |
| 3 | A message may cross the cut from the recorded past toward the unrecorded future. | A message crosses the cut from an unrecorded send toward an included receive. | No application-message arrow crosses the cut in either direction. |

To classify a cut, inspect each crossing message arrow. A receive-before/send-after crossing makes the cut inconsistent; a send-before/receive-after crossing represents an in-transit message in a consistent cut. If no message crosses a consistent cut, it is strongly consistent.

### 5.3 Distributed Snapshot

A **distributed snapshot** records a consistent global state while the system continues to run. It contains every process's local state and all messages that were in transit at the represented cut.

### 5.4 Chandy-Lamport Snapshot Algorithm

The **Chandy-Lamport algorithm** records a distributed snapshot using special **marker messages**. It assumes that processes do not fail during recording and that communication channels are reliable, unidirectional, and FIFO ordered.

1. **Initiation:** The initiator records its local state and sends a marker on every outgoing channel before sending any later application message.
2. **First marker at a process:** The receiving process records its local state, records the marker's incoming channel as empty, sends markers on all outgoing channels, and begins recording messages on its other incoming channels.
3. **Later marker at a process:** The process stops recording that incoming channel. Messages received after its local-state recording but before the marker form that channel's recorded state.
4. **Completion:** A process finishes after receiving a marker on every incoming channel. All recorded process and channel states together form the global snapshot.

FIFO ordering ensures that a marker separates messages sent before a process recorded its state from messages sent afterward. Thus, a sent-but-not-yet-received message is recorded as an in-transit channel message, and the resulting snapshot is consistent.

## 6. Distributed Debugging

### 6.1 Meaning and Difficulties

**Distributed debugging** finds faults by reconstructing the states and causal relationships of events across several communicating processes.

It is difficult because:

- different schedules and network delays make executions nondeterministic;
- clock skew can produce a misleading physical timestamp order;
- concurrent events have no single correct order;
- collecting local states at different times can create an inconsistent global view; and
- logging or breakpoints can change timing and hide a race condition.

### 6.2 Debugging with Logical Time and Snapshots

1. Each process logs important events, message identifiers, and Lamport or vector timestamps.
2. The debugger reconstructs happened-before instead of trusting raw physical timestamps.
3. Vector timestamps identify causal and concurrent events.
4. Consistent cuts or Chandy-Lamport snapshots provide valid global states for examining deadlock, termination, or inconsistent replicas.
5. The debugger traces the causal chain leading to a faulty global state.

A **distributed breakpoint** is a condition over the states of several processes and must be evaluated on a consistent cut. A **stable predicate**, such as permanent deadlock, remains true once reached; a **transient predicate** may become true and then false and can require examining several consistent cuts.
