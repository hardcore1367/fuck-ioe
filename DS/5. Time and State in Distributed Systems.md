# 5. Time and State in Distributed Systems

## 1. Physical and Logical Time

**Physical Time** — the actual wall-clock time measured by hardware oscillators on each machine. It represents real elapsed time (seconds, milliseconds) and is anchored to an external standard like UTC. The problem in a distributed system is that no two machines share one physical clock, and each machine's oscillator drifts at a slightly different rate, so their physical clocks naturally diverge over time.

**Logical Time** — a software-maintained counter that does not measure real time at all. Instead it captures the *order* and *causal relationship* between events across processes. If event A caused event B, logical time guarantees A gets a smaller timestamp than B — regardless of what the wall clock said on either machine. It solves the ordering problem without needing synchronized hardware clocks.

The distinction matters because physical clocks are needed when you care about *when* something happened, and logical clocks are needed when you care about *what happened before what*.

### 1.1 Importance of Time and Clock Synchronization

Time and clock synchronization are important in a distributed system because processes run on different machines and do not share one physical clock. Synchronized time is required for:

1. **Event ordering and auditing:** Reliable timestamps help determine when events and transactions occurred across different machines.
2. **Data consistency:** Replicas, caches, and distributed updates may use timestamps to select the latest value and apply operations in the correct order.
3. **Security:** Authentication protocols use time to check request freshness and reject expired or replayed requests.
4. **Duplicate detection and coordination:** Timestamps help identify duplicate updates and coordinate scheduled activities on several nodes.

### 1.2 Physical Clocks

A **physical clock** estimates real or wall-clock time by counting ticks from a hardware oscillator. Different machines cannot maintain exactly the same time because their oscillators run at slightly different rates and network delays vary.

- **Clock skew** is the difference between two clock readings at the same real instant.
- **Clock drift** is the difference between the rates at which clocks run.

Therefore, distributed clocks must be synchronized periodically. **External synchronization** keeps clocks close to an external source such as UTC, whereas **internal synchronization** keeps the clocks close to one another. A system normally corrects an advanced clock by slowing it temporarily so that time does not move backward; this is called **clock slewing**.

### 1.3 Physical Clock vs Logical Clock

| S.N. | Physical clock | Logical clock |
|---:|---|---|
| 1 | A physical clock estimates real or wall-clock time. | A logical clock represents the order and causal relationship of events. |
| 2 | It is maintained using a hardware oscillator and an operating-system clock. | It is maintained using software counters or timestamp vectors. |
| 3 | It may be synchronized with UTC or another physical clock. | It is updated from local events and exchanged messages without using UTC. |
| 4 | Its accuracy is affected by drift, skew, and uncertain network delay. | It does not measure elapsed real time and may represent only a partial order. |
| 5 | Its timestamp estimates when an event occurred. | Its timestamp represents an event's logical position or causal history. |
| 6 | Cristian's algorithm, Berkeley's algorithm, and NTP synchronize physical clocks. | Lamport clocks and vector clocks are examples of logical clocks. |

## 2. Physical Clock Synchronization Algorithms

### 2.1 Cristian's Algorithm

**Cristian's algorithm** externally synchronizes a client with a time server connected to an accurate UTC source. The core idea is that if the round-trip time (RTT) between client and server is known, the one-way delay can be estimated as RTT/2, and the client can correct its clock to the server's time plus that estimated delay.

**Steps:**

1. The client records its local time $T_0$ and sends a time request to the server.
2. The server reads its clock as close to sending the reply as possible and returns timestamp $S$.
3. The client records receipt time $T_1$ and computes:

![Cristian's Algorithm](https://github.com/user-attachments/assets/3b1f1eca-f30e-440f-b7c3-ba050040f227)


$$
\text{RTT} = T_1 - T_0
$$

4. It estimates one-way delay as $\text{RTT}/2$ and sets its clock to:

$$
C_{\text{new}} = S + \frac{\text{RTT}}{2}
$$

The worst-case synchronization error is:

$$
\epsilon \leq \frac{\text{RTT}}{2}
$$

**To improve accuracy**, the client sends multiple requests and uses the sample with the smallest RTT — the smallest RTT means the network was least congested, making the symmetric-delay assumption most valid. If request and reply delays are unequal (asymmetric network), the estimate drifts proportionally to that asymmetry.

**Limitations:**
- Accuracy degrades on asymmetric or high-latency networks.
- A single time server is a bottleneck and single point of failure.


### 2.2 Berkeley Algorithm

The **Berkeley algorithm** provides internal synchronization — the goal is not to match UTC but to bring all machines in the group to the same common time. It works by computing the average clock value across all participating machines and sending each machine the correction it needs. An elected **master (time daemon)** drives the process.

**Steps:**

1. The master polls every slave by sending a request for its current clock value.
2. The master estimates each slave's actual current time, accounting for network delay (similar to Cristian's RTT/2 approach).
3. The master discards readings that are too far from the majority — these are likely faulty or heavily delayed.
4. The master computes the average offset of all valid clocks including its own.
5. It sends each machine its individual positive or negative adjustment to apply.
6. Each machine slews its clock by the given amount. The process repeats periodically.

The average is used rather than any single machine's time because it minimises the influence of any one outlier and distributes the error evenly across the group.

![Berkeley Algorithm](https://github.com/user-attachments/assets/e1c5ea97-806d-4bb1-a784-e4f41be409b4)


**Example:** Master offset = 0, slave 1 = −10 min, slave 2 = +25 min:

$$
\text{Average offset} = \frac{0 + (-10) + 25}{3} = +5 \text{ min}
$$

The group time moves to master's time + 5 minutes. Each machine receives its individual delta to reach that new group time.

**Limitations:**
- The result can differ significantly from UTC since there is no external reference.
- Master failure requires re-election before synchronization can resume.

### 2.3 Network Time Protocol (NTP)

**NTP** synchronizes clocks across large-scale networks with UTC as the reference. Unlike Cristian's algorithm which uses a single server, NTP uses a **hierarchy of redundant servers called strata** so that no single point of failure can break synchronization globally.

| S.N. | Level | Meaning |
|---:|---|---|
| 1 | Stratum 0 | Accurate reference devices such as atomic, GPS, or radio clocks. Not on the network directly. |
| 2 | Stratum 1 | Primary servers connected directly to stratum-0 devices. Highest accuracy on the network. |
| 3 | Stratum 2 and above | Secondary servers synchronized from lower-stratum servers. Accuracy decreases with each hop. |

<img width="660" height="480" alt="image" src="https://github.com/user-attachments/assets/edbd5b4d-e010-47fa-beff-abde954b3fc5" />


A client synchronizes from the lowest-stratum server it can reach. Multiple servers are queried simultaneously and inaccurate observations are filtered out — the client prefers the server with the smallest delay.

#### 2.3.1 NTP Operating Modes

| S.N. | Mode | Description |
|---:|---|---|
| 1 | **Multicast** | Server broadcasts its time periodically. Low overhead but lower accuracy — clients cannot measure individual network delay. |
| 2 | **Procedure-call** | Client sends a request and uses the timestamped reply to estimate offset and delay. Similar to Cristian's method. Used for moderate accuracy. |
| 3 | **Symmetric** | Two NTP peers continuously exchange timing messages with each other. Best accuracy; used between high-stratum servers to provide redundancy. |

#### 2.3.2 NTP Timestamp Calculation

For one NTP exchange, four timestamps are recorded:

- $T_1$ — client sends the request
- $T_2$ — server receives the request
- $T_3$ — server sends the reply
- $T_4$ — client receives the reply

The server's clock offset relative to the client:

$$
\theta = \frac{(T_2 - T_1) + (T_3 - T_4)}{2}
$$

The estimated round-trip network delay:

$$
\delta = (T_4 - T_1) - (T_3 - T_2)
$$

The offset $\theta$ tells the client how much to adjust its clock. A positive $\theta$ means the server is ahead; a negative $\theta$ means the server is behind. The client collects several such samples, discards outliers, and applies the best estimate. This filtering step is what makes NTP more robust than a single Cristian-style exchange.

**Limitations:**
- Asymmetric routing and variable network delay reduce accuracy.
- Accuracy is typically in the low milliseconds on the internet; sub-millisecond on a LAN.

## 3. Logical Clocks and Event Ordering

### 3.1 Happened-Before Relation

A **logical clock** orders events according to causality instead of real time. Lamport's **happened-before relation**, written $a \rightarrow b$, is defined by three rules:

1. **Process order:** If $a$ and $b$ occur in the same process and $a$ occurs first, then $a \rightarrow b$.
2. **Message order:** If $a$ sends a message that $b$ receives, then $a \rightarrow b$.
3. **Transitivity:** If $a \rightarrow b$ and $b \rightarrow c$, then $a \rightarrow c$.

If neither $a \rightarrow b$ nor $b \rightarrow a$, the events are **concurrent**, written $a \parallel b$.

### 3.2 Lamport's Logical Clock

Lamport's logical clock assigns one increasing integer timestamp to every event. Each process $P_i$ maintains a scalar counter $L_i$.

**Implementation rules:**

1. Initially, every process sets $L_i = 0$.
2. Before every internal or send event, process $P_i$ increments its clock:

$$
L_i = L_i + 1
$$

3. A sent message carries the current timestamp $t = L_i$.
4. Before timestamping the receipt of a message carrying $t$, process $P_j$ updates:

$$
L_j = \max(L_j, t) + 1
$$

These rules satisfy the **clock condition**:

$$
a \rightarrow b \implies L(a) < L(b)
$$

For a total order, events can be ordered lexicographically using $(L(e), \text{process ID})$. The process ID breaks ties between equal timestamps.

**Example:**

| S.N. | Event | Timestamp calculation |
|---:|---|---|
| 1 | $P_1$ performs event $a$. | $L(a) = 1$ |
| 2 | $P_1$ sends a message at event $b$. | $L(b) = 2$ |
| 3 | $P_2$ independently performs event $c$. | $L(c) = 1$ |
| 4 | $P_2$ receives the message carrying timestamp 2 at event $d$. | $L(d) = \max(1,2)+1 = 3$ |

Events $b$ and $c$ are concurrent even though $L(c) < L(b)$. This demonstrates that timestamp order alone does not prove causality.

**Advantages:**

- It is simple to implement because each process maintains only one scalar counter.
- Each message carries a constant-size timestamp, so its storage and communication overhead is $O(1)$.
- It provides an order consistent with causality and can create a total order by combining the timestamp with a process-ID tie-breaker.

**Limitations:**

- $L(a) < L(b)$ does not imply $a \rightarrow b$; the converse of the clock condition is false.
- Scalar timestamps cannot determine whether two events are concurrent.
- Lamport clocks do not measure real time or elapsed duration.
- A total order arbitrarily orders concurrent events and loses concurrency information.

### 3.3 Vector Clocks

A **vector clock** extends Lamport's clock so that both causality and concurrency can be identified. In a system of $N$ processes, process $P_i$ maintains vector $V_i[1 \ldots N]$, where $V_i[k]$ records the number of events of $P_k$ known to $P_i$.

**Implementation rules:**

1. Initially, every component is zero.
2. Before an internal or send event, $P_i$ increments its own component:

$$
V_i[i] = V_i[i] + 1
$$

3. A sent message carries a copy of $V_i$.
4. When $P_i$ receives vector $T$, it first merges every component:

$$
V_i[k] = \max(V_i[k], T[k])
$$

5. It then increments its own component to timestamp the receive event.

For vectors $X$ and $Y$, $X < Y$ when every component of $X$ is less than or equal to the corresponding component of $Y$, and at least one component is smaller. Therefore:

- $a \rightarrow b$ if and only if $V(a) < V(b)$.
- $a \parallel b$ when neither vector is smaller; the vectors are incomparable.

**Example:** Consider three processes with vectors initially equal to $[0,0,0]$. Process $P_1$ increments its clock and sends a message with $[1,0,0]$. When $P_2$ receives it, it merges and increments to obtain $[1,1,0]$. Meanwhile, an independent event at $P_3$ is timestamped $[0,0,1]$, which is concurrent with $P_1$'s send because the two vectors are incomparable. If $P_2$ later sends $[1,2,0]$ to $P_3$, the receive event at $P_3$ is timestamped $[1,2,2]$ after merging and incrementing. This shows both a causal chain and a concurrent event.

### 3.4 Lamport Clock vs Vector Clock

| S.N. | Lamport clock | Vector clock |
|---:|---|---|
| 1 | It represents each timestamp using one scalar integer. | It represents each timestamp using an array of $N$ integers. |
| 2 | It guarantees that causality implies timestamp order. | It guarantees that timestamp order is equivalent to causality. |
| 3 | It cannot determine concurrency from timestamps alone. | It detects concurrency using incomparable vectors. |
| 4 | It naturally produces a scalar order and can form a total order using a process-ID tie-breaker. | It naturally produces a causal partial order. |
| 5 | It requires $O(1)$ timestamp space in each message. | It requires $O(N)$ timestamp space in each message. |
| 6 | It is suitable for simple event ordering and mutual-exclusion algorithms. | It is suitable for causal delivery, version tracking, and conflict detection. |

The vector clock solves Lamport clock's inability to infer causality, but its storage and communication overhead grow with the number of processes.

## 4. Causal Ordering of Messages

### 4.1 Causal Message Order

Messages have **causal order** when every process delivers causally related messages in an order consistent with happened-before. If:

$$
\operatorname{send}(m_1) \rightarrow \operatorname{send}(m_2)
$$

then every process that delivers both messages must deliver $m_1$ before $m_2$. Concurrent messages may be delivered in either order.

**Example:** $P_1$ sends update $m_1$. After receiving $m_1$, process $P_2$ sends dependent update $m_2$. If $P_3$ receives $m_2$ first, it buffers $m_2$ until $m_1$ arrives.

To maintain causal delivery, a sender attaches a vector timestamp or equivalent dependency information to each message. A receiver delivers the message only after all causally earlier messages are available; otherwise, it buffers the message. After delivering a missing predecessor, the receiver checks the buffered messages again.

## 5. Global State and State Recording

### 5.1 Global State

The **global state** of a distributed system consists of the local state of every process and the state of every communication channel, represented by messages in transit. It is used for deadlock or termination detection, checkpointing, recovery, debugging, and evaluating global conditions.

A **cut** is a boundary across all process timelines. Events before the boundary are included in the recorded state, while later events are excluded.

### 5.2 Cut Types

| S.N. | Consistent cut | Inconsistent cut | Strongly consistent cut |
|---:|---|---|---|
| 1 | It contains every cause of every included event. | It includes an event but excludes one of its causes. | It is a consistent cut with no messages in transit. |
| 2 | It may include a send while excluding its later receive, so the message is recorded in the channel state. | It includes a receive while excluding the corresponding send, which records an effect without its cause. | It includes both the send and receive of every recorded message. |
| 3 | A message may cross the cut from the recorded past toward the unrecorded future. | A message crosses the cut from an unrecorded send toward an included receive. | No application-message arrow crosses the cut in either direction. |

To classify a cut, inspect each crossing message arrow. A receive-before/send-after crossing makes the cut inconsistent; a send-before/receive-after crossing represents an in-transit message in a consistent cut. If no message crosses a consistent cut, it is strongly consistent.

### 5.3 Distributed Snapshot

A **distributed snapshot** records a consistent global state while the system continues to run. It contains every process's local state and all messages that were in transit at the represented cut.

### 5.4 Chandy-Lamport Snapshot Algorithm

The **Chandy-Lamport algorithm** records a distributed snapshot using special **marker messages**. It assumes that processes do not fail during recording, communication channels are reliable, unidirectional, and FIFO ordered, and every participating process is reachable through the communication graph.

1. **Initiation:** The initiator records its local state and sends a marker on every outgoing channel before sending any later application message.
2. **First marker at a process:** The receiving process records its local state, records the marker's incoming channel as empty, sends markers on all outgoing channels, and begins recording messages on its other incoming channels.
3. **Later marker at a process:** The process stops recording that incoming channel. Messages received after its local-state recording but before the marker form that channel's recorded state.
4. **Completion:** A process finishes after receiving a marker on every incoming channel. All recorded process and channel states together form the global snapshot.

FIFO ordering ensures that a marker separates messages sent before a process recorded its state from messages sent afterward. Thus, a sent-but-not-yet-received message is recorded as an in-transit channel message, and the resulting snapshot is consistent.

## 6. Distributed Debugging

### 6.1 Meaning and Difficulties

**Distributed debugging** finds faults by examining the local states and causal relationships of events across several communicating processes. Because no process can observe the complete system directly, a debugger must evaluate a global condition using a **consistent global state or cut**.

It is difficult because:

- different schedules and network delays make executions nondeterministic;
- clock skew and concurrent events prevent physical timestamps from giving one reliable event order;
- states collected independently from different processes may form an inconsistent global view; and
- logging or pausing a process can change timing and hide a race condition.

### 6.2 Debugging with Logical Time and Snapshots

1. Each process records important events, message identifiers, and Lamport or vector timestamps in a log.
2. The debugger uses these timestamps to reconstruct happened-before relationships and distinguish causal events from concurrent events.
3. It evaluates the suspected global condition on a consistent cut or Chandy-Lamport snapshot and traces the causal chain that produced the faulty state.

**Example:** Suppose processes $P_1$ and $P_2$ maintain related values $X_1$ and $X_2$, and an error exists when $|X_1 - X_2| > d$. The debugger must compare values belonging to the same consistent cut. Combining $X_1$ from one global state with $X_2$ from another may falsely indicate an error that never existed in the actual computation.
