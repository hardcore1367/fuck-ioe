# 4. Distributed Heterogeneous Applications and CORBA

## 1. Essential Background

### 1.1 Homogeneous vs Heterogeneous Distributed Applications

A **homogeneous distributed application** uses similar hardware, operating systems, programming languages, and data formats. A **heterogeneous distributed application** combines components built on different platforms but makes them cooperate as one system.

| S.N. | Homogeneous distributed application | Heterogeneous distributed application |
|---:|---|---|
| 1 | Its components use the same or compatible hardware and operating systems. | Its components may use different hardware and operating systems. |
| 2 | Its components usually use compatible languages, runtimes, and data formats. | Its components may use different languages, runtimes, and data representations. |
| 3 | Communication is comparatively simple because the components follow similar conventions. | Standard interfaces and data conversion are required for interoperability. |
| 4 | An example is a set of Java services running on similar Linux machines. | An example is a Java/Linux client using a C++/Windows server. |

### 1.2 Middleware and Its Role

**Middleware** is a software layer between distributed applications and the underlying operating systems and networks. It gives different components a common way to communicate.

Its main roles are:

- **Interoperability:** It provides standard interfaces and data representations so components written for different platforms can cooperate.
- **Communication:** It supplies high-level mechanisms such as remote invocation and messaging instead of requiring applications to use raw network operations.
- **Transparency and discovery:** It helps locate remote services and hides their physical location and communication details.
- **Common services:** It provides reusable facilities such as naming, security, transactions, and event handling.
- **Portability and reuse:** Application logic depends on the middleware interface rather than one operating system or network implementation.

## 2. CORBA Architecture

### 2.1 CORBA

**Common Object Request Broker Architecture (CORBA)** is an open distributed-object standard defined by the **Object Management Group (OMG)**. It allows a client to invoke an operation on a remote object without depending on that object's location, programming language, operating system, or hardware.

The remote interface is defined using **Interface Definition Language (IDL)**. Language-specific stubs and skeletons connect the client and server programs to the **Object Request Broker (ORB)**.

### 2.3 Main Components

| Component | Function |
|---|---|
| **IDL and IDL compiler** | IDL defines the remote operations and data types independently of programming language. The compiler generates client stubs and server skeletons. |
| **Client stub or DII** | A stub marshals a static request, while the **Dynamic Invocation Interface (DII)** constructs a request at run time without a generated stub. |
| **ORB** | Locates the target object and transports requests, results, and exceptions between the client and server. |
| **Object Adapter/POA** | Connects the ORB to server objects, manages object references and servants, and dispatches each request. |
| **Server skeleton or DSI** | Unmarshals a request and invokes the servant. A generated skeleton handles static interfaces, while the **Dynamic Skeleton Interface (DSI)** can handle requests dynamically. |
| **Servant** | The server-side program object that implements the operations declared in IDL. |
| **Repositories** | The Interface Repository stores IDL information for dynamic invocation, while the Implementation Repository stores server location and activation information. |

### 2.4 Object Request Broker

The **ORB** is CORBA's central communication bus. It accepts requests from a stub or the DII, interprets remote object references, locates target servers, transfers requests and replies between ORBs, and works with the Object Adapter to reach the correct servant. Communication between ORBs commonly uses **IIOP** over TCP/IP.

### 2.5 Object Adapter

The **Object Adapter** is the server-side bridge between the ORB and servant implementations. The standard **Portable Object Adapter (POA)** registers servants, creates remote object references, maps incoming requests to the correct servant, and activates or deactivates servants. The **ORB handles communication and routing**, whereas the **Object Adapter connects a routed request to an actual server object**.

### 2.6 Working of a CORBA Request

1. The interface is written in IDL, and the IDL compiler generates the client stub and server skeleton.
2. The server implements the interface as a servant, registers it with the Object Adapter, and publishes its object reference through a service such as the Naming Service.
3. The client obtains that reference and invokes an operation through a stub or constructs it through the DII.
4. The client side marshals the arguments, and the ORB sends the request to the server ORB.
5. The server ORB passes it through the Object Adapter and skeleton or DSI to the correct servant.
6. The result or exception returns through the same components in reverse order.

## 3. Static and Dynamic Invocation in CORBA

| S.N. | Static invocation | Dynamic invocation |
|---:|---|---|
| 1 | The interface must be known at compile time. | The interface may be discovered at run time. |
| 2 | The client invokes the object through an IDL-generated stub. | The client invokes the object through the Dynamic Invocation Interface. |
| 3 | The client makes a normal method call on the generated stub. | The client supplies the operation name, arguments, and types at run time. |
| 4 | Type checking is performed mainly at compile time. | Type checking is performed mainly at run time. |
| 5 | The Interface Repository is usually unnecessary during invocation. | The Interface Repository is commonly used to obtain operation and type information. |
| 6 | It is simpler and generally faster. | It is more flexible but also more complex and slower. |
| 7 | It is suitable for stable and known interfaces. | It is suitable for generic clients, gateways, and unknown interfaces. |

## 4. CORBA Services

**CORBA services** are standard reusable object interfaces for common distributed-system functions.

| Service | Purpose |
|---|---|
| **Naming Service** | Maps a meaningful name to an object reference so clients can locate an object. |
| **Trading Service** | Finds services by their type and advertised properties. |
| **Event/Notification Service** | Supports asynchronous communication between event suppliers and consumers. |
| **Transaction Service** | Coordinates operations on several objects as one atomic transaction. |
| **Security Service** | Provides authentication, authorization, confidentiality, integrity, and auditing. |
| **Concurrency Control Service** | Coordinates simultaneous access to shared objects using mechanisms such as locks. |
| **Life Cycle Service** | Defines common operations for creating, copying, moving, and deleting CORBA objects. |
