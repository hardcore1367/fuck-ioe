# 4. Distributed Heterogeneous Applications and CORBA

A **distributed heterogeneous application** is a distributed application whose components run on different hardware, operating systems, networks, programming languages, or data formats. The components must still exchange data and use one another's services correctly. **Middleware** provides the common layer that hides these differences, and **CORBA** is an object-oriented middleware standard designed for this purpose.

## 1. Heterogeneity in Distributed Systems

**Heterogeneity** means diversity among the components of a distributed system. A heterogeneous distributed system may contain computers and software from different vendors, but all components cooperate as one application.

### 1.1 Sources of Heterogeneity

1. **Hardware and instruction sets:** Processors may use different instruction sets, word sizes, byte orders, and memory organizations. A binary compiled for one processor usually cannot execute directly on another.
2. **Operating systems:** Nodes may run Linux, Windows, macOS, UNIX, or a real-time operating system. Their process, file, security, and communication interfaces are different.
3. **Networks and protocols:** Components may be connected through Ethernet, Wi-Fi, mobile networks, or other technologies and may support different communication protocols.
4. **Programming languages:** A client written in Java may need to invoke a service written in C++, Python, or another language. Their type systems and calling conventions are not the same.
5. **Data representation:** Machines may represent integers, floating-point values, characters, structures, and byte order differently. Data must be converted to a common external format before transfer.
6. **Database and service interfaces:** Existing databases, legacy systems, cloud services, and vendor products expose different APIs and data models.
7. **Capabilities and devices:** Nodes can differ in processing power, memory, storage, peripherals, and available services. They therefore perform different roles in the same system.

Heterogeneity is common because no single platform is best for every task, organizations must reuse legacy systems, and users choose different devices and vendors. The system must preserve the meaning of data and provide a common way to access services despite these differences.

### 1.2 Example of a Heterogeneous Distributed Application

An online banking application may contain:

- an Android client written in Kotlin;
- a browser client running on Windows or Linux;
- an application server written in Java on Linux;
- a transaction service written in C++ on a mainframe; and
- a database from another vendor.

The clients use the same banking operations even though the components use different languages, operating systems, and data formats. Middleware converts data, locates the correct service, transfers requests, and returns results in a form each component understands.

### 1.3 Homogeneous vs Heterogeneous Distributed Applications

| Basis | Homogeneous distributed application | Heterogeneous distributed application |
|---|---|---|
| **Platforms** | Components use the same or closely compatible hardware and operating system. | Components may use different processors and operating systems. |
| **Programming environment** | Components commonly use the same language, runtime, and libraries. | Components may be written in different languages and use different runtimes. |
| **Data representation** | Data types and byte representation are usually compatible. | Data must often be marshalled into a common external representation. |
| **Communication** | A platform-specific protocol may be sufficient. | A standard protocol and language-neutral interface are required. |
| **Interoperability** | Easy because all components follow the same conventions. | Middleware must translate and hide platform differences. |
| **Development and management** | Simpler to develop and test, but tied to one platform or vendor. | More complex, but supports legacy integration and freedom of platform choice. |
| **Example** | A cluster of Linux machines running services written in Java. | A Java/Linux client using a C++/Windows service and a mainframe database. |

## 2. Middleware

**Middleware** is a software layer placed between distributed applications and the underlying operating systems and networks. It provides uniform interfaces and common services so that components can communicate without handling platform-specific details directly.

### 2.1 Roles of Middleware in a Distributed System

1. **Masks heterogeneity:** It translates different data representations, communication protocols, language bindings, and operating-system interfaces into a common model.
2. **Provides distribution transparency:** Applications can invoke remote services in a form similar to local operations without knowing the service's physical location or communication details.
3. **Supports communication:** It provides higher-level mechanisms such as remote procedure calls, remote method invocation, message queues, and event notification instead of raw socket programming.
4. **Performs marshalling and unmarshalling:** It converts parameters and results into a standard transferable form and reconstructs them at the destination.
5. **Provides naming and discovery:** It maps a logical service or object name to its current network location or remote object reference.
6. **Provides common system services:** Security, transactions, concurrency control, persistence, time, failure handling, and other facilities can be reused by many applications.
7. **Improves portability and interoperability:** A component can be moved to another supported platform, and independently developed components can cooperate through standard interfaces.
8. **Reduces development effort:** Repeated networking and distributed-system functions are supplied by the middleware instead of being implemented separately in every application.

### 2.2 How Middleware Hides Heterogeneity

1. The client calls an operation through a uniform, platform-independent interface.
2. Client-side middleware marshals the operation name and arguments into a common external representation.
3. The middleware discovers the remote component and sends the request using a standard protocol.
4. Server-side middleware converts the request to the server's language and platform representation, then dispatches it to the service.
5. The result follows the reverse path and is returned in the client's native representation.

Examples of distributed middleware include **CORBA**, **Java RMI**, RPC frameworks, and message-oriented middleware. CORBA is specifically designed to support distributed objects across different languages and platforms.

## 3. Objects in Distributed Systems

An **object** contains state in the form of attributes and behavior in the form of methods. In a distributed object system, an object can reside in one process while clients in other processes invoke its methods through a remote object reference.

### 3.1 Distributed Object Model

- A **service provider** or **servant** implements an object and its operations on the server.
- A **client** obtains a remote object reference and requests the operations offered by that object.
- An **interface** declares the operations, parameter types, return types, and exceptions visible to clients; it does not reveal the implementation.
- A **remote object reference** uniquely identifies a distributed object and enables middleware to locate and address it.
- A **client proxy or stub** represents the remote object locally, while a **server skeleton** dispatches received requests to the actual servant.
- An **Object Request Broker (ORB)** connects clients and remote objects, and reusable **object services** support naming, security, transactions, and object life-cycle management.

### 3.2 Interface Definition Language

An **Interface Definition Language (IDL)** is a language-neutral notation used to define the public interface of a distributed object. CORBA IDL describes the available operations and their data types independently of the programming language used by either side.

The role of IDL is to:

- act as a formal contract between the client and server;
- provide common CORBA data types for language-independent communication;
- let an IDL compiler generate a client stub and server skeleton for selected programming languages; and
- separate the interface from the object implementation so that either side can change internally without changing the contract.

```idl
interface Account {
    double getBalance(in string accountNumber);
    void deposit(in string accountNumber, in double amount);
};
```

The same interface can generate, for example, a Java client stub and a C++ server skeleton. The generated code handles data conversion and connects application code to the ORB.

## 4. The CORBA Approach and Architecture

**Common Object Request Broker Architecture (CORBA)** is an open standard defined by the **Object Management Group (OMG)** for communication between distributed objects. It allows a client to invoke an operation on a remote object regardless of the object's location, operating system, hardware, or implementation language.

CORBA follows an object-oriented client-server model. The client depends on an IDL-defined interface, while the server provides a servant that implements that interface. The **ORB** carries requests and replies between them and provides location and communication transparency.

### 4.2 Components of CORBA Architecture

1. **Client application:** It obtains a remote object reference and invokes an operation on the remote CORBA object.
2. **CORBA object and servant:** A CORBA object is visible through an IDL interface. The servant is the language-specific server object that actually implements the operations.
3. **Client stub or proxy:** The IDL compiler generates it in the client's language. It provides normal method calls, marshals arguments into a request, and unmarshals results or exceptions from the reply.
4. **Dynamic Invocation Interface (DII):** It lets a client construct and invoke a request at run time when a compiled stub or complete interface is not known at compile time.
5. **Server skeleton:** The IDL compiler generates it in the server's language. It unmarshals request arguments, calls the correct servant method, and marshals the result or exception.
6. **Dynamic Skeleton Interface (DSI):** It allows a server to receive requests for interfaces without using a statically generated skeleton. It is the server-side counterpart of dynamic invocation.
7. **Interface Repository:** It stores run-time metadata about registered IDL interfaces, including operation names, parameter types, return types, and exceptions. The DII and development tools can inspect this information.
8. **Implementation Repository:** It stores information about object implementations and server programs, including their location and activation details. It helps the ORB and Object Adapter locate or start a required server.
9. **CORBA services:** They provide reusable distributed-object facilities such as naming, trading, transactions, security, time, and object life-cycle management.

### 4.3 Object Request Broker

The **Object Request Broker (ORB)** is the central communication component of CORBA. It acts as a software bus between clients and object implementations, so neither side needs to know the other's location, language, operating system, or communication mechanism.

The ORB performs the following functions:

- interprets remote object references and binds a client to the target object;
- accepts invocation requests from client stubs or the DII;
- locates the target server, working with the Implementation Repository when activation is needed;
- transfers request and reply messages between local or remote address spaces;
- preserves operation names, argument types, results, and exceptions across different language mappings;
- cooperates with the Object Adapter to dispatch a request to the correct servant; and
- supports interoperability between different ORB implementations through standard protocols.

### 4.4 Object Adapter

An **Object Adapter** is the server-side component that bridges the ORB's language-independent object model and the language-specific servant implementation. The commonly used standard adapter is the **Portable Object Adapter (POA)**.

Its main responsibilities are:

1. **Object registration:** It registers servants and associates them with CORBA object identities.
2. **Reference creation:** It creates remote object references that clients can use to address CORBA objects.
3. **Request dispatch:** It maps an incoming object reference to the correct servant and sends the request through the proper skeleton or DSI.
4. **Servant activation:** It activates a servant when a request arrives and deactivates it when it is no longer required, according to configured policies.
5. **Life-cycle and policy control:** It manages policies such as object persistence, servant retention, request processing, and object identity.

The ORB handles communication and routing, whereas the Object Adapter connects those requests to actual server objects.

### 4.5 Working of a CORBA Request

1. The remote interface is written in CORBA IDL. An IDL compiler generates language-specific client stubs and server skeletons for static invocation.
2. The server implements the interface in a servant, registers it with the Object Adapter, and publishes its remote object reference, usually through the Naming or Trading service.
3. The client obtains the object reference and invokes an operation through a generated stub or constructs the request through the DII.
4. The stub or DII marshals the operation name and arguments and passes the request to the client ORB.
5. The client ORB locates the server ORB and transfers the request. For different ORBs over TCP/IP, CORBA normally uses IIOP.
6. The server ORB passes the request to the Object Adapter, which identifies or activates the target servant.
7. The server skeleton or DSI unmarshals the arguments and invokes the required method on the servant.
8. The servant's result or exception is marshalled and returned through the ORBs. The client stub or DII then gives it to the client application.

### 4.6 GIOP and IIOP

- **General Inter-ORB Protocol (GIOP)** defines a standard message format and rules for requests, replies, object location, and errors between CORBA ORBs.
- **Internet Inter-ORB Protocol (IIOP)** is GIOP mapped onto TCP/IP. It allows CORBA products from different vendors to communicate over the Internet or an intranet.

These protocols make the CORBA architecture interoperable instead of restricting communication to one vendor's ORB.

## 5. Invocation Approaches in CORBA

CORBA supports **static invocation** when an interface is known during development and **dynamic invocation** when a request must be constructed at run time. The server object normally performs the same operation in either case; the difference is mainly how the client forms the request.

Show `IDL -> IDL Compiler -> Client Stub -> ORB -> Server Skeleton -> Servant` for static invocation, and `Interface Repository -> Dynamic Invocation Interface -> ORB -> Server Skeleton/DSI -> Servant` for dynamic invocation.

### 5.1 Static Invocation

In **static invocation**, the object's IDL interface and required operations are known at compile time. The IDL compiler generates a client stub and server skeleton.

1. The client calls a method directly on the generated stub.
2. The stub performs compile-time type-checked marshalling and sends the request through the ORB.
3. The server skeleton unmarshals the request and invokes the servant.
4. The result follows the reverse path to the client.

Static invocation is simpler, faster, and type-safe, so it is the usual choice for stable interfaces. However, an interface change requires regeneration and recompilation of the affected stubs or skeletons.

### 5.2 Dynamic Invocation

In **dynamic invocation**, the client constructs a request at run time through the **Dynamic Invocation Interface (DII)** instead of calling a pre-generated stub. It is useful when the target object's interface was not known when the client was compiled.

1. The client obtains the target object's remote reference.
2. It obtains operation and parameter information from the Interface Repository or from information supplied at run time.
3. It creates a request containing the operation name, argument list, argument types, expected result type, and exception information.
4. The DII submits the request to the ORB, which delivers it to the server.
5. The reply is returned in a generic form that the client examines at run time.

The server can use an ordinary generated skeleton, so it does not necessarily know whether the client used static or dynamic invocation. A highly dynamic server may instead use the **DSI**. Dynamic invocation is flexible but requires more code, has run-time type checking, and usually has greater overhead.

### 5.3 Static vs Dynamic Invocation

| Basis | Static invocation | Dynamic invocation |
|---|---|---|
| **Interface knowledge** | Interface is known at compile time. | Interface may be discovered at run time. |
| **Client mechanism** | Client calls a generated stub directly. | Client builds a request through the DII. |
| **IDL-generated client stub** | Required. | Not required for the invoked interface. |
| **Interface Repository** | Normally not needed during invocation. | Commonly used to discover operation and type information. |
| **Type checking** | Mostly performed at compile time. | Performed mainly at run time. |
| **Complexity and speed** | Simpler and generally faster. | More complex and generally slower. |
| **Flexibility** | Suitable for stable, known interfaces. | Suitable for generic tools, gateways, browsers, and changing or unknown interfaces. |

## 6. CORBA Services

**CORBA services** are standard, reusable object interfaces that solve common distributed-system problems. Applications access them through the ORB instead of implementing the same facilities separately.

| CORBA service | Purpose |
|---|---|
| **Naming Service** | Associates a meaningful hierarchical name with an object reference so that clients can find an object by name. |
| **Trading Service** | Lets a service provider advertise an object with properties and allows a client to find an object by required service type and properties. |
| **Transaction Service** | Coordinates recoverable operations on multiple objects as one transaction and supports atomic completion using protocols such as two-phase commit. |
| **Concurrency Control Service** | Supplies locks and coordination mechanisms so that transactions or threads can access shared objects without producing inconsistent results. |
| **Security Service** | Provides authentication, authorization, access control, confidentiality, integrity, delegation, and security auditing for distributed object interactions. |
| **Time Service** | Supplies common time values, clock-error bounds, time intervals, and support for time-triggered events in distributed applications. |
| **Life Cycle Service** | Defines common operations and factories for creating, copying, moving, and deleting CORBA objects while preserving their valid references and storage requirements. |
| **Query Service** | Provides declarative operations for selecting objects from collections according to specified conditions. |
| **Externalization Service** | Marshals and unmarshals an object's state so that it can be transferred, stored, and later reconstructed. |
| **Event and Notification Services** | Support asynchronous communication through event channels, allowing suppliers and consumers to exchange events without being directly coupled. |
