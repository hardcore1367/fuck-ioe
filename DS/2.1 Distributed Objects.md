
A **distributed object** is an object (data + methods) that resides on one machine but can be accessed and invoked by processes running on other machines over a network — just like calling a local object.

The key idea is **location transparency** — the caller does not need to know where the object physically lives. It calls a method on the object the same way it would call a local one, and the underlying system handles the network communication.

This is the foundation for both **RPC** (Remote Procedure Call) and **RMI** (Remote Method Invocation) which are the two main mechanisms for implementing distributed object communication.

## 1. Remote Procedure Call (RPC)

**RPC** is a mechanism that allows a process on one machine to call a procedure (function) on a remote machine as if it were a local call. The network communication is completely hidden from the programmer — the calling code looks identical to a local function call. RPC is the foundation of distributed computing, enabling client-server communication without the programmer manually handling sockets or message passing.

### Architecture

The RPC architecture has five key components:

- **Client** — the process that wants to call a remote procedure. It calls the procedure as if it were local, with no knowledge of the network communication happening underneath.
- **Client Stub** — a local proxy on the client side that represents the remote procedure. It receives the call from the client, **marshals** (serializes) the procedure name and parameters into a network message, and hands it to the RPC runtime for sending.
- **RPC Runtime** — the middleware layer that handles all low-level network communication on both sides. It is responsible for sending and receiving messages, managing connections, handling retransmissions, and reporting errors. The client and server stubs never touch the network directly — they always go through the runtime.
- **Server Stub (Skeleton)** — receives the incoming message from the RPC runtime on the server side, **unmarshals** (deserializes) the parameters, and calls the actual procedure with those parameters. After the procedure returns, it marshals the result and hands it back to the runtime to send to the client.
- **Server** — the machine that hosts the actual procedure. It executes the procedure normally and returns the result to the server stub. It has no knowledge that the caller is a remote client.

### Functional Steps of RPC

1. **Client calls the client stub** — the call looks like a normal local function call with parameters.
2. **Marshalling** — the client stub packages the procedure name and parameters into a network message (serializes data into a common format).
3. **Message is sent** — the RPC runtime sends the message over the network to the server.
4. **Server receives the message** — the server RPC runtime receives the message and passes it to the server stub.
5. **Unmarshalling** — the server stub unpacks the message and extracts the parameters.
6. **Actual procedure executes** — the server stub calls the real procedure with the extracted parameters.
7. **Result is marshalled** — the return value is packaged into a response message.
8. **Result is sent back** — the response is sent over the network to the client.
9. **Client stub receives result** — the client stub unmarshals the result and returns it to the calling code as if it were a local return value.

### Communication Semantics

Communication semantics define what happens when an RPC fails — how many times the procedure is guaranteed to execute:

- **Maybe Semantics** — the procedure may execute once or not at all. No retry on failure. Used when performance matters more than reliability. Unreliable but fast.
- **At Least Once Semantics** — the client retries until it gets a response. The procedure may execute multiple times. Acceptable only for **idempotent** operations (operations that give the same result when repeated, like reading a value).
- **At Most Once Semantics** — the system ensures the procedure executes either once or not at all — never more. Duplicate requests are detected and suppressed. Most commonly used in practice.
- **Exactly Once Semantics** — the procedure executes exactly one time regardless of failures. The ideal but hardest to achieve, requires complex protocols and persistent state.

### Advantages

1. Simple programming model — remote calls look identical to local calls, hiding network complexity.
2. Language and platform independent when combined with IDL (Interface Definition Language).
3. Well-established and widely supported across many frameworks and languages.
4. Promotes modularity — services can be developed and deployed independently.

### Disadvantages

1. **Tight coupling** — client and server must agree on the same interface; any change breaks compatibility.
2. **Failure handling is hard** — network failures, server crashes, and timeouts are difficult to handle transparently.
3. **Performance overhead** — marshalling, unmarshalling, and network round trips are much slower than local calls.
4. **No support for asynchronous communication** — traditional RPC is synchronous; the client blocks and waits for a response.
5. **Heterogeneity issues** — different data representations on different machines require careful marshalling.

### Issues in RPC

1. **Parameter Passing** — passing complex data structures (pointers, arrays, objects) over a network is non-trivial. Pointers cannot be passed directly — data must be serialized into a flat format.
2. **Binding** — the client must locate and connect to the correct server before making a call. This is done through a **name server or binder** that maps procedure names to server addresses.
3. **Failure Handling** — it is hard to distinguish between a slow server and a crashed server. The client must decide when to timeout and whether to retry.
4. **Security** — the server must authenticate the client to prevent unauthorized procedure calls.
5. **Performance** — network latency makes RPC inherently slower than local calls, making it unsuitable for fine-grained frequent interactions.


### Types of RPC

**1. Callback RPC** — in standard RPC the client always calls the server. In callback RPC, the roles can reverse — the server calls back the client after completing the operation. The client registers a callback function before making the request. When the server finishes processing, it invokes that function on the client. This is useful when the server needs to notify the client of an event, a result that takes time to compute, or a state change.

**2. Broadcast RPC** — the client sends a single RPC request to all servers on the network simultaneously rather than targeting one specific server. Every server that receives the message executes the procedure and sends back a response. The client collects all responses. This is useful for discovering which server holds a particular resource, querying all nodes for their status, or performing a system-wide operation in one shot.

**3. Batch Mode RPC** — instead of sending one network message per RPC call, multiple RPC calls are grouped together and sent as a single message to the server. The server processes all calls in the batch and returns all results together. This significantly reduces network overhead and improves throughput when many small sequential calls need to be made, since the cost of network round trips is paid once for the entire batch rather than once per call.




## 2. Remote Method Invocation (RMI)

**RMI** is the object-oriented extension of RPC. Instead of calling a remote procedure, a client invokes a **method on a remote object** that lives on another machine — exactly as if it were a local object. RMI is the foundation of distributed object computing and is most commonly associated with Java RMI, though the concept applies broadly.

The key difference from RPC: RMI works with **objects** (data + methods together) rather than plain procedures, and supports object-oriented concepts like inheritance and polymorphism across the network.


### Architecture

- **Remote Object** — an object that lives on the server but can be accessed by clients on other machines. It implements a **remote interface** that defines which methods are accessible remotely.
- **Remote Interface** — a publicly defined interface (written in IDL or Java interface) that specifies the methods the remote object exposes. The client only ever sees this interface, never the actual object implementation.
- **Client Stub (Proxy)** — a local representative of the remote object on the client side. The client calls methods on the stub as if calling the real object. The stub marshals the method name and parameters and sends them to the server.
- **Server Skeleton (Dispatcher)** — receives the incoming request on the server side, unmarshals the parameters, and dispatches the call to the actual remote object. After execution, it marshals the result and sends it back.
- **RMI Registry** — a naming service where servers register their remote objects under a name. Clients look up this registry to obtain a reference to the remote object before invoking methods on it.
- **Remote Reference Module** — handles the translation between local references and remote object references. Maintains a table mapping remote object references to actual objects.

### Static vs Dynamic RMI

**Static (Compiled) RMI:**
- Stubs and skeletons are **generated at compile time** from the remote interface using a tool (e.g. `rmic` in Java).
- The client knows the remote interface at compile time and calls methods directly on the stub.
- Simple, type-safe, and fast since all method signatures are known in advance.
- Inflexible — if the remote interface changes, stubs must be recompiled and redistributed.
- Used when the interface is stable and known ahead of time.

**Dynamic (Reflection-based) RMI:**
- Stubs are **generated at runtime** using reflection — the system inspects the remote interface dynamically without pre-compiled stubs.
- The client can invoke methods on remote objects whose interfaces were not known at compile time.
- More flexible — new remote objects and methods can be added without recompiling clients.
- Slightly more complex and overhead is higher due to runtime reflection.
- Used in dynamic environments where interfaces change or are not known in advance. Modern Java RMI uses dynamic stubs by default.


### Operation of RMI

1. **Server creates the remote object** and registers it with the RMI registry under a name (e.g. `"BankService"`).
2. **Client looks up the registry** using the object name and obtains a stub (remote reference) to the object.
3. **Client calls a method** on the stub — the call looks identical to a local method call.
4. **Stub marshals** the method name, parameters, and object reference into a request message and sends it over the network.
5. **Skeleton receives** the message, unmarshals the parameters, and invokes the corresponding method on the actual remote object.
6. **Remote object executes** the method and returns a result.
7. **Skeleton marshals the result** into a reply message and sends it back.
8. **Stub receives the reply**, unmarshals the result, and returns it to the client as if it were a local return value.


### Structure of Request-Reply Message

Every RMI communication follows a **request-reply protocol**. The messages have a defined structure:

**Request Message:**
| Field | Description |
|---|---|
| **Message Type** | Identifies this as a request (e.g. type = 0) |
| **Request ID** | Unique identifier to match this request with its reply |
| **Remote Object Reference** | Identifies which remote object the method should be invoked on |
| **Method ID** | Identifies which method to call (method name or hash) |
| **Arguments** | Marshalled parameters passed to the method |

**Reply Message:**
| Field | Description |
|---|---|
| **Message Type** | Identifies this as a reply (e.g. type = 1) |
| **Request ID** | Same ID as the request — used to match reply to the original call |
| **Status** | Success or exception/error indicator |
| **Result / Exception** | The marshalled return value, or an exception if the call failed |


### Remote Object Reference

A **remote object reference** is a unique identifier that allows a client to refer to a specific remote object anywhere in the distributed system. It is the distributed equivalent of a local memory pointer.

A remote object reference contains:
- **Internet address (IP)** of the machine hosting the object
- **Port number** the server is listening on
- **Time of creation** to distinguish objects created at different times on the same machine
- **Object number** — a local identifier unique within that server process
- **Interface reference** — identifies the remote interface the object implements

These fields together guarantee that no two remote objects in the entire system share the same reference. The client uses this reference every time it wants to invoke a method, and the remote reference module on the server uses it to locate the correct object.



### Issues in RMI

1. **Parameter Passing** — local objects passed as parameters must be **serialized** (deep copied) and sent over the network. Unlike local calls where references are passed, RMI passes objects by value. This means the remote side gets a copy, not the original — changes on the server do not affect the client's object.

2. **Distributed Garbage Collection** — when a remote object is no longer referenced by any client, it should be garbage collected. But the server cannot know when all remote clients have dropped their reference. Java RMI uses a **lease-based** mechanism where clients periodically renew their interest in an object; if the lease expires, the object can be collected.

3. **Failure Handling** — it is hard to distinguish between a slow server, a crashed server, and a network failure. RMI must define what semantics apply (at-most-once is standard in Java RMI) and the client must handle `RemoteException` for any network or server failure.

4. **Security** — since remote objects are accessible over the network, unauthorized clients could invoke sensitive methods. RMI must be combined with authentication and access control mechanisms.

5. **Performance** — marshalling, unmarshalling, serialization, and network round trips make remote method calls orders of magnitude slower than local calls. Fine-grained frequent remote calls are a major performance anti-pattern in RMI-based systems.

6. **Naming and Registry** — clients must be able to locate remote objects. The RMI registry solves this for simple cases but becomes a bottleneck and single point of failure in large systems. More robust naming services (like JNDI or CORBA naming) are needed at scale.



| **S.N.** | **RPC (Remote Procedure Call)** | **RMI (Remote Method Invocation)** |
|---|---|---|
| 1 | A mechanism to call a **procedure/function** on a remote machine as if it were a local function call. | A mechanism to invoke a **method on a remote object** residing on another machine as if it were a local object. |
| 2 | Based on **procedural programming** — no concept of objects, state, or inheritance. | Based on **object-oriented programming** — supports objects, inheritance, and polymorphism across the network. |
| 3 | Passes **simple data types** as parameters. Complex structures like pointers are difficult to pass and require manual handling. | Passes **objects as parameters** through serialization. Entire objects can be sent across the network automatically. |
| 4 | Uses **client stub and server stub** to marshal and unmarshal procedure calls. | Uses **client stub (proxy) and server skeleton** to marshal and unmarshal method calls on remote objects. |
| 5 | **Language independent** — RPC can work across different programming languages using IDL to define the interface. | Traditionally **language dependent** — Java RMI works only within Java environments, though CORBA extends this. |
| 6 | Does not maintain **object state** — each procedure call is independent with no notion of an object persisting between calls. | Maintains **object state** on the server — the remote object persists and its state is preserved across multiple method calls. |
| 7 | Uses a **binder/name server** to locate the remote procedure before making a call. | Uses an **RMI registry** to register and look up remote objects by name before invoking methods. |
| 8 | **Older and lower level** — widely used in systems programming, OS services, and network protocols like NFS and DCE. | **Newer and higher level** — used in modern distributed object systems, enterprise applications, and middleware like CORBA and EJB. |



## 3. Interface Definition Language (IDL)

**IDL (Interface Definition Language)** is a language-neutral specification language used to define the interface of a remote object or procedure — describing what methods are available, what parameters they take, and what they return — without specifying how they are implemented.

IDL sits between the client and server, acting as a **contract** that both sides must follow. Neither side needs to know how the other is implemented — they only need to agree on the interface defined in IDL.

### Role of IDL in Distributed System

**1. Language Independence** — IDL allows a client written in Java to communicate with a server written in C++ because both sides generate their stubs from the same IDL definition. The IDL compiler translates the interface into the native language of each side automatically.

**2. Stub Generation** — the IDL compiler takes the IDL definition and automatically generates **client stubs** and **server skeletons**. This saves the programmer from manually writing marshalling and unmarshalling code.

**3. Interface Contract** — IDL acts as a formal contract between client and server. As long as both sides implement the same IDL-defined interface, they can communicate regardless of internal implementation differences.

**4. Separation of Interface from Implementation** — IDL forces a clean separation between what a service exposes (interface) and how it works internally (implementation). This promotes modularity and allows either side to change its implementation without affecting the other.

**5. Interoperability** — in systems like CORBA, the IDL definition is the single source of truth for all participating components across different languages, operating systems, and hardware. This makes large heterogeneous distributed systems possible.

### Example

```idl
interface BankService {
    float getBalance(in string accountId);
    void deposit(in string accountId, in float amount);
};
```

This IDL definition is compiled into Java stubs for the client and C++ skeletons for the server. Both sides communicate correctly without knowing the other's language.




## 4. Events and Notifications in Distributed Object Communication

In standard RPC/RMI, communication is **client-initiated** — the client always calls the server and waits for a response. But in many real systems, the server needs to **notify the client** when something happens — without the client constantly asking. This is the problem that event-based communication solves.

### What is an Event

An **event** is a significant change in state or an occurrence that happens within a distributed object — such as a stock price changing, a new message arriving, or a sensor reading crossing a threshold. When an event occurs, interested parties need to be informed.

### What is a Notification

A **notification** is the message sent to all interested parties when an event occurs. Instead of clients polling the server repeatedly asking "has anything changed?", the server automatically pushes a notification to all subscribers the moment the event happens.

### Event-Based Communication Model

The standard model used is **Publish-Subscribe**:

- **Publisher (Event Source)** — the object that generates events. It does not know who is listening — it simply publishes events when they occur.
- **Subscriber (Listener)** — the object that wants to be informed of events. It registers its interest with the event service and provides a callback method to be called when the event occurs.
- **Event Service (Notification Bus)** — the middleware that sits between publishers and subscribers. It receives events from publishers and delivers notifications to all registered subscribers.

```
Publisher → Event Service → Subscriber 1
                         → Subscriber 2
                         → Subscriber 3
```

### How it Works

1. **Subscriber registers** its interest in a specific type of event with the event service, providing a callback method or listener interface.
2. **Publisher generates an event** when a state change occurs and sends it to the event service.
3. **Event service matches** the event to all registered subscribers interested in that event type.
4. **Notifications are delivered** to all matching subscribers by invoking their registered callback methods.
5. **Subscriber handles the notification** in its callback — updating UI, triggering further actions, etc.
6. **Subscriber can unregister** when it no longer wants to receive notifications.
