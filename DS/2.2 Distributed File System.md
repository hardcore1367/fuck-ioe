# 2.2 Distributed File System

A **Distributed File System (DFS)** is a file system that allows files to be stored, managed, and accessed across multiple machines on a network — while appearing to the user as a single unified file system on one machine. Users can read and write files without knowing which physical machine or disk actually stores them.

The core idea is **location transparency** — a file named `/shared/report.pdf` looks the same to every user regardless of which server it is physically stored on.

## 1. File Service Architecture

The DFS architecture is divided into three distinct components:

### 1.1 Flat File Service

The **flat file service** is responsible for the actual storage and retrieval of file data. It deals with the contents of files — reading, writing, and managing data at the byte level. It works with **Unique File Identifiers (UFIDs)** instead of file names — every file is identified by a unique, location-independent ID. The flat file service does not know about directory structure or file names at all — it only knows UFIDs and data.

Operations it provides: `Read`, `Write`, `Create`, `Delete`, `GetAttributes`, `SetAttributes`.

### 1.2 Directory Service

The **directory service** maps human-readable file names to UFIDs. It maintains the directory structure — folders, paths, and the mapping between names and actual files. When a client wants to access `/home/prime/notes.md`, the directory service resolves that path and returns the corresponding UFID to the client, which then uses it to talk to the flat file service.

Operations it provides: `Lookup(filename)`, `AddName`, `RemoveName`, `GetNames`.

### 1.3 Client Module

The **client module** runs on each client machine and provides a unified interface to the application. It integrates the flat file service and directory service into a single familiar file system interface (`open`, `read`, `write`, `close`). The application never talks to the flat file or directory service directly — it always goes through the client module.

```
Application
     ↓
Client Module
     ↓          ↓
Directory   Flat File
Service      Service
```

## 2. Sun NFS (Sun Network File System)

**Sun NFS** is a specific implementation of a distributed file system developed by **Sun Microsystems** that enables transparent access to remote files over a network. It was designed to allow any machine on the network to access files stored on a remote Sun server as if they were local. Sun NFS is built entirely on top of **RPC and XDR (External Data Representation)** for machine-independent communication, and is **stateless** by design — the server maintains no information about connected clients between requests. Every request is completely self-contained and includes all information needed to complete it.

### 2.1 Architecture

**1. NFS Client** — runs on the client machine and intercepts all file system calls from applications (`open`, `read`, `write`, `close`). It translates these calls into Sun NFS remote procedure calls and sends them to the server over the network. The application has no idea it is accessing a remote file — the client module makes remote access completely transparent. The client is also responsible for **caching** file data locally to reduce repeated network requests, and for tracking the current file offset itself since the server is stateless.

**2. NFS Server** — runs on the Sun machine that physically stores the files. It listens for incoming RPC requests from clients, performs the actual file operations on its local disk, and returns the result. Since Sun NFS is stateless, the server does not remember anything about previous requests — each request arrives with the file handle, byte offset, and byte count fully specified, so the server simply executes it and responds. Multiple clients can connect to the same server simultaneously. The server exports specific directories that clients are permitted to mount and access.

**3. NFS Protocol (RPC/XDR-based)** — all communication between client and server happens through the Sun NFS protocol built on top of RPC and XDR. XDR ensures that data is represented in a machine-independent format so clients and servers running different hardware can communicate correctly. Originally ran over UDP for performance; modern versions use TCP for reliability. Every operation defined in the protocol is **idempotent** by design — if a request is lost and retransmitted, executing it twice produces the same result, which is essential for stateless operation.

### 2.2 Operation

1. **Search for the file within directory** — the client sends a `LOOKUP` RPC with the directory file handle and filename. The server searches its directory structure and returns the file handle for the requested file, which the client uses for all subsequent operations on that file.

2. **Read a set of directory entries** — the client sends a `READDIR` RPC to retrieve the list of files and subdirectories within a given directory. Used when a user lists folder contents — the server returns names and file handles of all entries.

3. **Manipulate links and directories** — the client can create, remove, and rename files and directories, and manage hard and symbolic links through RPC calls (`CREATE`, `REMOVE`, `RENAME`, `MKDIR`, `SYMLINK`). All these operations are performed on the server's local file system.

4. **Read file attribute** — the client sends a `GETATTR` RPC to retrieve metadata about a file — size, owner, permissions, and timestamps. Also used by the client to validate its locally cached copy by comparing the server's timestamp with the cached version.

5. **Write file attribute** — the client sends a `SETATTR` RPC to modify file metadata such as permissions, ownership, or access and modification timestamps on the server.

6. **Read file data** — the client sends a `READ` RPC with the file handle, byte offset, and byte count. The server reads the requested bytes from disk and returns them. The client maintains the offset itself since the server is stateless.

7. **Write file data** — the client sends a `WRITE` RPC with the file handle, offset, and data to be written. The server writes to disk immediately and returns success or error. Sun NFS uses **write-through** — writes go directly to the server to maintain consistency across all clients.


## 3. Stateful vs Stateless DFS

| **S.N.** | **Stateful DFS** | **Stateless DFS** |
|---|---|---|
| 1 | Server maintains client session information between requests — open files, current position, access rights. | Server maintains no information about clients between requests — every request is completely self-contained. |
| 2 | Client sends short requests since the server already has the context: `read next 100 bytes`. | Client must include all context in every request: `read 100 bytes from offset 400 of file X using handle H`. |
| 3 | More efficient for sequential access — less data transmitted per request. | More data per request since full context must be resent every time, increasing network overhead. |
| 4 | Server crash causes all session state to be lost — crash recovery is complex and expensive. | Server crash has no impact on correctness — client simply resends the same self-contained request. |
| 5 | Harder to replicate since session state must be kept synchronized across all replicas. | Easy to replicate — since no state is stored, any replica can handle any request identically. |
| 6 | Server must handle timeouts to detect crashed clients and clean up their state. | No timeout management needed — server has no concept of connected clients at all. |
| 7 | Scales poorly — each connected client consumes server memory for session state. | Scales well — server resource usage does not grow with the number of connected clients. |
| 8 | Example: FTP, OpenAFS. | Example: Sun NFS. |


## 4. Role of DFS in Resource Sharing

Resource sharing is one of the primary motivations for building a distributed file system. A DFS allows hardware and data resources stored on one machine to be accessed and used by many machines on the network simultaneously.

**1. Shared Storage** — instead of every machine having its own separate copy of files, a DFS provides a single shared storage pool accessible to all nodes. Any file stored on the server is immediately available to all authorized clients without manual copying or transfer.

**2. Eliminates Data Duplication** — without a DFS, users on different machines maintain separate copies of the same file, leading to inconsistency. A DFS ensures all users access the same single copy, eliminating duplication and keeping data consistent.

**3. Centralized Data Management** — files, databases, and application data are stored centrally and managed from one place. Backup, access control, and administration are all handled at the server level rather than on each individual machine.

**4. Hardware Resource Sharing** — disk storage on one machine can be shared across the entire network. Organizations do not need to provision large local disks on every machine — clients use remote storage transparently.

**5. Collaborative Access** — multiple users can work on the same files simultaneously. DFS provides locking and concurrency control mechanisms to ensure that concurrent access does not corrupt shared data.

**6. Platform Independence** — a DFS allows machines running different operating systems and hardware to share the same files through standard protocols, making resource sharing possible in heterogeneous environments.

**7. Scalability of Shared Resources** — as storage needs grow, new servers can be added to the DFS and their storage becomes immediately available to all clients — without any changes on the client side.


## 5. DNS (Domain Name System)

**DNS** is a distributed naming system that translates human-readable domain names (like `www.google.com`) into IP addresses (like `142.250.180.46`) that machines use to communicate. DNS is itself a distributed system — no single server knows all mappings. The responsibility is distributed across a hierarchy of servers.

### 5.1 DNS Hierarchy

![](https://github.com/user-attachments/assets/5b873345-c41e-4711-8522-dd72986a5cf5)


- **Root Server (.)** — the top of the hierarchy. Knows the addresses of all TLD servers. When resolving `www.darkrelay.com`, resolution starts here.
- **TLD Server (Top Level Domain)** — handles extensions like `.com` and `.in`. Knows the authoritative servers for all second level domains registered under it.
- **Second Level Domain** — the actual registered domain name like `darkrelay`, `google`, or `amazon`. The authoritative server at this level holds records for all its subdomains.
- **Subdomain (Host)** — the leftmost part of the domain like `www`, `courses`, `mail`, or `kindle`. These are individual machines or services within the second level domain.

So for the query `www.darkrelay.com` (highlighted in green in the diagram):

```
. (root) → com (TLD) → (2nd level) → www (host)
```

Each level only knows about the level directly below it — no single server knows the full mapping.

### 5.2 Recursive Query

In a **recursive query**, the local DNS client passes the full responsibility of resolution to the **DNS resolver**. The resolver contacts each server in the hierarchy on behalf of the client and only returns when it has the final IP address. The client makes one request and waits — it has no involvement in the intermediate steps. This is simpler for the client but puts a heavier load on the DNS resolver since it must manage all the back-and-forth communication.

```
① Client → Local DNS Client        (query: www.google.com)
② Local DNS Client → DNS Resolver  (forward the query)
③ DNS Resolver → Root Name Server  (who handles .com?)
④ Root Name Server → TLD Name Server (who handles google.com?)
⑤ TLD Name Server → DNS Resolver   (here is the IP)
⑥ DNS Resolver → Local DNS Client  (resolved IP)
⑦ Local DNS Client → Client        (final IP answer)
```

### 5.3 Iterative Query

In an **iterative query**, the local DNS client does all the work itself. Each server it contacts does not resolve the query fully — it simply returns a **referral** pointing to the next server that might know the answer. The local DNS client must then contact that next server and repeat the process until it finally gets the IP address. This distributes the workload across the local DNS client rather than concentrating it on the resolver.

```
① Client → Local DNS Client          (query: www.google.com)
② Local DNS Client → DNS Resolver    (query)
③ DNS Resolver → Local DNS Client    (referral: ask Root server)
④ Local DNS Client → Root Name Server (query)
⑤ Root Name Server → Local DNS Client (referral: ask TLD server)
⑥ Local DNS Client → TLD Name Server  (query)
⑦ TLD Name Server → Local DNS Client  (here is the final IP)
⑧ Local DNS Client → Client           (final IP answer)
```

### 5.4 Recursive vs Iterative

| **S.N.** | **Recursive Query** | **Iterative Query** |
|---|---|---|
| 1 | DNS Resolver takes full responsibility and returns only the final IP to the local DNS client. | Local DNS client drives the entire process by following referrals from each server itself. |
| 2 | Client makes one request and waits for one final answer — no involvement in intermediate steps. | Local DNS client makes multiple requests, one to each level of the hierarchy. |
| 3 | More work is done by the DNS Resolver — higher load on it. | Work is distributed — local DNS client contacts each server independently. |
| 4 | Simpler for the client — only talks to local DNS once. | More complex for local DNS — must manage multiple ongoing queries. |

### 5.5 Caching

DNS servers cache resolved mappings for a period defined by **TTL (Time To Live)**. When the local DNS resolves `www.google.com`, it stores the result. The next client that asks for the same name gets the answer instantly from cache without contacting any other server. This dramatically reduces DNS traffic and speeds up resolution.
