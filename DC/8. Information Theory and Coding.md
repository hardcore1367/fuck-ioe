# Information Theory and Coding

## 1. Entropy

**Entropy (H)** is the measure of average information (uncertainty) in a source. It represents the theoretical minimum number of bits required per symbol.

$$H = \sum_{k=1}^{n} p_k \log_2 1/p_k \quad \text{bits/symbol}$$

**Average Code Length:**

$$\bar{L} = \sum_{k=1}^{n} p_k \cdot l_k \quad \text{bits/symbol}$$

**Efficiency:**

$$\eta = \frac{H}{\bar{L}} \times 100\%$$

Higher efficiency means the code is closer to the theoretical minimum. A perfect code has $\eta = 100\%$, meaning $\bar{L} = H$.


## 2. Source Coding
Source coding is a technique used in information theory to efficiently represent data for transmission, ensuring that the original information can be accurately reconstructed. Without source coding, transmission wastes bandwidth on redundant data.

### Necessity

- Removes redundancy from the source data.
- Reduces average bit rate → less bandwidth required.
- Makes transmission more efficient and storage more compact.


| Fixed Length | Variable Length |
|---|---|
| Same bits for all symbols | Shorter bits for frequent symbols |
| Simple, easy to decode | More complex decoding |
| Wasteful of bandwidth | Bandwidth efficient |
| Example: ASCII (8 bits) | Example: Huffman coding |

### 1. Binary Huffman Coding
Huffman coding is a lossless source coding technique that assigns shorter binary codes to more frequent symbols and longer codes to less frequent symbols, minimizing the average number of bits needed to represent a message. It builds an optimal prefix-free binary tree called the **Huffman tree**.

**Algorithm**
1. List all symbols with their probabilities.
2.  Sort symbols in **decreasing order** of probability.
3.  Take the **two lowest probability nodes**, merge them into a new node with probability equal to their sum.
4.  Insert the new node back and **re-sort**. (If the new probability is same as other, the new probability is given higher priority).
5.  Repeat steps 3–4 until only two branches remains.
6.  Assign **0 to top branch and 1 to the bottom** consistently throughout.
7.  Trace path from **root to each symbol** to get its codeword *and flip the codeword to get the actual one*.

### 2. Shannon-Fano Coding
Shannon-Fano coding is a lossless source coding technique that assigns shorter
codes to more probable symbols. Unlike Huffman which builds the tree bottom-up,
Shannon-Fano builds the tree **top-down** by recursively splitting symbols into
two groups of nearly equal probability.

**Algorithm**
1. List all symbols in **decreasing order** of probability.
2. Split the list into **two groups** such that the sum of probabilities of each
   group is as nearly equal as possible.
3. Assign **0 to the top group** and **1 to the bottom group**.
4. Recursively repeat steps 2–3 for each group until every group has only
   one symbol.
5. The codeword for each symbol is the sequence of 0s and 1s assigned along
   its path.

## 3. Error detection/correction

**Error Detection Only**

- **Parity Check** adds a single extra bit to data so the total number of 1s is either always even (even parity) or always odd (odd parity). If a single bit flips during transmission, the parity breaks and the error is detected — but it cannot locate or fix it.

- **Cyclic Redundancy Check (CRC)** treats the data as a binary polynomial and divides it by a agreed-upon **generator polynomial**. The remainder of this division is appended as a checksum. At the receiver, the same division is performed — a non-zero remainder means an error occurred.

- **Checksum** splits data into fixed-size blocks, adds them all together, and sends the sum along with the data. The receiver repeats the addition and compares — a mismatch signals an error. It's simple but weak against certain multi-bit errors.


**Error Correction**

- **Hamming Code** inserts redundant parity bits at specific power-of-2 positions within the data. Each parity bit covers a particular subset of bits, so when an error occurs, the pattern of failed parity checks points to the exact bit position that flipped, allowing it to be corrected.

- **Convolutional Codes** encode data continuously by passing it through a shift register with feedback, producing output that depends on both current and past input bits. Errors are corrected at the receiver using the **Viterbi algorithm**, which finds the most likely original sequence.


**Both Detection & Correction**

- **Linear Block Codes** are the general family where any valid codeword is a linear combination of other valid codewords. Hamming, BCH, and Reed-Solomon are all special cases. They are defined by their **(n, k)** structure — k data bits encoded into n-bit codewords.


## 4. Hamming Distance
**Hamming Distance** between two codewords is the number of bit positions in which they differ. It is a measure of how different two codewords are from each other.

$$d(x, y) = \text{number of positions where } x \text{ and } y \text{ differ}$$

XOR trick: XOR the two codewords and count the `1`s in the result.

$$10101 \oplus 11100 = 01001 \rightarrow d = 2$$


**Minimum Hamming Distance ($d_{min}$)** is the smallest Hamming distance between any two valid codewords in the entire codebook. It is the most important parameter of a code because it directly determines how many errors the code can **detect** and **correct**.

The idea is simple — when a codeword is transmitted and some bits flip due to noise, the received word moves "away" from the original codeword in Hamming space. If the minimum distance between any two valid codewords is large enough, the receiver can tell that an error occurred (detection) or even figure out which codeword was originally sent (correction).

$$d_{min} = \min_{x \neq y} \ d(x, y)$$

| Capability | Requirement |
|---|---|
| Detect up to $e$ errors | $d_{min} \geq e + 1$ |
| Correct up to $t$ errors | $d_{min} \geq 2t + 1$ |

A larger $d_{min}$ means the codewords are more spread apart, giving the code stronger error handling ability. For example, a code with $d_{min} = 3$ can detect up to 2 errors or correct up to 1 error.

## 5. Linear Block Coding

A **linear block code** is a channel coding scheme where a block of **k information bits** is encoded into a larger block of **n bits** called a **codeword**, with the extra **n − k bits** being redundancy (parity) bits. It is called *linear* because the XOR of any two valid codewords is also a valid codeword. These codes are denoted **(n, k)** and form the foundation of almost all practical error control systems.

- The **code rate R = k/n** measures efficiency — closer to 1 means less redundancy.
- The XOR of any two valid codewords always produces another valid codeword (**closure under addition**).
- The all-zero word is always a valid codeword.
- The **minimum Hamming distance d_min** equals the minimum weight (number of 1s) of any non-zero codeword, and it completely determines the error control capability of the code.

### Error Detection and Correction Capability

| Capability | Condition |
|---|---|
| Detect up to **e** errors | d_min ≥ e + 1 |
| Correct up to **t** errors | d_min ≥ 2t + 1 |
| Correct **t** and detect **e** errors (e > t) | d_min ≥ t + e + 1 |

For example, if d_min = 5, the code can correct up to **2 errors**, or detect up to **4 errors**, or correct 1 while detecting 3.


### Simple Parity Check Code

In a simple parity check code, one extra **parity bit** is added to a block of data bits to make the total count of 1s either always even (**even parity**) or always odd (**odd parity**). The receiver recounts the 1s in the received codeword — if the count violates the agreed parity, an error is flagged.

**What it does:** Detects any single-bit (or any odd number of) errors. Cannot detect even-numbered errors. Cannot correct any error — only detects.

**How it works:**
1. Count the number of 1s in the data bits.
2. If using even parity, set the parity bit to 0 if the count is already even, or 1 if it is odd — making the total even.
3. Append the parity bit to the data and transmit.
4. At the receiver, count all 1s including the parity bit. If the total is odd (for even parity), an error has occurred.

**Example:**

```
Data bits        :  1  0  1  1  0  0  1
Number of 1s     :  4  (even)
Parity bit (even):  0

Transmitted codeword:  1  0  1  1  0  0  1  |  0
                       ----------------------   ---
                           data bits           parity
```

Now suppose bit 3 flips during transmission:

```
Received  :  1  0  0  1  0  0  1  |  0
Number of 1s = 3  (odd → parity violated → ERROR DETECTED)
```

If two bits flip (say bit 1 and bit 3), the count of 1s stays even — **error not detected.** This is the key weakness.

### Two-Dimensional Parity Check (2D Parity)

**2D parity** arranges data bits into a rectangular grid of rows and columns. A parity bit is computed separately for each row and each column. All these parity bits are transmitted alongside the data. This catches burst errors and, for a single-bit error, it can not only detect but also **locate and correct** the error at the intersection of the failing row and column.

**What it does:** Detects all single-bit errors and most burst errors. Can correct one single-bit error by finding which row and column both fail — their intersection is the bad bit.

**How it works:**
1. Arrange the data bits into an m × n grid.
2. Compute a parity bit for each row (append to the right).
3. Compute a parity bit for each column (append at the bottom).
4. Transmit the entire grid including all row and column parity bits.
5. At the receiver, recompute parity for every row and column.
6. If exactly one row and one column fail, the error is at their intersection — flip that bit.

**Example:**

Data bits arranged in a 3 × 4 grid (even parity):

```
           C1   C2   C3   C4   |  Row Parity
  R1        1    0    1    1   |      1
  R2        0    1    0    1   |      0
  R3        1    1    0    0   |      0
           --   --   --   --
Col Parity  0    0    1    0
```

Now suppose the bit at **R1, C3** flips from 1 → 0 during transmission:

```
Received:
           C1   C2   C3   C4   |  Row Parity
  R1        1    0    0    1   |      0   ← should be 1, FAIL
  R2        0    1    0    1   |      0   ← OK
  R3        1    1    0    0   |      0   ← OK
           --   --   --   --
Col Parity  0    0    1    0
Recomputed  0    0    0    0
                      ↑
                    C3 FAIL
```

Row 1 fails and Column 3 fails → error is at **(R1, C3)** → flip it back to 1 → **corrected.**

### Hamming Code

**Hamming code** is a linear error-correcting code that inserts parity bits at specific positions within the codeword — at every **power-of-2 position** (1, 2, 4, 8, …). Each parity bit is responsible for checking a particular subset of bit positions. When an error occurs anywhere in the received word, the pattern of failed parity checks forms a binary number that directly gives the **position of the erroneous bit**, which is then flipped to correct it.

**What it does:** Detects and corrects any **single-bit error**. Can also detect (but not correct) double-bit errors in an extended version.

**How it works (step by step):**

**Step 1 — Determine number of parity bits.**
For m data bits, find the smallest r such that:
> **2ʳ ≥ m + r + 1**

For m = 4 data bits: try r = 3 → 2³ = 8 ≥ 4 + 3 + 1 = 8 ✓ → so r = 3, total codeword length n = 7.
This gives a **(7, 4) Hamming code.**

**Step 2 — Place parity and data bits.**
Parity bits occupy positions 1, 2, 4 (powers of 2). Data bits fill the rest.

```
Position :   1    2    3    4    5    6    7
Bit type :   P1   P2   D1   P4   D2   D3   D4
```

Data bits = `1  0  1  1` placed at positions 3, 5, 6, 7:

```
Position :   1    2    3    4    5    6    7
            P1   P2    1   P4    0    1    1
```

**Step 3 — Calculate each parity bit.**

Each parity bit Pi covers all positions whose binary representation has a 1 in the same bit position as i.

- **P1** (position 1 = 001) covers positions with bit-0 set: **1, 3, 5, 7**
  → bits: P1, 1, 0, 1 → for even parity, P1 must make count even → count of known = 2 (even) → **P1 = 0**

- **P2** (position 2 = 010) covers positions with bit-1 set: **2, 3, 6, 7**
  → bits: P2, 1, 1, 1 → count of known = 3 (odd) → **P2 = 1**

- **P4** (position 4 = 100) covers positions with bit-2 set: **4, 5, 6, 7**
  → bits: P4, 0, 1, 1 → count of known = 2 (even) → **P4 = 0**

```
Final codeword:

Position :   1    2    3    4    5    6    7
            P1   P2   D1   P4   D2   D3   D4
             0    1    1    0    0    1    1
             ↑    ↑         ↑
          parity bits    parity bit       data bits
```

**Step 4 — Error detection and correction at receiver.**

Suppose the received word is `0 1 1 0 1 1 1` (bit at position 5 flipped from 0 → 1).

Recompute each parity group:

```
P1 checks positions 1, 3, 5, 7  →  0, 1, 1, 1  →  count = 3 (odd)  →  FAIL (1)
P2 checks positions 2, 3, 6, 7  →  1, 1, 1, 1  →  count = 4 (even) →  PASS (0)
P4 checks positions 4, 5, 6, 7  →  0, 1, 1, 1  →  count = 3 (odd)  →  FAIL (1)
```

Write the fail/pass results as a binary number with P4 as MSB and P1 as LSB:

```
P4  P2  P1
 1   0   1   =  5  (in decimal)
```

Error is at **position 5** → flip bit 5 from 1 → 0 → **codeword restored to `0110011`.**
