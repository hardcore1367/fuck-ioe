# 7. Project Integration Management

**Project Integration Management** is a collection of processes and activities needed to identify, define, combine, unify, and coordinate the various processes and project management activities within the project management process groups. These actions are applied from the start of the project through to its completion, and the project manager must coordinate all nine knowledge areas throughout the project's life cycle using dedicated integration processes, each of which has its own inputs, tools and techniques, and outputs. Integration management exists to manage change and communication, reduce project time and cost, involve stakeholders early and often, make results visible, identify problems and solutions early, and make use of relevant experience as early as possible in the project.

## 7.1 Project Charter

Developing the **project charter** is part of the project initiation phase, and the charter authorizes the project manager to start the project and to use organizational resources as required. The project charter serves as an agreement or contract between the project sponsor and the project team, documenting the project formally. A project charter should address the project's scope, its objectives and deliverables, the project team members, the project risks, the expected benefits or return on investment, the budget, and the business case.

### 7.1.1 Inputs to the Project Charter

- **Agreements or contracts**, which should already include much of the information needed to build a good project charter.
- **Enterprise Environmental Factors (EEF)**, which managers should review when developing a charter, and which include culture and infrastructure, tools, human resources, personnel policies, and marketplace conditions.
- **Organizational Process Assets (OPA)**, which include the formal and informal plans, policies, procedures, guidelines, information systems, financial systems, management systems, lessons learned, and historical information that can influence a project's success.
- The **Statement of Work (SOW)**, a legally binding document that captures and defines all the work management aspects of the project, and which itself includes the business need, the product description, and the strategic plan; organizations often carry out a **SWOT analysis** (identifying strengths, weaknesses, opportunities, and threats) as part of building this strategic plan.

### 7.1.2 Tools and Techniques to Develop the Project Charter

- **Project selection methods** help an organization choose which projects to pursue, and include focusing on broad organizational needs by checking whether the project has genuine need, secure funding, and strong will behind it; categorizing IT projects as addressing a problem, an opportunity, or a directive; performing financial analyses such as Net Present Value (NPV), Return on Investment (ROI), and payback analysis; using a weighted scoring model; and referring to a balanced scorecard.
- **Project management methodology** is a set of project management process groups whose related control functions are merged and combined into a single, cohesive working system.
- **Project Management Information System (PMIS)** refers to enterprise project management software, often accessible over the internet and tied into other systems such as financial systems, that helps managers and teams create schedules, track progress, and generate artifacts like Gantt charts; PMIS plays an important role in integration management by giving the project manager one consolidated system through which information from every knowledge area can be tracked and coordinated.
- **Expert judgment** supports charter development by letting the project manager consult people with specialized knowledge or experience on questions such as which methodology to follow, which programming language to use, and which training approach to adopt, so that the charter is built on informed decisions rather than guesswork.

## 7.2 Preliminary Project Scope Statement

Developing the preliminary project scope statement is an output of the project initiation process group, and its purpose is to identify the high-level project objectives, which must be clear, actionable, and measurable. It typically includes the project objectives, constraints and assumptions, the product requirements, an initial view of risk, a preliminary estimated cost, and the approval requirements for the project.

## 7.3 Project Management Plan

Developing the project management plan is part of the planning process group, and its primary use is to document assumptions and decisions, facilitate communication among stakeholders, and record what has been formally approved. It typically includes the project charter, the project management approach, the scope statement, and the budget, schedule, and resources, along with common elements such as an introduction and overview of the project, the project organization, the management and technical processes to be used, the scope of work to be performed, schedule and budget information, and references to other project planning documents.

### 7.3.1 Project Planning Matrix (PPM)

The **Project Planning Matrix (PPM)**, also called a logical framework, is a tool used in project planning to develop a well-designed, objectively described, and evaluable project, and it provides a summary of the project's design documents. Its characteristics are that project logic moves vertically from inputs up to goals, horizontal logic moves from the narrative summary across to the important assumptions, all items within a cell should be linked through a proven, non-causal relationship, and a better cause-and-effect linkage between cell items produces a better project design overall. The matrix itself is organized as a grid with **Goal**, **Purpose**, **Outputs**, and **Activities** running down one side, and **Narrative Summary**, **Objectively Verifiable Indicators (OVI)**, **Means of Verification (MOV)**, and **Assumptions/Risk** running across the other.

### 7.3.2 Stakeholder Analysis

**Stakeholder analysis** is the process of identifying the individuals or groups who are likely to affect, or be affected by, the proposed system, and sorting them according to their impact. A stakeholder analysis document usually records the stakeholder's name and organization, their role on the project, unique facts about them, their level of influence and interest in the project, and suggestions for managing the relationship along with the nature of reporting expected. Stakeholders are grouped into three types: **primary stakeholders**, who are directly involved in the monetary transactions of the organization; **secondary stakeholders**, who are invested in the social transactions of the organization; and **key stakeholders**, who have significant influence within the organization and help it make strategic decisions and minimize risk.

### 7.3.3 Baseline Management

A **baseline** is a fixed reference point used to measure and compare a project's progress against, allowing the team to assess performance over time. A project baseline typically covers the schedule baseline, the cost baseline, the scope baseline, and the quality baseline. A baseline project plan is developed by setting measures, recording progress, calculating the variance between planned and actual performance, and taking remedial action where needed, and this is what makes baseline management genuinely useful when developing a project management plan, since it gives the plan a fixed point to be checked against as the project moves forward.

## 7.4 Direct and Manage Project Execution

Directing and managing project execution is part of the executing process group, and it means managing and performing the tasks described in the project management plan; project planning and project execution are related and inseparable activities, since execution is simply the plan being carried out in practice. **Work performance information**, generated during execution, includes schedule progress, the status of completed, pending, and incomplete deliverables, the extent to which quality standards are being met, resource utilization details, lessons learned, and any conflicting issues that need to be addressed.

## 7.5 Monitor and Control Project Work

Monitoring and controlling project work is part of the monitoring and controlling process group. Since changes are mostly unavoidable during a project, the better approach is to develop and follow a defined monitoring and control strategy rather than reacting to changes as they appear. Monitoring project work includes collecting, measuring, and circulating performance information, with the overall goal of watching all project activities to ensure the project continues to meet its performance objectives.

## 7.6 Integrated Change Control

Performing integrated change control is part of the monitoring and controlling process group, and its objective is to influence the factors that create changes so that changes remain beneficial, to determine when a change has actually occurred, and to manage those changes properly as they occur. Integrated change control is carried out through four types of control.

- A **change control system** is a formal, documented process describing when and how official project documents and work may be changed, and who is authorized to make those changes and how.
- A **Change Control Board (CCB)** is a formal group of people responsible for approving or rejecting changes to a project, and it provides guidelines for preparing change requests, evaluating those requests, and managing the implementation of any approved changes.
- **Configuration management** ensures that a project's product descriptions are correct, complete, and consistent, by identifying and controlling the functional and physical design characteristics of the product and its documentation.
- **Version control and build management** covers the practice of tracking and managing changes to software code over time, using version control systems that maintain history and support rollback, alongside a formal check-in/check-out policy to guarantee code quality during builds.

## 7.7 Close Project

Closing a project is part of the closing process group, and to close a project or phase, all activities must be finalized and the completed or cancelled work transferred to the appropriate people; the main outputs are the transition of the final product, service, or result, along with updates to the organizational process assets. Closing a project follows four main steps.

1. **Phase-end activities** help the project team transition from one phase to the next, and include peer review, release management, review meetings, updating the project plan for the next phase, status reporting, and end-user education.
2. **Administrative closure procedure** ensures the project plan has proper budget and time set aside for closure, and populates the project archives with formal client acceptance, recommendation support from sponsors, maintenance and upgrade plans, training records, and lessons learned.
3. **Contract closure procedure** verifies that the contract requirements have been met, obtains formal acceptance from the client, evaluates the performance of both the buyer and the seller under the contract, carries out a procurement audit, and updates the project files and historical databases.
4. **Updated organizational process assets** are produced as a result, including formal acceptance documentation, project files, project closure documentation, and finally archiving, which preserves historical information and lessons learned for future projects.

## 7.8 Project Scope Management

**Scope** refers to all the work involved in creating the products of a project, along with the processes used to create them, and **project scope management** includes the processes involved in defining and controlling what is and is not included in a project. The project team and its stakeholders must share the same understanding of what will be produced and how, since a mismatch here is one of the most common causes of project difficulty later on.

### 7.8.1 Processes in Project Scope Management

1. **Initiation** begins a project or moves it into its next phase.
2. **Scope planning** develops the documents that provide the basis for future project decisions.
3. **Defining scope** reviews the project charter, requirement documents, and organizational process assets to create a scope statement.
4. **Creating the WBS** subdivides the major project deliverables into smaller, more manageable components.
5. **Scope verification** formalizes acceptance of the completed project scope.
6. **Scope change control** controls any changes made to the project scope.

### 7.8.2 Project Scope vs Product Scope

| S.N. | Project Scope | Product Scope |
|---|---|---|
| 1 | Is the work that is required to deliver the product. | Is the sum of all features, functions, and characteristics of the product. |
| 2 | Focuses on the work needed to deliver the product. | Focuses on the features of the end product. |
| 3 | Is a subset of the product scope. | Is a superset of the project scope. |
| 4 | Is oriented toward the "how" (work-related). | Is oriented toward the "what" (functional requirements). |
| 5 | Example: constructing a bridge. | Example: the technical specifications of the bridge, such as its length and width. |

## 7.9 Work Breakdown Structure (WBS)

A **Work Breakdown Structure (WBS)** is a process of systematically breaking down the major work of a project into smaller, more manageable components and subcomponents arranged in a hierarchical order. A **work package** is a task at the lowest level of the WBS, and it is often represented using a Gantt chart.

### 7.9.1 Advantages of WBS

- Planning can be performed more easily once the work is broken down.
- Cost and budgets can be established for each smaller component.
- Schedules can be established at the work-package level.
- Responsibility can be assigned to each individual activity.
- The risk of each activity can be analyzed separately.

### 7.9.2 What Creating a WBS Identifies

- It defines the work itself (**what** needs to be done).
- It identifies the time needed to complete each work package (**how long**).
- It identifies a time-phased budget for each work package (**cost**).
- It identifies the resources needed to complete each work package (**how much**).
- It identifies a person responsible for each unit of work (**who**).
- It identifies monitoring points, or milestones, for measuring success.

### 7.9.3 Levels of WBS

Total Program, Project, Task (Activity), Sub-task, Work Package, and Level of Effort, arranged from the broadest level down to the most detailed.

### 7.9.4 Approaches to Developing a WBS

- **Using guidelines** provided by the organization for preparing WBSs.
- **The analogy approach**, which reviews the WBS of a similar past project and customizes it for the current one.
- **The top-down approach**, which starts with the largest items of the project and breaks them down further.
- **The bottom-up approach**, which starts with the specific tasks and rolls them up into larger groupings.
- **The mind-mapping approach**, which uses branches radiating out from a core idea to structure thoughts and ideas.

An **Organizational Breakdown Structure (OBS)** is a related type of organizational chart that links cost, activity, and responsibility together, and it allows work definition, the assignment of owners to work packages, and the assignment of budgets to departments.

### 7.9.5 Principles for Creating a WBS

1. A unit of work should appear only once in the WBS.
2. The work content of a WBS item is the sum of the WBS items below it.
3. A WBS item is the responsibility of only one individual, even though many people may work on it.
4. The WBS must be consistent with the way the work is actually going to be performed.
5. Project team members should be involved in developing the WBS to ensure consistency.
6. Every WBS item should be documented in a WBS dictionary.

The **100% rule** is a core characteristic of the WBS, stating that the WBS must include 100% of the work defined by the project scope and capture all deliverables, whether internal, external, or interim, including project management itself; this is one of the most important principles guiding how a WBS is developed, decomposed, and evaluated. The **8/80 rule** states that the lowest level of work in a WBS should take no less than 8 hours and no more than 80 hours to complete.

## 7.10 Responsibility Assignment Matrix (RAM)

A **Responsibility Assignment Matrix (RAM)**, also known as a **RACI matrix**, maps the work of the project to the people responsible for performing it, with the work itself described in the WBS. The letters in RACI stand for **Responsible** (who does the task), **Accountable** (who signs off on the task or holds authority over it), **Consulted** (who has the information necessary to complete the task), and **Informed** (who needs to be notified of the task's status and results). For smaller projects, it is best to assign individual people to WBS activities, while for very large projects, it is more effective to assign the work to organizational units or teams instead.

## 7.11 Scope Verification

**Scope verification** is the process of formally accepting the finished outcomes of a project. It includes obtaining formal acceptance of the project scope from stakeholders such as the sponsor, client, or customer, reviewing outcomes and work results to confirm they were completed correctly and satisfactorily, and determining the level of completion, particularly if the project has been terminated early. Its inputs include the project scope statement, the WBS dictionary, the project scope management plan, and the actual outcomes of the project. Its main tool and technique is **inspection**, which includes measuring, examining, and verifying that the work and its results meet the stated requirements and product acceptance criteria, and which is also known as a product review, review, audit, or walkthrough. Its outputs are the accepted results, requested changes, and recommended corrective actions.

## 7.12 Scope Control

**Scope control** is a key project management process that involves controlling changes to the project scope, with the goals of influencing the factors that cause scope changes, guaranteeing that any changes are processed according to established procedures, and managing changes properly once they occur; **variance** here refers to the difference between planned and actual performance. To avoid scope problems, a project should keep its scope realistic, involve users directly in scope management, use off-the-shelf hardware and software wherever possible, and follow good project management processes throughout.

**Scope creep** refers to uncontrolled growth in a project's scope, usually caused by poor change control, a lack of proper initial identification or documentation of the features required to meet project objectives, or a weak project manager or executive sponsor. It can be reduced by clearly defining requirements as "must haves" versus "nice to haves," setting clear project expectations, agreeing on and documenting deliverables, and documenting and reviewing business requirements regularly. Poor project scope management is, in fact, one of the key reasons IT projects fail, which is why good scope management depends on strong user involvement, executive support, a clear statement of requirements, and a defined process for managing scope changes.
