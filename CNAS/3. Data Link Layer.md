# 3. Data Link Layer


The **Data Link Layer (DLL)** is the second layer of the OSI model, responsible for node-to-node delivery of frames over a single physical link. It takes raw bits received from the Physical Layer, packages them into frames, and provides reliable, error-controlled, and flow-controlled transmission to the Network Layer above it. Examples: **Ethernet**, **HDLC**.

![Data Link Layer in OSI](https://github.com/user-attachments/assets/add6a1f9-cf5f-40e1-b8fd-1024cb0facf1)


## 1.1. Functions

- **Framing** — divides the raw bit stream into discrete, identifiable units called frames.
- **Error Detection and Correction** — detects (and sometimes corrects) errors introduced during transmission using CRC, Hamming code, etc.
- **Flow Control** — prevents a fast sender from overwhelming a slow receiver.
- **Error Control** — ensures lost or damaged frames are retransmitted using ARQ mechanisms.
- **Access Control** — decides which device may use the shared channel at a given time (via MAC sub-layer).
- **Piggybacking** — acknowledgements are bundled into outgoing data frames to reduce overhead.

## 1.2. Services Provided to Network Layer

### 1. Unacknowledged Connectionless Service
No connection is established before sending, and the receiver does not send any acknowledgement back. Lost or corrupted frames are not retransmitted by the DLL. This service is used on highly reliable channels like Ethernet LAN, where errors are rare and upper layers can handle recovery.

### 2. Acknowledged Connectionless Service
No connection is established, but every sent frame must be individually acknowledged by the receiver. If the sender does not receive an acknowledgement within a timeout period, it retransmits the frame. This is suited for unreliable channels like wireless networks (e.g., IEEE 802.11 Wi-Fi).

### 3. Acknowledged Connection-Oriented Service
A connection is explicitly established before transmission, each frame is acknowledged during transfer, and the connection is released afterward. This guarantees every frame is received exactly once and in the correct order. It is used where reliability is critical, such as satellite links or WAN connections using HDLC.

## 1.3. Design Issues

- **Services to the Network Layer** — the DLL must expose a clean interface so the Network Layer is shielded from physical transmission details.
- **Framing** — delimiting the start and end of frames correctly; poor design causes synchronization loss.
- **Error Control** — choosing whether to only detect errors or also correct them, and selecting the appropriate ARQ scheme.
- **Flow Control** — balancing throughput without causing receiver buffer overflow.
- **Shared Medium Access** — arbitrating channel use among multiple devices, handled by the MAC sub-layer.

## 2. Framing

**Framing** is the process by which the Data Link Layer divides a continuous stream of raw bits received from the Physical Layer into discrete, manageable units called **frames**. Framing allows the receiver to identify where one frame ends and the next begins, which is essential for error control and flow control to work correctly.

A **frame** is the basic unit of data transmission at the Data Link Layer. It consists of the actual data (payload) along with control information added by the DLL.

### Structure of a Frame

```
| Header | Payload (Data) | Trailer |
```

- **Header** — contains source and destination addresses of frame.
- **Payload** — carries the actual data passed down from the Network Layer.
- **Trailer** — contains error detection/correction bits (e.g., CRC) and marks the end of the frame.

### 2. Framing Methods

### 2.1 Character Count

In character count framing, the header of each frame contains a field that specifies the total number of characters (bytes) in that frame, including the count field itself. The receiver reads this count to determine where the current frame ends and where the next frame begins.

**Structure:**

```
| Count | Data (payload) |
```

- **Count field** — the first byte of the frame, specifies the total number of bytes in the frame including itself.
- **Data field** — the actual payload bytes that follow.

**Example:**

```
Frame 1: | 5 | A | B | C | D |
Frame 2: | 4 | E | F | G |
```

Frame 1 has count 5, meaning 5 bytes total (1 count + 4 data). Frame 2 has count 4, meaning 4 bytes total.

**Problem:**

If the count field gets corrupted during transmission, the receiver reads a wrong count value, picks up the wrong number of bytes, and loses synchronization with all subsequent frames. There is no way to recover from this without higher-level intervention.


### 2.2 Byte Stuffing (Character Stuffing)

In byte stuffing, special reserved bytes are used to mark the boundaries of a frame. A **FLAG** byte marks the start and end of every frame. If the same FLAG byte pattern appears inside the data, the sender inserts an **ESC** (escape) byte before it so the receiver can distinguish it from a real frame delimiter. The receiver strips out all ESC bytes before passing the data up to the Network Layer.

**Structure:**

```
| FLAG | Data (with ESC if needed) | FLAG |
```

**Rules:**
- If FLAG appears in data → sender inserts ESC before it → `ESC FLAG`
- If ESC appears in data → sender inserts ESC before it → `ESC ESC`
- Receiver sees ESC → removes it and treats the next byte as plain data.

**Example:**

```
Original data : | A | FLAG | B | ESC | C |
Transmitted   : | FLAG | A | ESC FLAG | B | ESC ESC | C | FLAG |
Received data : | A | FLAG | B | ESC | C |
```

**Problem:**

Byte stuffing only works with byte-oriented (character-based) data. It also increases frame size whenever FLAG or ESC bytes appear frequently in the data.


### 2.3 Bit Stuffing

In bit stuffing, frame boundaries are marked using a special flag pattern **01111110** (six consecutive 1s between a 0 on each side). If the sender's data contains five consecutive 1s, it automatically inserts a `0` bit after them to prevent the data from being mistaken for the flag pattern. The receiver, upon seeing five consecutive 1s followed by a `0`, removes the stuffed `0` and reconstructs the original data. This is used in protocols like **HDLC**.

**Structure:**

```
| 01111110 | Data (with stuffed 0s if needed) | 01111110 |
```

**Rules:**
- Sender sees five consecutive 1s in data → inserts a `0` after them.
- Receiver sees five consecutive 1s followed by `0` → removes the `0`.
- Receiver sees five consecutive 1s followed by `1` → it is the flag, frame ends.

**Example:**

```
Original data : 0 1 1 1 1 1 1 0
Transmitted   : 0 1 1 1 1 1 0 1 0  (0 stuffed after five 1s)
Received data : 0 1 1 1 1 1 1 0    (stuffed 0 removed)
```

**Advantage over Byte Stuffing:**

Bit stuffing works at the bit level and is not restricted to byte-oriented data, making it more flexible and widely applicable.


## 3. Flow Control 

**Flow Control** is a mechanism used by the Data Link Layer to prevent the sender from transmitting frames faster than the receiver can process them. Without flow control, a fast sender can overwhelm a slow receiver's buffer, causing frames to be dropped. There are two main flow control mechanisms.

![Types of Flow Control](https://github.com/user-attachments/assets/354f68d7-5269-48e0-9e1a-7821042f8312)

### 1. Stop-and-Wait Flow Control

The sender transmits one frame and then completely stops and waits for the receiver to send an acknowledgement (ACK) before transmitting the next frame. The receiver processes the incoming frame, stores it in its buffer, and sends an ACK back only when it is ready to accept the next frame. This way, the receiver directly controls how fast the sender transmits — if the receiver is slow, it simply delays sending the ACK, and the sender waits. No frame is ever sent unless the receiver has explicitly signaled that it is ready.

**Problem:** Only one frame is in transit at any time. During the entire round-trip time (frame travel + processing + ACK travel), the sender sits completely idle. On high-bandwidth or high-latency links, this idle time dominates and channel utilization drops severely, making Stop-and-Wait highly inefficient for any serious data transfer.

### 2. Sliding Window Flow Control

The sliding window mechanism allows the sender to transmit multiple frames continuously without waiting for individual acknowledgements, keeping the channel busy at all times. Both the sender and receiver maintain a **window** — a fixed-size range representing how many frames can be in transit or buffered at any given time.

The sender can keep transmitting new frames as long as its window is not full. Each time an ACK is received, the sender's window **slides forward**, removing the acknowledged frame and allowing one new frame to be sent. If the window fills up completely with no ACKs returned, the sender pauses until an ACK arrives.

The window size determines the maximum number of unacknowledged frames in transit at any moment. A larger window keeps more frames in the pipeline, which is critical on high-speed or high-latency links where round-trip time is significant.

![Sliding Window Flow Control](https://github.com/user-attachments/assets/f97fd3ec-39c5-4dc3-88f5-4472c1f94936)

**Example:** If window size is 4 and frames 0, 1, 2, 3 are sent, the sender waits. When ACK 0 arrives, the window slides and frame 4 is sent. When ACK 1 arrives, frame 5 is sent, and so on.

**Go-Back-N** and **Selective Repeat** are both implementations of the sliding window mechanism, differing in how they handle lost frames.

### Piggybacking

**Piggybacking** is an optimization technique where instead of sending a separate standalone ACK frame for every received data frame, the receiver waits for its own outgoing data frame and attaches the acknowledgement onto it. This combines ACK and data into a single frame, reducing overhead and saving bandwidth.

For example, when node A sends a data frame to node B, node B needs to send an ACK back. But if node B also has data to send to node A, instead of sending two separate frames (one ACK + one data), it bundles the ACK field inside its outgoing data frame and sends just one frame. Node A receives both the data and the acknowledgement in a single transmission.

**How it is applied in DLL flow control:**

In sliding window flow control, every received frame must be acknowledged. Without piggybacking, the receiver would send a separate ACK frame for each incoming data frame, consuming bandwidth in both directions just for control traffic. With piggybacking, the ACK is carried inside the next outgoing data frame, so the channel is used more efficiently. However, the receiver cannot wait too long for an outgoing data frame to attach the ACK to — if no data frame is ready within a short timeout, the receiver must send a standalone ACK immediately to avoid causing the sender to time out and retransmit unnecessarily.


## 4. Error Control
**Error Control** is the mechanism used by the Data Link Layer to detect and recover from frames that are lost or corrupted during transmission. It is implemented using **ARQ (Automatic Repeat reQuest)** protocols, where the receiver sends acknowledgements and the sender retransmits frames that are not acknowledged within a timeout period. 

There are three main ARQ protocols.

### 1. Stop-and-Wait ARQ

The sender transmits one frame and then completely stops and waits for an acknowledgement (ACK) before sending the next frame. If the ACK is received within the timeout period, the sender moves on and transmits the next frame. If no ACK arrives within the timeout, the sender assumes the frame was lost and retransmits the same frame.

![Stop-and-Wait ARQ](https://github.com/user-attachments/assets/ff4c9608-c4aa-408d-ab27-e680a1d1d863)

**Edge cases:**
- If the frame arrives correctly but the ACK is lost, the sender times out and retransmits the same frame. The receiver gets a duplicate frame. To handle this, frames are numbered with a 1-bit sequence number (0 and 1 alternating), so the receiver can detect and discard duplicates.
- If the ACK arrives after the timeout (delayed ACK), the sender has already retransmitted. The late ACK must be ignored.

**Problem:** The sender is completely idle during the entire round-trip time waiting for the ACK. This wastes channel capacity and makes Stop-and-Wait very inefficient over high-latency links.

### 2. Go-Back-N ARQ

To fix the inefficiency of Stop-and-Wait, Go-Back-N allows the sender to transmit multiple frames without waiting for individual ACKs. The sender maintains a **sending window** of size N, meaning it can have up to N unacknowledged frames in transit at any time. The receiver maintains a window of size 1, meaning it only accepts frames in strict sequential order.

If a frame is lost or corrupted, the receiver discards that frame and also discards all subsequent frames that arrive after it, even if those frames were received correctly. The receiver sends a **NAK (Negative Acknowledgement)** or simply stops ACKing. The sender, upon timeout or receiving a NAK, must go back and retransmit the lost frame and every frame that was sent after it.

![Go-Back-N ARQ](https://github.com/user-attachments/assets/f6290d61-6330-4262-a1a2-f0584ae3eec6)

**Edge cases:**
- If frame 4 is lost but frames 5, 6, 7 arrive correctly, the receiver discards 5, 6, and 7. The sender retransmits 4, 5, 6, and 7 — wasting bandwidth even though 5, 6, 7 were fine.
- The maximum window size is **2^n - 1** (where n is the number of sequence number bits). If window size equals 2^n, the receiver cannot distinguish between a new frame and a retransmitted old frame after a full window cycle.

**Sender window = N, Receiver window = 1.**

### 3. Selective Repeat ARQ

Selective Repeat solves the bandwidth waste problem of Go-Back-N. Both the sender and receiver maintain a window of size N. When a frame is lost, only that specific lost frame is retransmitted — all correctly received frames after it are buffered at the receiver until the missing frame arrives and the complete sequence can be reassembled and passed up to the Network Layer.

This is the most efficient ARQ protocol in terms of bandwidth usage, but it requires more memory and processing at the receiver since it must buffer out-of-order frames and track which frames have been received and which are still missing.

![Selective Repeat ARQ](https://github.com/user-attachments/assets/7a5fe259-ef3b-4076-b128-8d203ed7bcc8)

**Edge cases:**
- The maximum window size is **2^(n-1)** (half of 2^n), strictly smaller than in Go-Back-N. If the window were larger, the receiver could confuse a retransmitted old frame with a new frame because their sequence numbers would overlap.
- If both frame 4 and frame 7 are lost, only frames 4 and 7 are retransmitted individually. Frames 5 and 6 (received correctly) stay in the buffer.

**Sender window = N, Receiver window = N.**

## 6. HDLC
**HDLC (High-Level Data Link Control)** is a bit-oriented, synchronous data link protocol developed by ISO for reliable data transmission over both point-to-point and multipoint links. It operates at the Data Link Layer and uses **bit stuffing** for framing — the flag pattern `01111110` marks frame boundaries. HDLC forms the basis for many modern protocols including PPP and X.25.

### 1. Modes of Operation

- **Normal Response Mode (NRM)** — used in point-to-multipoint links. The primary station controls communication; secondary stations can only transmit when explicitly given permission by the primary.
- **Asynchronous Balanced Mode (ABM)** — used in point-to-point links. Both stations are equal (called combined stations) and either can initiate transmission at any time without permission. This is the most commonly used mode.
- **Asynchronous Response Mode (ARM)** — secondary station can initiate transmission without permission from primary, but primary still controls the link. Rarely used.

### 2. Frame Format

```
| Flag | Address | Control | Data | FCS | Flag |
```

**Starting Flag (1 byte)**
The frame begins with the fixed bit pattern `01111110`. This marks the start of the frame and allows the receiver to synchronize. Bit stuffing ensures this pattern never appears inside the frame data.

**Header — Address + Control (2-3 bytes)**
- **Address field (1 or more bytes)** — identifies the secondary station involved in the transmission, either as the destination (for commands) or the source (for responses).
- **Control field (1 or 2 bytes)** — specifies the frame type (I, S, or U) and carries sequence numbers for flow and error control.

**Body — Data field (variable)**
Carries the actual payload passed down from the Network Layer. Present only in I-frames; S-frames and U-frames carry no data.

**FCS — Frame Check Sequence (2 or 4 bytes)**
Contains a **CRC** value computed over the Address, Control, and Data fields. The receiver recomputes the CRC on arrival and compares it with the FCS field to detect any transmission errors.

**Ending Flag (1 byte)**
The frame ends with the same `01111110` flag pattern, marking the frame boundary. The ending flag of one frame can serve as the starting flag of the next.


## 7. MAC Sublayer

The **MAC (Medium Access Control) sublayer** is the lower part of the Data Link Layer, sitting just above the Physical Layer. Its primary responsibility is to control how multiple devices on the same network share a single broadcast channel. When only two devices are connected point-to-point, there is no contention and no need for MAC — but in shared medium networks (like Ethernet LAN or Wi-Fi), many devices compete for the same channel simultaneously. The MAC sublayer defines the rules and protocols that govern who gets to transmit, when, and for how long, preventing simultaneous transmissions from colliding and corrupting each other. It also handles physical addressing using **MAC addresses**, which are hardware-level addresses that uniquely identify every network interface on a local network.

### 1. Channel Allocation Problem

When multiple stations share a single broadcast channel, the central problem is how to allocate the channel among them fairly and efficiently.

### 1.1 Static Allocation
The channel is divided in advance and each station is assigned a fixed portion, regardless of whether it has data to send or not. Done using **FDM (Frequency Division Multiplexing)** or **TDM (Time Division Multiplexing)**.

**Problem:** If a station has nothing to send, its allocated slot is wasted. If a station suddenly has a burst of data, it cannot use more than its fixed share. This makes static allocation highly inefficient for bursty, unpredictable traffic like computer networks.

### 1.2 Dynamic Allocation
Channel is allocated on demand — a station gets access only when it actually has data to send. This makes much better use of channel capacity for bursty traffic. Dynamic allocation is the basis for all multiple access protocols.

**Key assumptions for dynamic allocation:**
- Any station can transmit at any time.
- If two stations transmit simultaneously, a **collision** occurs and both frames are lost.
- Collisions can be detected and recovery can be attempted.


## 8. Multiple Access Protocals

Multiple access protocols define the rules by which stations dynamically share the channel.

![Multiple Access Protocals Types](https://github.com/user-attachments/assets/e07bee19-b8b9-4969-8417-b393edc1bfde)


### 1. ALOHA
**ALOHA** is one of the earliest and simplest multiple access protocols, originally developed for satellite communication. The core idea is straightforward — any station that has data to send simply transmits it. If a collision occurs, the station waits for a random amount of time and retransmits. There are two variants.

### 1.1. Pure ALOHA

Any station can transmit a frame at any time without checking whether the channel is busy. If two or more stations transmit at the same time, a collision occurs and all colliding frames are destroyed. Each station then waits for a random backoff time and retransmits. The randomness is important — if all stations waited the same amount of time, they would collide again.

**Vulnerable period:** In Pure ALOHA, a frame can be destroyed if another station begins transmitting any time within one full frame transmission time before or after the current transmission. This gives a vulnerable period of **2 × frame transmission time (2T)**, making collisions very likely and channel efficiency very low.

**Condition for no collision:** A station's frame will be received without collision only if no other station transmits during the entire vulnerable period of 2T around that transmission.

### 1.2. Slotted ALOHA

Slotted ALOHA improves efficiency by dividing time into discrete slots, each exactly one frame transmission time long. Stations are only allowed to begin transmitting at the start of a slot — never in the middle. This synchronization means two frames either collide completely (both started at the same slot) or not at all.

**Vulnerable period:** Because transmissions are aligned to slot boundaries, the vulnerable period is reduced to **T (one frame transmission time)** — exactly half that of Pure ALOHA. This doubles the maximum channel efficiency.

**Condition for no collision:** A station's frame will be received without collision only if no other station transmits in the same slot.

| Pure ALOHA | Slotted ALOHA |
|---|---|
| Stations can transmit at any time without waiting for a slot boundary. | Stations can only transmit at the beginning of a fixed time slot. |
| Vulnerable period is 2T — a frame can be destroyed if another station transmits within one full frame time before or after. | Vulnerable period is T — a frame can only be destroyed if another station transmits in the exact same slot. |
| No synchronization is required among stations. | All stations must be synchronized to the same clock to know when slots begin. |
| Higher probability of collision because transmissions can overlap partially or fully. | Lower probability of collision because transmissions either overlap completely or not at all. |
| Maximum channel efficiency is 18.4% — most of the channel capacity is wasted on collisions and idle time. | Maximum channel efficiency is 36.8% — double that of Pure ALOHA due to the reduced vulnerable period. |
| Simpler to implement as no global timing mechanism is needed. | More complex to implement due to the requirement of global time synchronization. |

### 2. CSMA (Carrier Sense Multiple Access)

Before transmitting, a station first **listens to the channel** (carrier sense) to check whether it is currently busy or idle. If the channel is busy, the station holds back and does not transmit. If the channel is idle, it goes ahead and transmits its frame. This "sense before transmit" principle is the key improvement over ALOHA — rather than blindly transmitting at any time, stations are aware of ongoing transmissions and avoid sending into an already occupied channel. This significantly reduces the number of collisions compared to ALOHA, since a station will never intentionally transmit while it can detect another ongoing transmission.

**Problem:** Collisions can still occur due to **propagation delay**. If station A starts transmitting, station B at the far end of the cable may not yet detect the signal and also starts transmitting — resulting in a collision. Once a collision occurs, both frames are fully transmitted before the error is detected, wasting the entire transmission time of both frames.

**Variants based on what a station does when channel is busy:**
- **1-persistent** — waits until channel is idle then immediately transmits. Simple but has high collision probability since multiple waiting stations all transmit at once when channel becomes free.
- **Non-persistent** — waits a random amount of time then senses the channel again. Reduces collisions but introduces unnecessary delay even when the channel is free.
- **p-persistent** — when channel becomes idle, transmits with probability p and waits for next slot with probability (1-p). Balances between collision probability and channel delay.

### 3. CSMA/CD (CSMA with Collision Detection)

CSMA/CD extends CSMA by adding **collision detection during transmission**. While transmitting, a station continuously monitors the channel and compares its own transmitted signal with what is present on the channel. If it detects that its signal has been distorted due to a collision, it immediately **aborts the transmission**, sends a brief **jam signal** to notify all other stations that a collision has occurred, and then waits a random backoff time before attempting to retransmit. This "detect and abort" approach saves significant bandwidth compared to plain CSMA, where a station wastes the entire frame transmission time even after a collision.

**Why better than CSMA:** In plain CSMA, once a collision occurs, both stations continue transmitting their entire frame before realizing something went wrong — wasting the full transmission time of both frames. CSMA/CD stops transmission the moment a collision is detected, reducing wasted channel time to just the collision detection period rather than the full frame transmission time. This makes the channel significantly more efficient under heavy load.

**Problem:** For collision detection to work correctly, the sender must still be transmitting when the collision signal returns back to it. This means the frame must be large enough to last at least one full round-trip propagation time. This is why Ethernet enforces a minimum frame size of **64 bytes**. If a frame is too short, the sender may finish transmitting before detecting the collision, making CD ineffective.

**Cannot be used in wireless networks** because a transmitting station's own strong signal drowns out any incoming collision signal — the station cannot hear collisions while it is transmitting.

**Used in:** IEEE 802.3 (Ethernet) on wired networks.


### 4. CSMA/CA (CSMA with Collision Avoidance)

CSMA/CA is designed specifically for wireless networks where collision detection is not feasible. Instead of detecting collisions after they happen, it tries to **avoid collisions before they occur**. Before transmitting, a station senses the channel — if the channel is busy, it waits. If the channel is idle, it waits for an additional random **backoff time** before transmitting. This random backoff ensures that if multiple stations were waiting for the channel to become free, they do not all transmit simultaneously the moment it does.

Additionally, the receiver sends an explicit **ACK** for every successfully received frame. If the sender does not receive an ACK within a timeout, it assumes a collision or loss occurred and retransmits.

**Why better than CSMA/CD for wireless:** CSMA/CD cannot work on wireless networks due to two key problems. First, a transmitting station's own strong signal drowns out any incoming signal, making collision detection impossible. Second, the **hidden node problem** — two stations may be out of range of each other but both transmit to the same access point simultaneously, causing a collision at the receiver that neither sender can detect. CSMA/CA addresses both problems through random backoff and explicit ACKs, minimizing collision probability without requiring collision detection.

**Used in:** IEEE 802.11 (Wi-Fi).

**Collision avoidance in CSMA/CA is achieved through:**

**IFS (Inter-Frame Space):** Before transmitting, a station must wait for a specific idle period called IFS after the channel becomes free. Different frame types use different IFS durations to set priority — **SIFS (Short IFS)** is the shortest and is used for high-priority frames like ACKs and CTS; **DIFS (Distributed IFS)** is longer and is used by regular data stations before entering the contention window. This ensures ACKs are sent before any new data transmission begins.

**Contention Window:** After waiting for DIFS, the station picks a random backoff slot from a range called the **contention window (CW)**. Each station counts down its slots. The station that reaches zero first transmits. Others pause their countdown while the channel is busy and resume after it goes idle. If a collision occurs, the window is **doubled (exponential backoff)** and a new random slot is picked, reducing the chance of repeated collisions.


## 9. IEE Standards

### 1. Ethernet (IEEE 802.3)
**Ethernet (IEEE 802.3)** is the most widely used wired LAN technology, operating at the Data Link and Physical layers. It uses **CSMA/CD** for medium access control and defines a specific frame format for data transmission over a shared medium. Every device on an Ethernet network is identified by a unique **MAC address** burned into its network interface card, and frames are delivered based on these hardware addresses within a local network.

**Ethernet (IEEE 802.3)** is the most widely used wired LAN technology, operating at the Data Link and Physical layers. It uses **CSMA/CD** for medium access control and defines a specific frame format for data transmission over a shared medium.

#### Ethernet Frame Structure

```
| Preamble | SFD | Destination | Source | Type/Length | Data | PAD | CRC |
```

| Field | Size | Function |
|---|---|---|
| **Preamble** | 7 bytes | Alternating 1s and 0s (`10101010`) used to synchronize the receiver's clock with the sender before actual data arrives. |
| **SFD (Start Frame Delimiter)** | 1 byte | Fixed pattern `10101011` — marks the end of the preamble and signals that the actual frame is about to begin. |
| **Destination Address** | 6 bytes | MAC address of the intended receiver. Can be unicast, multicast, or broadcast. |
| **Source Address** | 6 bytes | MAC address of the sending station. |
| **Type / Length** | 2 bytes | If value > 1500, it indicates the protocol type of the payload (e.g., 0x0800 for IPv4). If value ≤ 1500, it indicates the length of the data field. |
| **Data (Payload)** | 46–1500 bytes | The actual data passed down from the Network Layer. |
| **PAD** | Variable | If data is less than 46 bytes, padding is added to meet the minimum frame size of 64 bytes required for CSMA/CD to work correctly. |
| **CRC** | 4 bytes | Cyclic Redundancy Check computed over the entire frame. The receiver recomputes CRC and compares it to detect any transmission errors. |


**Token Bus (IEEE 802.4)** and **Token Ring (IEEE 802.5)** are deterministic LAN protocols that eliminate collisions entirely by using a **token** — a special control frame that circulates among stations. Only the station holding the token is allowed to transmit. This guarantees every station gets a fair turn and no collisions ever occur, unlike CSMA/CD.
**Token Bus (IEEE 802.4)** and **Token Ring (IEEE 802.5)** are deterministic LAN protocols that eliminate collisions entirely by using a **token** — a special control frame that circulates among stations. Only the station holding the token is allowed to transmit, guaranteeing every station gets a fair turn with zero collisions.

### 2. Token Bus (IEEE 802.4)

Token Bus uses a **physical bus topology** but forms a **logical ring** among stations. The token is passed from station to station in a predetermined logical order, not necessarily following the physical layout of the cable.

**Operation:**
1. A free token circulates among stations in a predetermined logical order.
2. When a station receives the token, it checks if it has data to send.
3. If it has data, it transmits its frames for a maximum allowed time, then passes the token to the next station in the logical ring.
4. If it has no data, it immediately passes the token forward.
5. If a station fails to receive the token within a timeout, the previous station assumes the token is lost and generates a new one.

**Key points:**
- Physical topology is a bus; logical topology is a ring.
- Token is passed in a defined logical order regardless of physical position.
- Used in industrial automation and manufacturing environments where deterministic access is critical.
- Defined under **IEEE 802.4**.

### 3. Token Ring (IEEE 802.5)

Token Ring uses both a **physical and logical ring topology**. Stations are connected in a ring and the token travels around the ring in one direction only.

**Operation:**
1. A free token continuously circulates around the physical ring in one fixed direction.
2. When a station wants to transmit, it waits for the free token, captures it, and changes it to a busy token.
3. It transmits its frame onto the ring. The frame travels around the entire ring, passing through every station.
4. Each intermediate station reads the destination address — if the frame is addressed to them, they copy it, mark it as received, and pass it forward.
5. The frame eventually returns to the original sender, who removes it from the ring and releases a new free token for the next station.
6. One station acts as the **Active Monitor** — it detects lost tokens or frames stuck circling the ring and recovers by generating a new token.

**Key points:**
- Both physical and logical topology are a ring.
- Token travels in one fixed direction around the ring.
- The sending station removes its own frame after it completes a full loop.
- Requires an Active Monitor for fault recovery.
- Defined under **IEEE 802.5**.

| Token Bus (IEEE 802.4) | Token Ring (IEEE 802.5) |
|---|---|
| Physical topology is a bus but logical topology is a ring. | Both physical and logical topology are a ring. |
| Token is passed in a predetermined logical order among stations. | Token travels around the physical ring in one fixed direction. |
| Stations can be at any physical position regardless of logical order. | Station order in the ring determines the token passing order. |
| Used mainly in industrial and manufacturing environments. | Used mainly in office LAN environments. |
| More complex to manage due to mismatch between physical and logical topology. | Simpler to manage since physical and logical order are the same. |

### 4. FDDI
**FDDI (Fiber Distributed Data Interface)** is a high-speed LAN technology that uses **optical fiber** as its transmission medium, operating at **100 Mbps**. It uses token passing over a **dual ring topology** — a primary ring and a secondary ring running in opposite directions — for both data transfer and fault tolerance.

#### Features

- Operates at **100 Mbps** over optical fiber.
- Uses **dual ring topology** — primary ring carries data, secondary ring stays idle as backup.
- Supports up to **500 stations** with a maximum ring length of **200 km**.
- Uses **timed token protocol** — each station holds the token for a fixed time only, ensuring fair access.
- Immune to electromagnetic interference due to fiber medium.

#### Fault Tolerance Mechanism

FDDI uses a **dual ring topology** — a primary ring and a secondary ring running in opposite directions — as its fault tolerance mechanism. Under normal operation, only the primary ring carries data while the secondary ring remains idle as a backup.

When a cable break or station failure occurs, the two stations on either side of the fault detect the failure and automatically perform **ring wrapping** — they loop the primary ring back onto the secondary ring, forming a single continuous loop that bypasses the failed segment. This self-healing process requires no manual intervention and the network continues to operate normally, though with reduced redundancy since both rings are now merged into one.

However, if both the primary and secondary rings are physically cut at the same location (such as when both cables run through the same conduit and are cut together), ring wrapping cannot help. The network splits into two completely isolated segments that can no longer communicate with each other. This is the only scenario where FDDI's fault tolerance mechanism completely fails.


## 10. VLAN
**VLAN (Virtual Local Area Network)** is a logical grouping of devices on a network that makes them behave as if they are on the same physical LAN, even if they are physically located on different switches or segments. VLANs are configured through software, allowing administrators to segment a network without physically rewiring anything. Each VLAN has its own separate broadcast domain, meaning broadcast traffic stays within the VLAN and does not reach devices in other VLANs, improving both performance and security.

### 1. Design

Each VLAN is assigned a unique **VLAN ID (1–4094)**. Devices are grouped into VLANs based on their switch port, MAC address, or protocol type. Devices within the same VLAN can communicate directly with each other as if they were on the same physical network. Devices in different VLANs cannot communicate without passing through a **router or Layer 3 switch**, even if they are physically connected to the same switch. This separation allows administrators to logically group devices by department, function, or security level — for example, placing all HR devices in VLAN 10 and all Finance devices in VLAN 20 — without any physical recabling.

### 2. Configuration

VLANs are configured on managed switches by assigning each port to a specific VLAN ID. Each switch port is designated as either:
- **Access port** — belongs to a single VLAN and connects end devices like PCs and printers. The device connected to an access port is unaware it is part of a VLAN — it simply sees a normal network connection.
- **Trunk port** — carries traffic from multiple VLANs simultaneously between switches or between a switch and a router. Trunk ports use **IEEE 802.1Q tagging** — a 4-byte tag is inserted into each Ethernet frame identifying which VLAN it belongs to, allowing the receiving device to correctly separate and route traffic to the right VLAN.
