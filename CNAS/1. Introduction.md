# 1. Computer Network

A computer network is the infrastructure that allows computers and networking devices to exchange data. It is a collection of computers and other devices that use a common network protocol to share information with each other over a network medium.

## 1.1 Uses of Computer Networks

- **Resource sharing** — Hardware like printers and storage devices, as well as software and files, can be shared across all devices on the network without needing physical duplication.
- **Communication** — Networks enable email, instant messaging, video conferencing, and VoIP, allowing people to communicate in real time regardless of location.
- **Remote access** — Users can access files, applications, and systems from a remote location as if they were physically present on the local network.
- **Centralized data management** — Organizations can store and manage data centrally on servers, making backup, security, and access control easier.
- **Distributed computing** — Complex tasks can be split across multiple machines on the network, improving processing speed and efficiency.

## 1.2 Networking Models

### 1.2.1 Client Server Network

A client-server network is a model where two types of devices exist — clients and servers — each with a fixed role. The server is a powerful, always-on machine that stores data and provides services. The client is the device that requests those services. They never swap roles. A good example is when you open a website — your browser is the client, and the machine hosting that website is the server.

![Client-Server Network](https://github.com/user-attachments/assets/0d32719b-93d2-4740-82ec-551e91cea097)

### 1.2.2 Peer to Peer Network

A peer-to-peer network is a model where every device (called a peer) acts as both a client and a server at the same time. There is no central machine in control — each peer shares its own resources directly with others. BitTorrent is a classic example, where users simultaneously download from and upload to other users.

![Peer to Peer Network](https://github.com/user-attachments/assets/b5191d10-aa5a-46cf-b3e9-e52497aa8eb8)

### 1.2.3 Active Network

An active network is a network model where the intermediate nodes such as routers and switches are not just passing data packets from one point to another — they can actually process, modify, or even execute code inside those packets. Unlike traditional networks where nodes are passive and only forward data, active network nodes are programmable and can perform custom computations on the data as it travels through the network. A simple example would be a router that compresses or encrypts your data on the fly before forwarding it, without the sender or receiver needing to do anything extra.

## 1.3 Client-Server vs Peer-to-Peer

| S.N. | Client-Server                                                    | Peer-to-Peer                                                          |
| ---- | ---------------------------------------------------------------- | --------------------------------------------------------------------- |
| 1    | One dedicated server handles all requests from clients.          | Every device acts as both client and server simultaneously.           |
| 2    | Centralized management makes administration and security easier. | No central authority — each peer manages itself independently.        |
| 3    | Server failure brings down the entire network's services.        | No single point of failure — if one peer goes down, others continue.  |
| 4    | Scales well for large networks with many users.                  | Performance degrades as the number of peers grows unmanaged.          |
| 5    | Requires expensive, high-performance server hardware.            | Low cost — any device can participate without dedicated hardware.     |
| 6    | Better security since access control is enforced at the server.  | Harder to enforce security since there is no central control point.   |
| 7    | Used in enterprises, web hosting, and banking systems.           | Used in file sharing systems like BitTorrent and blockchain networks. |


**Advantages of Client-Server:** Centralized backup and recovery, consistent security policies, easy to manage at scale, reliable performance for many simultaneous users.

**Disadvantages of Client-Server:** Single point of failure at the server, high hardware and maintenance cost, server can become a bottleneck under heavy load.

**Advantages of P2P:** No single point of failure, low setup cost, easy to expand by adding peers, each peer contributes resources to the network.

**Disadvantages of P2P:** Difficult to manage and secure, inconsistent performance depending on peer availability, no centralized backup or data integrity guarantee.

## 1.4 Network Topology

Network topology refers to the arrangement or layout of devices (nodes) and connections (links) in a network. It defines how devices are physically or logically connected to each other.

**Examples:**

* **Bus Topology** – All devices are connected to a single backbone cable.
* **Star Topology** – All devices are connected to a central hub or switch.
* **Ring Topology** – Devices are connected in a circular loop.
* **Mesh Topology** – Every device is connected to multiple other devices.
* **Tree Topology** – Hierarchical arrangement of star networks.
* **Hybrid Topology** – Combination of two or more topologies (e.g., Star-Bus, Star-Ring).


## 1.5 Protocols and Standards

A protocol is a set of rules that defines how data is transmitted and received between devices in a network. It ensures that two devices can communicate even if they are made by different manufacturers. Examples include HTTP, FTP, TCP, and IP.

A standard is a set of rules that has been officially approved and adopted by a recognized body so that different systems and devices can work together. Standards can be de facto (widely used in practice, like the QWERTY keyboard) or de jure (officially approved by organizations like ISO, IEEE, or ITU).

## 1.6 Layered Architecture

Layered architecture is a design approach where the communication system is broken down into distinct layers, each responsible for a specific task. Each layer provides services to the layer above it and uses services from the layer below it.

![Layered Architecture](https://github.com/user-attachments/assets/1bdde01e-d167-453c-aa4c-5b529dd54005)

### 1.6.1 Why Layered Architecture

1. Breaks a complex networking problem into smaller, manageable pieces.
2. Each layer can be developed, tested, and updated independently without affecting others.
3. Allows different manufacturers to build compatible products as long as they follow the same layer rules.
4. Makes troubleshooting easier since each layer has a defined responsibility.
5. Promotes modularity and reusability across different network systems.

### 1.6.2 Design Issues

1. **Addressing** — Each layer needs a way to identify the sender and receiver at that level.
2. **Error Control** — Detecting and correcting errors that occur during data transmission must be handled at appropriate layers.
3. **Flow Control** — Ensuring a fast sender does not overwhelm a slow receiver with too much data at once.
4. **Multiplexing** — Multiple conversations must be able to share the same physical connection simultaneously.
5. **Routing** — When multiple paths exist, the network must decide which path the data should take.
6. **Encapsulation** — Each layer adds its own header to the data before passing it down to the next layer.

## 1.7 OSI Model

The OSI (Open Systems Interconnection) model is a conceptual framework developed by ISO that standardizes how different network systems communicate with each other. It divides the communication process into 7 distinct layers, where each layer has a specific responsibility and communicates only with the layers directly above and below it.

**Layer 7 - Application Layer** — This is the topmost layer and the one closest to the end user. It provides network services directly to user applications such as browsers, email clients, and file transfer programs. Protocols at this layer include HTTP, FTP, SMTP, and DNS.

**Layer 6 - Presentation Layer** — This layer is responsible for translating, encrypting, and compressing data so that the application layer can understand it. It acts as a translator between the network and the application, handling data format conversions such as JPEG, MP3, and encryption formats like SSL.

**Layer 5 - Session Layer** — This layer manages and controls the connections (sessions) between two communicating devices. It is responsible for establishing, maintaining, and terminating sessions. It also handles session recovery in case a connection drops in the middle of a transfer.

**Layer 4 - Transport Layer** — This layer is responsible for end-to-end communication and reliable data delivery between two devices. It breaks data into smaller segments, sends them, and reassembles them at the destination. It also handles error control and flow control. Key protocols here are TCP (reliable) and UDP (fast but unreliable).

**Layer 3 - Network Layer** — This layer is responsible for logical addressing and routing of data packets across multiple networks. It determines the best path for data to travel from source to destination. The most important protocol at this layer is IP (Internet Protocol). Routers operate at this layer.

**Layer 2 - Data Link Layer** — This layer is responsible for node-to-node delivery of data within the same network. It takes raw bits from the physical layer and organizes them into frames. It also handles error detection and MAC addressing. Switches and bridges operate at this layer.

**Layer 1 - Physical Layer** — This is the lowest layer and deals with the actual physical transmission of raw bits (0s and 1s) over a physical medium such as cables, fiber optics, or radio waves. It defines the hardware specifications like voltage levels, cable types, and data rates.

### 1.7.1 OSI Task Mapping

| Task | OSI Layer |
|---|---|
| Routing between networks | Network Layer |
| Framing (organizing bits into frames) | Data Link Layer |
| Encryption, compression, data format conversion | Presentation Layer |
| Reliable end-to-end delivery, segmentation | Transport Layer |
| Physical transmission of raw bits | Physical Layer |
| Session establishment, maintenance, termination | Session Layer |
| User-facing network services (browsers, mail) | Application Layer |
| MAC addressing | Data Link Layer |
| Logical (IP) addressing | Network Layer |
| Error detection (CRC) | Data Link Layer |
| Flow control between adjacent nodes | Data Link Layer |
| End-to-end flow and error control | Transport Layer |

## 1.8 TCP/IP Model

The TCP/IP (Transmission Control Protocol/Internet Protocol) model is a practical, four-layer networking model that was developed by the US Department of Defense. Unlike the OSI model which is a theoretical framework, the TCP/IP model is the actual model used in the real world and forms the foundation of the modern internet. It is simpler than OSI as it combines several OSI layers into one.

**Layer 4 - Application Layer** — This is the topmost layer and combines the functions of OSI's application, presentation, and session layers into one. It provides protocols that applications use to communicate over the network. Common protocols include HTTP, FTP, SMTP, DNS, and Telnet.

**Layer 3 - Transport Layer** — This layer is responsible for end-to-end communication and reliable data delivery between two devices across the network. It breaks data into segments and ensures they are delivered correctly using two main protocols — TCP (Transmission Control Protocol) which is reliable and connection-oriented, and UDP (User Datagram Protocol) which is fast but connectionless.

**Layer 2 - Internet Layer** — This layer is responsible for logical addressing and routing of data packets across multiple networks. It determines the best path for data to reach its destination. The main protocol at this layer is IP (Internet Protocol). It also includes ICMP (for error reporting) and ARP (for address resolution).

**Layer 1 - Network Access Layer** — This is the lowest layer and combines the functions of OSI's data link and physical layers. It handles how data is physically transmitted over the network medium, including MAC addressing, framing, and the actual hardware transmission of bits over cables, fiber, or wireless signals.

## 1.9 OSI vs TCP/IP

![OSI vs TCP/IP](https://github.com/user-attachments/assets/eac8d108-0c60-4034-9a60-3bc7c8d021d5)

| OSI Model | TCP/IP Model |
|---|---|
| Has 7 layers. | Has 4 layers. |
| Protocols are independent of the model, making it more flexible. | Protocols are built into the model, making it less flexible. |
| Each layer has a strict and clearly defined role and function. | Layer boundaries are less strict and functions can overlap. |
| Separate presentation layer handles encryption, compression and translation. | No separate presentation layer — these functions are handled by the application layer. |
| Separate session layer manages session establishment and termination. | No separate session layer — session management is done by the application layer. |
| Error control is performed at both data link and transport layer. | Error control is primarily performed at transport layer only. |
| Supports both connection-oriented and connectionless at network layer. | Network layer (internet layer) is connectionless only. |
| More reliable for large and complex enterprise networks. | More reliable for real world internet communication. |

## 1.10 Data Encapsulation

Data encapsulation is the process where each layer of the OSI model adds its own header (and sometimes trailer) to the data before passing it down to the next layer. At the receiving end, each layer removes its own header as the data moves up the layers. This process is called de-encapsulation.

### 1.10.1 The Process

At the Application Layer, the user data is created. This raw data is then passed down to the Presentation Layer which formats, encrypts or compresses it and passes it further down.

At the Session Layer, session information is added to manage the communication and the data is passed to the Transport Layer. Here a header containing source and destination port numbers, sequence numbers and error control information is added. At this stage the data is called a Segment.

At the Network Layer, a header containing the source and destination IP addresses is added for routing purposes. At this stage the data is called a Packet.

At the Data Link Layer, both a header containing the MAC address and a trailer containing error detection information (CRC) is added. At this stage the data is called a Frame.

At the Physical Layer, the frame is converted into raw bits (0s and 1s) and transmitted over the physical medium such as cables or wireless signals.

### 1.10.2 Summary of Data Names at Each Layer

| Layer | Data Name |
|---|---|
| Application, Presentation, Session | Data |
| Transport Layer | Segment |
| Network Layer | Packet |
| Data Link Layer | Frame |
| Physical Layer | Bits |

## 1.11 Service Primitives

A **service primitive** is a formal operation through which one layer requests a service from the layer directly below it, or is notified of an event by it. Primitives define the interface between adjacent layers without specifying how the service is implemented internally.

There are four types of service primitives:

- **REQUEST** — a layer asks the layer below to perform a service (e.g., establish a connection).
- **INDICATION** — the layer below notifies the layer above that an event has occurred (e.g., a connection request arrived).
- **RESPONSE** — the layer above responds to an indication (e.g., accepting or rejecting the connection).
- **CONFIRM** — the layer below confirms to the requesting layer that the requested service has been completed.

### 1.11.1 Connection-Oriented Service Primitives

Connection-oriented service operates in three phases — connection establishment, data transfer, and connection release — each using specific primitives.

```
Client Side                          Server Side

CONNECT.request  ─────────────────► CONNECT.indication
CONNECT.confirm  ◄───────────────── CONNECT.response

DATA.request     ─────────────────► DATA.indication
DATA.indication  ◄───────────────── DATA.request

DISCONNECT.request ───────────────► DISCONNECT.indication
```

- In the **connection establishment** phase, the client issues a CONNECT.request. The server receives a CONNECT.indication, sends back a CONNECT.response, and the client receives a CONNECT.confirm — the connection is now established.
- In the **data transfer** phase, both sides use DATA.request to send and DATA.indication to receive data.
- In the **connection release** phase, either side issues a DISCONNECT.request and the other receives a DISCONNECT.indication, terminating the connection cleanly.
